## 这一章真正要解决的问题

当数据或请求已经大到一台机器无法承载时，复制并不能解决容量问题。复制是把**同一份数据**放到多处，主要改善容错与读取吞吐；分区（partitioning，很多产品称 sharding）则把**不同的数据子集**交给不同节点，目的是让磁盘容量、CPU 与请求吞吐能够随节点数增长。

最常见的组合不是“复制还是分区”，而是二者同时存在：每条记录按主键只属于一个逻辑分区，但该分区有多个副本。一个节点可以是分区 A 的 leader，同时是分区 B、C 的 follower。

```text
逻辑分区 P0          逻辑分区 P1          逻辑分区 P2
leader: Node A       leader: Node B       leader: Node C
follower: B, C       follower: C, A       follower: A, B

分区回答：这条数据属于哪一组？
复制回答：这一组数据在哪些节点有副本，谁接收写入？
```

这两个决策大体正交，但运维上会互相放大：移动一个分区，实际可能要重建多个副本；某节点失效，不只是少了一台机器，还会触发 leader 转移、补副本和路由更新。设计分片方案时如果只计算“每台机器存多少 GB”，而忽略复制流量和恢复时间，容量模型通常是不完整的。

> [!note] 两种 partition 不要混淆
> 本章的 partition 是有意把数据拆开；network partition 是节点之间的网络故障，将在 Ch8 讨论。二者同名，但一个是数据布局，一个是故障模型。

## 分区设计的目标：均匀不是只看字节数

理想情况下，10 台机器应接近承载单机 10 倍的数据和吞吐。但这依赖一个很强的前提：数据与工作量能均匀分开。某些分区承担远多于平均值的数据或请求，叫 **skew（倾斜）**；负载异常高的分区叫 **hot spot（热点）**。

“均匀”至少有四个维度：

| 维度 | 只看平均值会漏掉什么 |
| --- | --- |
| 存储字节 | 对象大小分布、压缩率、墓碑与历史版本 |
| 读请求 | 单次读取成本、缓存命中率、扫描行数 |
| 写请求 | 写放大、索引数量、同步复制确认数 |
| 后台工作 | compaction、备份、修复、迁移与副本追赶 |

所以“每个 shard 都是 100 GB”不代表负载均衡。一个冷归档 shard 与一个高频更新 shard 即使大小相同，CPU、磁盘 IOPS 和网络成本也完全不同。生产调度器最好按多维成本做 placement，而不是仅按分区数量轮询。

随机把记录分给节点能得到相当均匀的数据量，却无法定位：读取一个 key 时不知道该问谁，只能广播。因此真正有用的方案必须同时满足两个条件：**尽可能打散负载，并且能从 key 确定目标分区。**

## Key-range partitioning：保留顺序，承担连续热点

范围分区给每个分区一段连续 key，例如 `[a, f)`、`[f, n)`、`[n, z]`。知道边界和分区所有者后，路由器可以直接定位。分界不能简单等宽，因为真实数据通常不均匀；字典中不同首字母的词数不同，用户 ID 的业务前缀也可能严重偏斜。

范围分区最大的价值是**保留排序**：分区内部可用 SSTable/B-Tree 维持有序数据，范围扫描和相邻记录读取很自然。复合 key 尤其常见：

```text
key = (tenant_id, event_time, event_id)

先按 tenant_id 定位一段连续范围
再按 event_time 获取该租户某个时间窗的数据
event_id 用于打破同一时间戳的排序冲突
```

代价是访问模式若与排序方向一致，热点也会被保留。以纯时间戳作为首字段时，所有“现在发生”的写都落到最右端分区，旧分区空闲而最新分区过载。把 `sensor_id` 放在时间戳前可以让活跃传感器分散写入，但查询多个传感器时就必须分别扫描再合并。

这说明分区键不是数据建模的附属选项，而是在决定系统优先优化哪一种访问路径：

- `timestamp` 优化全局时间范围扫描，却制造单调写热点。
- `(device_id, timestamp)` 打散设备写入并优化单设备时间线，但跨设备查询会 fan-out。
- `(time_bucket, hash(device_id), timestamp)` 可控制 fan-out 数量，但查询与迁移逻辑更复杂。

### 时间桶不是免费的万能药

工程上常把时间切成小时或天，再加若干 hash bucket。桶过少仍有热点，桶过多则每次查询要访问大量小分区；跨桶查询还要处理分页、排序与重复。时间桶粒度应由峰值写入、保留期、查询窗口和单分区目标大小共同决定，而不是凭直觉选“一天”。

## Hash partitioning：打散 key，放弃全局顺序

Hash 分区先计算稳定的 `hash(key)`，再把 hash 空间的区间交给不同分区。好的 hash 能把相似甚至有规律的输入映射到近似均匀的数字空间。这里不要求密码学安全，却要求：跨语言和进程结果稳定、分布良好、版本升级不能悄悄改变算法。Java `Object.hashCode()` 一类可能因类型或运行时变化而不适合作为持久分区契约。

```text
user:alice ─hash─> 0x19... ─> P0
user:alicia ─hash─> 0xd4... ─> P7
user:bob   ─hash─> 0x52... ─> P2
```

它通常比原始 key range 更均匀，却破坏了相邻性：`alice` 与 `alicia` 不再相邻，按主键做范围查询可能必须访问所有分区。因此 hash 分区不是“比范围分区高级”，而是在**负载均匀性和范围局部性**之间选择另一侧。

一个常见折中是复合主键只 hash 第一部分，剩余部分作为分区内排序键。例如 `(user_id, update_timestamp)`：`user_id` 决定分区，同一用户的内容在一个分区内按时间排序。它支持“读取某用户某时间段”，却不支持“按 user_id 范围读取所有用户”。

> [!warning] consistent hashing 里的 consistent 不是一致性
> 它不表示 ACID consistency，也不表示副本强一致。原始 consistent hashing 描述节点变化时如何少移动 key。许多数据库文档沿用这个词，实际实现却是预建分区或 vnode。讨论架构时最好明确说 hash space、partition 与 ownership，避免只丢出一个含混名词。

## Hash 也救不了单 key 热点

如果 30% 的请求都访问同一个 celebrity ID，同一个 key 的 hash 永远相同。Hash 只能打散**不同 key 的分布**，不能拆开一个 key 自身的负载。

应用层常用 hot-key salting：把逻辑 key 拆为 `hot-key#00` 到 `hot-key#99`，写入随机选择后缀。它能把写扩散到最多 100 个分区，但读必须 fan-out 并聚合，而且系统需要记录哪些 key 被拆过、拆成多少份、何时可以恢复。若所有 key 都固定拆 100 份，冷 key 会产生巨大的元数据和读放大。

更完整的热点治理顺序通常是：

1. 区分读热点与写热点。读热点优先用缓存、副本或 CDN；写热点才需要拆 key 或改变数据模型。
2. 找出热点持续时间。突发热点可以用队列和限流吸收；长期热点更适合专门分片。
3. 决定允许的语义。分片计数器只能近似聚合，强一致余额则不能随意拆写。
4. 建立合并路径。读取、对账、过期和缩容都必须知道逻辑 key 与物理 key 的映射。

热点检测不应只看 node CPU。建议至少暴露 per-partition QPS、bytes/s、P99、队列长度、top-K keys、拒绝率与 cache hit ratio。Node 平均负载健康时，某个 partition 的执行队列可能已经饱和。

## 二级索引：写局部与读局部无法同时免费获得

仅按主键访问时，路由函数足以找到记录。二级索引按非唯一属性查找，如“所有红色汽车”“包含 partition 一词的文章”，索引项与主记录的分区边界往往不同。本章给出两种基本布局。

### 按文档分区：local index

主记录按 document ID 分区，每个分区只维护自己文档的二级索引。更新一辆车时，主记录和 `color:red`、`make:honda` 的本地索引可以在同一分区完成，写路径局部、事务边界清晰。

但搜索“所有红色 Honda”不知道答案在哪些 document partition，只能向所有分区 scatter，再 gather、去重、排序和截断结果。

```text
query color:red
       ├──> P0 local index ─┐
       ├──> P1 local index ─┼─> merge + sort + top K
       ├──> P2 local index ─┤
       └──> P3 local index ─┘
```

并行不会消除代价。端到端延迟受最慢必要分区支配，分区越多，遇到一个长尾的概率越大；节点故障时还要定义部分结果是否可接受。全局 top-K 也不能简单让每个 shard 返回任意 K 条：必须按同一排序规则返回局部 top-K，协调器再合并；深分页会使每个 shard 读取和传输越来越多候选项。

### 按词项分区：global/term-partitioned index

另一种布局让所有 `color:red` 索引项聚在同一个逻辑索引分区，词项本身或其 hash 决定索引 shard。查询单个 term 只访问一个分区，读更局部；范围属性按 term range 划分还能支持价格范围扫描。

代价转移到写入：一个文档包含多个 term，一次主记录更新可能要修改许多索引分区。如果要求主记录与所有索引原子可见，就需要跨分区事务；很多系统选择异步更新，于是搜索结果在短时间内落后于事实库。

```text
主记录写入成功
   └─ change log / CDC ─> index workers ─> 多个 term partitions
                              │
                              └─ watermark 表示已追到哪个源位置
```

工程上应把这种延迟明确成契约，而不是把它伪装成“最终会好”：展示“刚创建的商品”时可先合并主库结果；搜索页可以接受秒级新鲜度；权限撤销和合规删除则可能不能等待普通异步索引。关键指标包括 indexing lag、失败重试队列、源 offset 与索引 watermark、漏索引/多索引对账差异。

## Rebalancing：真正困难的是在线改变 ownership

节点扩容、数据增长或机器失效都会要求把负载从一处移到另一处。一次合格的 rebalancing 应同时满足：迁移后负载更公平；迁移期间继续读写；只移动必要数据；前台延迟和复制安全性不越过预算。

### 为什么 `hash(key) mod N` 是陷阱

当节点数从 10 变 11 时，绝大多数 key 的取模结果都会改变，几乎全库搬迁。问题在于把“逻辑分区数”直接绑定到“当前机器数”。更稳妥的做法是增加稳定的中间层：

```text
key ─hash─> logical partition ─assignment─> node
```

扩容时 key 到逻辑 partition 的映射不变，只改变少量 partition 到 node 的 assignment。

### 三类分区数量策略

| 策略 | 如何变化 | 优点 | 主要风险 |
| --- | --- | --- | --- |
| 固定大量 partitions | 初始建立远多于节点数的分区，只移动整分区 | 元数据与迁移模型简单 | 初始数量难选；过大恢复慢，过小管理开销高 |
| 动态 split/merge | 超过大小阈值则拆，过小则合并 | 随数据量调整，范围边界可自适应 | 空库先集中在一个分区；split 本身有负载 |
| 每节点固定数量 | 节点加入时切走若干 hash ranges/vnodes | 分区大小可随节点数大致稳定 | 随机选择可能不均，元数据和修复任务较多 |

动态范围分区的冷启动值得特别注意：空库没有分布信息，可能从一个分区开始，在首次 split 前所有写都压在一台节点。已知 key 分布时可 pre-split；不知道时可采样生产流量或设计带 hash 前缀的 key，但这也会影响范围查询。

### 一套可恢复的在线迁移协议

“把文件复制过去，再改路由”隐藏了并发写与失败恢复。一个更完整的迁移状态机可以是：

1. **Plan**：控制面选择源节点 S、目标节点 T、partition P，生成新的 assignment epoch `e+1`，但尚未对外生效。
2. **Snapshot**：T 从 S 复制 P 的一致性快照，并记录快照对应的 log position `L0`。
3. **Catch up**：S 继续服务并把 `L0` 后的增量日志传给 T；限制迁移带宽，避免挤占前台复制。
4. **Quiesce/cutover**：当 lag 足够小，短暂冻结或把写双发到新 owner，提交 epoch `e+1`，路由开始指向 T。
5. **Forwarding grace**：S 在租约窗口内转发或返回带新 epoch 的重定向，吸收旧客户端缓存。
6. **Verify and cleanup**：对比记录数、checksum、最新 log position 和副本健康，等待回滚窗口后再删除 S 上旧数据。

每一步都必须幂等，控制器重启后能从持久状态继续。失败时不能出现“不知道谁是 owner”的中间态；删除源副本永远应晚于新 owner 被验证和足够副本恢复。

### Epoch、lease 与 fencing

路由缓存不可能瞬时同步。旧 owner 可能因暂停或网络隔离继续接受请求，因此 assignment 需要单调递增的 **epoch/generation**。写请求携带客户端所知 epoch，节点只接受当前 epoch；下游存储也拒绝旧 epoch 的写，这就是 fencing。

```text
旧 owner S 持有 epoch 41 ──迟到写──> storage: reject
新 owner T 持有 epoch 42 ──正常写──> storage: accept
```

仅依赖“控制面已经通知 S”不够，因为 S 可能正处于 GC pause；仅依赖 lease 也要求明确时钟与到期语义。Fencing token 把正确性检查放到最终被修改的资源上，能阻止复活的旧 owner 覆盖新数据。Ch8 会进一步讨论暂停、时钟和 lease 的危险。

## 自动还是人工：选择的是控制环风险

全自动 rebalancing 减少日常操作，却可能和故障检测形成正反馈：某节点因过载变慢，被误判失效；系统开始搬走其数据，迁移 I/O 又让节点与网络更慢；更多心跳超时，最终级联失败。

因此现实方案常采用“自动规划 + 策略护栏 + 人工批准”，或仅在低风险条件下自动执行。护栏应包括：

- 同时迁移的 partition 数和每节点并发任务上限。
- 网络、磁盘、compaction 与 replication bandwidth 预算。
- 前台 P99、错误率或复制 lag 恶化时自动暂停。
- 单 AZ/机架同时丢失副本数限制，始终保持最小冗余。
- 冷却时间，避免监控抖动导致 partition 来回搬动。
- 明确的 abort 与 resume，而不是只能从头再来。

手工参与并不意味着靠 SSH 执行临时命令。好的半自动系统仍应生成可审计计划、估算搬迁字节和风险、持久化状态机，并提供一键暂停与回滚。

## Request routing：数据平面必须跟上控制面

分区 ownership 会变化，客户端必须知道请求发给谁。本章归纳三种路径：

1. 客户端连任意节点；若该节点不是 owner，则内部转发。
2. 客户端先连 partition-aware routing tier，由路由层转发。
3. 客户端持有分区表，直接连接 owner。

| 模式 | 客户端复杂度 | 额外 hop | 更新分区表的责任 |
| --- | --- | --- | --- |
| 任意节点协调 | 低 | 非 owner 请求有 | 数据库节点 |
| 独立路由层 | 低 | 通常有 | 路由服务 |
| 智能客户端 | 高 | 无 | SDK/客户端缓存 |

不论选择哪种，核心都是一个版本化、权威的 partition map。它可由 ZooKeeper 一类协调服务、专用 config server 或 consensus-backed metadata log 维护；gossip 可传播信息，但必须定义冲突与收敛规则。DNS 适合发现相对稳定的入口地址，不适合承担高频 partition ownership 变化。

路由协议至少要定义三种响应：成功；`NOT_OWNER(current_epoch, hint)` 重定向；元数据不可确定时失败而非盲写。客户端收到新 epoch 后刷新缓存并有限重试，避免所有请求同时击穿 metadata 服务。服务端在迁移宽限期转发可以提高可用性，但长期转发会掩盖陈旧客户端并增加 hop，应监控 forwarding rate 并设截止时间。

## Parallel query：并行既降低时间，也放大长尾

点查只访问一个分区；local secondary index 会 scatter/gather；MPP 分析系统则把 join、filter、group 和 aggregate 拆成多个 stage，在数据所在节点并行执行。

```text
scan/filter on shards
        │
        ▼
shuffle by join/group key
        │
        ▼
partial aggregate
        │
        ▼
final merge
```

并行的关键不是“多发几个 RPC”，而是让计算贴近数据、尽早过滤、局部聚合，并控制跨网 shuffle。一个 skewed join key 会让某个 reducer 收到大部分数据，整条查询仍由最慢 stage 决定。查询引擎需要跟踪每个 task 的输入字节、输出字节、spill、shuffle、执行时间与 skew ratio，并用动态拆分、广播小表、salted join 或 speculative execution 治理 straggler。

在线系统还要防止大范围查询挤占点查：独立资源池、并发配额、扫描字节上限、超时与 admission control 往往比“再多并行一点”更重要。

## 一套生产评审清单

在确定 shard key 前，至少回答：

| 类别 | 需要写清楚的内容 |
| --- | --- |
| 访问模式 | 点查、范围、top-K、聚合分别占多少；能否限定 tenant |
| 分布 | key 基数、对象大小、top-K 热 key、时间增长与地域分布 |
| 语义 | 跨 partition 写是否原子；索引允许多旧；部分查询结果能否返回 |
| 扩缩容 | 单 partition 目标大小、迁移耗时、带宽上限、最小副本数 |
| 路由 | map 的权威来源、epoch、缓存 TTL、旧 owner 行为、重试预算 |
| 运维 | skew、lag、迁移进度、forwarding、scatter fan-out 与 tail latency |

建议持续监控：partition bytes/QPS/P99 的最大值与中位数比、top 1% partitions 占总负载比例、每节点 leader/follower 分布、迁移 remaining bytes/ETA、复制与索引 lag、跨分区查询 fan-out、路由重定向率、恢复一个最大分区所需时间。只看 cluster average 会系统性隐藏热点。

## 常见理解偏差

1. **Hash 后就不会热点**：它只能打散不同 key；单 key、低基数 key 和大对象仍会热点。
2. **分区数等于节点数**：把二者绑定会让扩容接近全库搬迁，也无法细粒度均衡异构负载。
3. **数据量均匀等于负载均匀**：读写 QPS、索引成本与后台任务可能高度倾斜。
4. **Scatter/gather 并行所以很快**：总延迟受最慢 shard 影响，成本随 fan-out 上升。
5. **异步全局索引只是慢一点**：它改变 read-after-write 和删除可见性，必须有 lag 契约与对账。
6. **复制完文件就能切流**：并发增量、旧路由、失败恢复和 fencing 才决定迁移是否正确。
7. **自动均衡一定优于人工**：没有资源护栏的控制环可能在过载时制造级联故障。
8. **分区键以后可以轻松改**：它进入存储布局、路由、索引和 API，重分区通常是一次全量在线迁移。

## 本章小结

- 分区让数据容量与处理吞吐跨机器扩展，复制则让每个分区容错；生产系统通常组合二者。
- Key range 保留顺序和范围扫描，但容易承接连续热点；hash 更均匀，却牺牲全局顺序。复合 key 是常见折中。
- Local secondary index 使写局部、读 scatter/gather；global term index 使读局部、写跨分区且常带异步延迟。
- Rebalancing 是一个持续在线的 ownership 变更协议，需要快照、增量追赶、epoch/fencing、校验和可恢复状态机。
- 路由层、客户端与节点必须通过版本化 partition map 收敛；陈旧信息是常态，不是例外。
- 真正的均衡要同时观察字节、请求成本、尾延迟和后台工作，并对热点、迁移和大查询设置明确护栏。
