## 这一章真正要回答的问题：并发与故障发生时，哪些中间状态可以假装不存在

Transaction（事务）不是数据库附赠的语法糖，而是一层故障与并发抽象。应用希望把几个读写当作一个逻辑动作；现实中却可能在任意指令后崩溃，两个请求也可能交错执行。事务试图让应用在一定范围内假装：一组操作要么完成、要么没有发生，并发操作像按照某种顺序执行。

这份抽象很有价值，也很容易被高估。“用了事务”没有说明隔离级别、对象范围、跨节点边界、失败语义或应用不变量。弱隔离会允许某些并发异常；分布式事务会增加协调成本；重试会让业务代码执行多次。设计的第一步不是勾选 ACID，而是写清：

```text
哪些数据必须一起改变？
并发请求之间不能观察到什么？
允许暂时不一致多久？
冲突时等待、失败，还是接受近似结果？
```

原书从事务的意义出发，逐级分析 read committed、snapshot isolation，再讨论 lost update、write skew 和 phantom，最后比较三种 serializable 实现。本章按这条论证链展开，并把工程延伸单独标注。

## Transaction 的价值：把不确定失败收束成统一结果

数据库面临的现实包括：进程或机器在写到一半时崩溃；网络中断导致客户端不知道提交结果；多个客户端覆盖彼此更新；读取看到半完成状态；磁盘数据局部损坏。没有事务时，每个应用都要自己追踪部分成功并设计补偿，复杂度会蔓延到所有调用者。

事务通常用 `BEGIN` 和 `COMMIT` 包围一组操作。若不能完成则 `ABORT/ROLLBACK`，应用可重试。关系数据库把它作为核心能力；一些 NoSQL 早期淡化事务，以扩展性或可用性为理由，但“事务一定不可扩展”和“任何系统都必须用强事务”都过于绝对。正确问题是：特定工作负载需要哪种保证，以及获得它的成本是什么。

## ACID 四个字母并不一样硬

ACID 最初用来描述 transaction safety guarantee，但不同产品对它的实现和营销含义差别很大。必须拆开看。

### Atomicity：失败时不留下半套效果

Atomicity 在多线程语境常被理解为“不可分割操作”，在 ACID 中重点是 **abortability**：一个事务包含多次写，如果中途失败，数据库丢弃已发生的局部写，使调用者可以重试整个事务。

例如转账：

```text
T:
  A.balance -= 100
  B.balance += 100
```

若第一步成功、第二步失败，没有原子性就会凭空丢钱。有原子性，数据库把两步都回滚。原子性不保证客户端一定知道结果：`COMMIT` 已落盘但响应丢失时，客户端仍处于 uncertain outcome，需要查询结果或使用幂等业务键。

### Consistency：不变量主要由应用定义

ACID 的 consistency 指数据始终满足某些 invariant，例如账户总额守恒、订单必须引用存在的用户。数据库可以通过 unique、foreign key、check constraint 提供帮助，但“不变量是什么”仍由应用决定；错误事务会一致地写入错误数据。

原书因此指出，atomicity、isolation、durability 是数据库属性，而 consistency 是应用与数据库共同实现的性质。它与复制系统中的 consistency model 也不是同一概念。

### Isolation：并发事务不能破坏彼此的推理

隔离的理想形式是 serializability：多个并发事务的结果等价于它们按某个顺序逐个执行。数据库实际上可以并行，但对应用呈现一个合法串行结果。

现实数据库常提供较弱隔离，因为完全 serializable 可能有性能或实现成本。弱隔离不是自动错误；错误在于应用依赖了级别没有提供的保证。

### Durability：提交后的数据不会因故障消失

单机数据库通常依靠 WAL、fsync 与非易失介质；复制数据库依靠多个副本。但没有绝对耐久：磁盘和备份可同时损坏，异步副本可能落后，固件 bug 可关联失败。durability 是风险降低，不是数学意义的永不丢失。

提交返回前是否等待本地 fsync、多少副本确认，会改变延迟与 RPO。备份存在也不等于可恢复；恢复演练、校验和、跨故障域副本才把耐久承诺变成能力。

## 单对象与多对象事务

### 单对象写也需要原子性与隔离

更新一个大 JSON/document 时，电源可能在写到一半失败；同一对象的两个 read-modify-write 也可能覆盖。存储引擎通常用 WAL 保证单对象原子写，用锁或 CAS 保证单对象隔离。一些数据库还提供文档字段级原子操作。

“单条语句”不总是“单对象”。SQL 更新可能影响许多行，数据库通常把整条语句作为事务；某些 NoSQL 只保证单 key 原子，批量请求可能部分成功，必须读产品文档。

### 多对象事务为什么必要

多对象一致性常见于：

- 外键关系：订单与订单行必须共同存在。
- 反规范化：同一事实复制到多个记录或索引。
- 二级索引：更新 value 时索引也必须同步。

原书以邮件未读计数为例：插入一封邮件和增加计数必须一起对其他读取者可见，否则 UI 会显示“有未读但列表为空”或相反。多对象事务让一组读写共享 transaction ID，并在同一 commit 点可见。

### Error handling：abort 后重试也有边界

事务鼓励“失败就重试”，但重试并非万能：

1. commit 成功但响应丢失，重试会执行两遍，需 idempotency key 或唯一约束。
2. 系统过载时立即重试会进一步加压，应指数退避、jitter 与总预算。
3. 永久性错误（约束失败、校验失败）不应重试。
4. 事务内已发送邮件或调用外部服务，数据库 rollback 无法撤销外部效果。
5. 客户端进程丢失重试状态，需要数据库或消息系统保存工作意图。

因此“数据库事务”与“端到端 exactly once”不是一回事。外部副作用通常通过 transactional outbox、幂等 consumer 或显式 saga 管理。

## Read Committed：最低限度是不读未提交，也不覆盖未提交

Read committed 是常见默认隔离级别，提供两项保证：

- **No dirty reads**：只能看到已提交数据。
- **No dirty writes**：不能覆盖另一个尚未提交事务的写。

### Dirty read 为什么危险

假设 T1 更新用户头像和资料，T2 在 T1 提交前读到新头像、旧资料；若 T1 随后 abort，T2 还使用了从未存在过的状态。数据库可为每个对象保留旧已提交值与新未提交值，其他事务继续读旧值，直到 writer commit。

### Dirty write 为什么危险

两个事务同时更新同一对象，若后写者可覆盖未提交值，事务内部的多对象效果会混合。例如两个人竞争购买同一车辆：付款人与所有者字段可能分别来自两个事务。数据库通常对被修改对象持有 write lock 到事务结束，第二个 writer 等待或失败。

### Read committed 仍允许 read skew

在同一事务中连续读取同一行，可能得到两个已提交版本：

```text
T1: read balance(A) = 500
T2: A -= 100; B += 100; COMMIT
T1: read balance(B) = 600
```

T1 观察到总额 1100，虽然每一次读都只读了已提交值。这叫 nonrepeatable read/read skew。许多短查询能接受，但备份、分析查询或一致性校验不能接受一个不断变化的视图。

## Snapshot Isolation 与 MVCC：让每个事务读取同一时间切片

Snapshot isolation（SI）让事务从一致快照读取：开始后即使其他事务提交，当前事务继续看到自己快照中的旧版本。这对长查询、备份和分析尤其重要。

### MVCC 如何实现快照

Multi-Version Concurrency Control 为一条逻辑记录保留多个 committed version。每个事务获得单调递增 transaction ID；每个版本记录 created-by，有些实现记录 deleted-by 或结束时间。更新不是原地覆盖，而是把旧版本标记删除并写新版本。

一个 reader 通常按可见性规则选择版本：

1. 忽略读取事务开始后才提交的版本。
2. 忽略未提交或已 abort 的版本。
3. 对快照时点已经提交且尚未删除的版本可见。
4. 始终看到本事务自己的写。

```text
logical row: account=42
v1: balance=500, created=T5, deleted=T9
v2: balance=400, created=T9, deleted=null

T8 的快照看 v1；T10 的快照看 v2
```

旧版本不能立即清理，因为长事务可能仍需要它。vacuum/garbage collection 必须等所有潜在 reader 越过版本；一个遗忘的长事务会阻止回收，引发 table bloat、索引膨胀和性能下降。

### 索引怎样支持多个版本

一种实现让所有版本指向相同索引项，再由 heap 可见性过滤；更新若未改变被索引列，可减少索引变更。另一种实现把版本直接放进 copy-on-write/B-tree page，旧 root 对应旧快照；后台在无人引用时回收旧页。具体机制不同，但原则是索引不能让 reader 穿越自己的 snapshot。

### “Repeatable read”名称非常混乱

SQL 标准用 dirty read、nonrepeatable read、phantom 描述级别，但实现比这些现象更复杂。PostgreSQL 的 Repeatable Read、Oracle 的 Serializable 在一些版本中实际提供 snapshot isolation，却不是真正 serializable。只看隔离级别名称会误判，必须查产品对 lost update、write skew、predicate read 的明确保证。

## Lost Update：read-modify-write 悄悄丢掉一次写

两个事务读取相同值，各自计算并写回，后写覆盖先写：

```text
初始 counter = 10
T1 read 10          T2 read 10
T1 write 11         T2 write 11
最终 11，而不是 12
```

Read committed 和标准 snapshot isolation 并不天然阻止所有 lost update。原书列出几种方案。

### Atomic write operation

让数据库内部完成更新：

```sql
UPDATE counters SET value = value + 1 WHERE id = 42;
```

数据库可对对象加排他锁，或让操作在单线程执行。原子操作适合计数、集合 append 等可表达的更新，但复杂业务 read-modify-write 不一定能改写成一个表达式。

### Explicit locking

应用先锁定将要修改的行：

```sql
SELECT * FROM seats WHERE id = 42 FOR UPDATE;
-- 检查状态并更新
```

锁必须覆盖决策依赖的所有对象，漏一处就仍有 race。还要处理 deadlock：数据库选择 victim abort，应用重试。

### Automatic lost-update detection

数据库允许并发读取与计算，但 commit 时检测“从读取以来对象已变化”，abort 一个事务。相比锁，它让读取并行，特别适合 snapshot isolation。PostgreSQL Repeatable Read、Oracle Serializable、SQL Server Snapshot 的具体行为需以版本文档为准。

### Compare-and-set

只有值仍等于先前读取值才更新：

```sql
UPDATE wiki_pages
SET content = :new_content
WHERE id = 42 AND content = :old_content;
```

必须检查 affected row count 为 1；为避免大字段比较，常用 version column。若数据库 `WHERE` 在旧 snapshot 上求值而不重新核验当前值，CAS 也可能失效，需确认实现语义。

### 复制环境中的冲突

多主或 leaderless 系统允许同一 key 在不同副本并发写，不能靠单主锁检测。常见做法是保留多个 sibling/version，使用 CRDT、可交换操作或应用 merge。Last-write-wins 会主动丢弃写，不能算“防止 lost update”；它只提供确定的冲突裁决。

## Write Skew：不同对象上的写也能共同破坏不变量

Write skew 是 lost update 的广义形式。两个事务读取同一组对象，各自更新不同对象，因此行级 write conflict 检测不到，但组合结果违反约束。

经典值班医生例子：至少一名医生必须 on call。

```text
初始：Alice=on, Bob=on

T1(Alice): 查询 on-call 数量 = 2
T2(Bob):   查询 on-call 数量 = 2
T1: Alice=off
T2: Bob=off
结果：0 人值班
```

每个事务在自己的 snapshot 中判断都成立，写入的又是不同行，所以 SI 允许两者提交。serializable 隔离必须让其中一个失败或使效果等价于串行顺序。

### Write skew 的共同模式

1. `SELECT` 一组符合条件的行，检查某个业务条件。
2. 根据查询结果决定是否继续。
3. 写入影响这个条件的记录。
4. 并发事务基于相同旧前提做出互相不兼容的决定。

原书的其他例子包括：会议室重复预订；多人游戏中两个角色抢占同一位置；注册用户名时双方都查询“不存在”；双重消费与余额检查。

如果可把约束写成 unique/check/foreign key，让数据库直接执行最可靠。例如用户名唯一索引。但“某时间段无重叠预订”或“至少一人值班”通常跨多行/谓词，普通 constraint 不易表达。

### Phantoms：冲突对象可能尚不存在

会议室预订先查询：

```sql
SELECT * FROM bookings
WHERE room_id = 123
  AND end_time > :start
  AND start_time < :end;
```

两事务都查询到空集，然后各插入一行。造成冲突的不是已存在的同一行，而是“满足谓词的行不存在”；并发插入制造了 phantom。对已有行 `FOR UPDATE` 无能为力，因为没有行可锁。

### Predicate lock 与 materializing conflicts

真正的 predicate lock 锁定查询条件，而不是当前结果行。任何将要新增/修改成满足该谓词的写都必须等待。它能防 phantom，但匹配任意谓词成本高。

应用有时把谓词冲突物化成可锁对象：预先创建每个房间/时间段 slot 行，预订前锁 slot。这叫 materializing conflicts。它把并发控制结构泄漏进数据模型，容易出错，应视为无法使用 serializable 时的最后手段。

## Serializability：三条实现路线，不是一种机制

Serializable 通常被视为最强隔离：执行结果等价于某个串行顺序。原书讨论三类实现：真正串行执行、two-phase locking（2PL）、serializable snapshot isolation（SSI）。三者性能模型和失败方式完全不同。

## 路线一：Actual Serial Execution

数据库在一个线程上按顺序执行事务，从根源消除并发冲突。它重新可行，依赖两个变化：内存足够容纳活跃数据集，OLTP 事务短小且每笔只做少量读写。

关键限制是事务必须作为整体提前提交给数据库，常通过 stored procedure：若每一步都与应用网络往返，单线程会在等待网络时闲置，吞吐极差。VoltDB/H-Store 一类系统使用确定性存储过程；Redis/Lua 也体现类似思路。

### Stored procedure 的现实摩擦

历史 stored procedure 语言工具差、难调试，数据库内错误代码还会拖慢所有事务；不同数据库语言也形成生态锁定。现代系统可用通用语言、沙箱和预编译改善，但事务逻辑仍须短、确定、不能等待外部 API。

### 分区扩大吞吐，也引入协调

若事务只触碰单 partition，每个 partition 可有独立单线程，吞吐近似水平扩展。跨 partition 事务必须协调所有相关 partition，系统吞吐受最慢参与者和协调协议约束。数据分区键因此直接决定 serial execution 架构是否成立。

适用边界：事务短；活跃集在内存；写吞吐能按 partition 扩展；跨分区很少。长分析查询应在一致快照或单独副本执行，不能阻塞串行 OLTP loop。

## 路线二：Two-Phase Locking（2PL）

2PL 在约三十年里是数据库 serializability 的主要算法。它与 two-phase commit（2PC）没有关系：2PL 是隔离锁协议，2PC 是分布式原子提交协议。

基本规则：

- 多个事务可共享 read lock。
- 写需要 exclusive lock；与其他读锁/写锁冲突。
- 事务读过一个对象后，writer 必须等它结束；事务写后，其他 reader/writer 也等待。
- 锁通常持有到 commit/abort，因此形成 growing/shrinking 两阶段概念。

这比 read committed 的写锁强得多：reader 会阻塞 writer，writer 也阻塞 reader。它直接防止基于旧读做决策，但等待和 deadlock 成为主要成本。

数据库维护 wait-for graph 检测死锁，abort 一个 victim。应用必须重试整个事务。长事务或热点对象会形成 lock convoy；一个慢事务可让队列中的许多事务延迟，尾延迟不稳定。因此即使平均吞吐可接受，P99 也可能很差。

### Predicate lock 与 index-range lock

为阻止 phantom，serializable 2PL 还需锁住谓词。精确 predicate lock 成本较高，实际常用 index-range/next-key lock 近似：查询走索引时锁住命中的范围及间隙，插入该范围会冲突。

锁粒度有权衡：细粒度并发高但锁管理开销大；粗粒度开销小却产生不必要冲突。如果查询没有合适索引，数据库可能扩大到整表锁，因此索引不仅决定查询性能，也可能决定隔离并发度。

## 路线三：Serializable Snapshot Isolation（SSI）

SSI 结合 snapshot isolation 的非阻塞读取与 commit-time 冲突检测。它是 optimistic concurrency control：先让事务执行，若检查发现结果不能序列化，再 abort。

### Pessimistic 与 optimistic 的取舍

2PL 悲观地假设冲突危险，先等待；serial execution 悲观地把所有事务排队；SSI 乐观地允许继续，在提交前验证。争用低时 SSI 避免无谓等待；争用高时大量事务做完工作后 abort，成本可能更高。

“optimistic”不等于不需要锁：writer 之间仍可能用短锁防 dirty write。区别主要在 reader 不阻塞 writer，数据库跟踪读写依赖以发现危险结构。

### 基于过时前提做决定

SI 下 write skew 都有同一模式：事务读取某个前提，之后另一个事务使前提失效，前者仍据此写入。SSI 需要检测两类情况：

1. **读取了旧 MVCC version**：读时另一个事务已有未提交新版本；若对方后来提交，reader 的前提可能过时。
2. **写影响了此前的读**：事务读取某谓词后，另一个事务覆盖该行或插入 phantom；数据库用 read/predicate marker 记住谁读过什么。

单个 read-write dependency 并不一定不可序列化。SSI 寻找可能形成 cycle 的危险依赖组合，并选择事务 abort。实现可以有 false positive（本来可串行化却被中止），但不能漏过真正 anomaly。

### SSI 性能特征

读取不必阻塞，特别适合 read-heavy 与长一致性查询；与单线程串行相比，它不受单 CPU 核限制。性能依赖冲突率、事务长度和访问模式。长事务读取范围大、维持 marker 久，更可能造成 false positive 或阻碍清理。

工程上需监控 serialization failure/abort rate，并要求事务可安全重试。减少事务持续时间、缩小读写集合、建立精确索引可降低冲突。不要遇到 abort 就把 isolation 降级；那等于删除保证，应先定位热点不变量和访问路径。

## 工程延伸：把事务保证落实到业务边界

以下内容是由原书机制推导的生产实践，不是原文逐段翻译。

### 先写 invariant，再选隔离与约束

以票务为例，可写出：同一场次座位最多一个有效订单；支付成功必须对应订单；过期 hold 可被回收。实现优先级通常是：

1. unique/check/foreign key 等声明式约束。
2. 单语句原子更新，例如 `UPDATE ... WHERE status='AVAILABLE'` 并检查行数。
3. 明确行锁或 serializable transaction。
4. 只有跨系统且无法共享事务时，才引入 saga、补偿与对账。

约束越靠近权威数据，越不容易被另一个代码路径绕过。仅在 service 层先查再写，通常存在 TOCTOU race。

### 事务重试必须包住完整决策

serialization failure 或 deadlock 后，不能只重放最后一条 UPDATE，因为之前读取的前提已过时。重试闭包必须从事务开始重新读取并决策：

```text
for attempt in bounded_backoff:
  begin
  read current state
  validate invariant
  write
  commit
```

闭包内部不能直接发送邮件、扣第三方款或生成不可重复随机结果。把待发送事件写入同一事务的 outbox，提交后异步发送；外部 API 使用稳定 idempotency key。

### 用观测验证隔离假设

应记录 deadlock、lock wait、事务时长、serialization abort、vacuum lag、oldest snapshot 与重试次数。压测不能只做单线程 happy path，要用并发 schedule 验证 invariant：同时抢最后一张票、同时创建同名用户、同时让两位医生下班。

Jepsen 类测试关注系统在网络分区与故障下的历史是否满足模型；数据库本地隔离测试则可用 barrier 精确编排交错。最终断言应检查业务不变量，而不只是 HTTP 状态码。

## 常见误区

1. **ACID consistency 等于副本强一致**：前者主要指应用不变量，后者是分布式读写可见性模型。
2. **Snapshot isolation 就是 serializable**：SI 防 read skew，却通常允许 write skew 和 phantom 导致的不变量破坏。
3. **事务里查过不存在，就能安全插入**：并发事务可能同时看到空集；需要 unique constraint、predicate/range lock 或 SSI。
4. **`SELECT FOR UPDATE` 能锁住空结果**：普通行锁没有对象可锁；需范围锁、哨兵行或 serializable。
5. **Last-write-wins 防止 lost update**：它只是选择一个写并丢弃其他写。
6. **失败重试一定安全**：commit response 丢失与外部副作用会造成重复，需要幂等和结果查询。
7. **2PL 就是 2PC**：2PL 管隔离与锁；2PC 管多个参与者的原子提交。
8. **数据库标为 Serializable 就无需看文档**：产品命名和实现历史不同，还要验证 read-only、predicate、跨分区等边界。
9. **长事务只占一个连接**：它还可能持锁、保留 MVCC 旧版本、扩大 SSI read set，并拖累 vacuum 与尾延迟。

## 本章小结

- 事务用 atomicity、isolation、durability 把部分失败与并发交错收束成应用可处理的结果；业务 consistency 仍需正确不变量。
- Read committed 防 dirty read/write，但允许 read skew；snapshot isolation 用 MVCC 提供一致快照，却通常不是 serializable。
- Lost update 发生在同对象 read-modify-write，可用原子操作、锁、自动检测或 CAS；write skew 发生在不同对象共同约束，通常需要约束或 serializable。
- Phantom 的本质是写入改变了先前查询谓词，即使当时没有可锁的行。
- Serial execution、2PL 与 SSI 都能提供 serializability，但分别以单线程/分区约束、阻塞/死锁、乐观 abort/重试为代价。
- 工程实现应从 invariant 出发，把约束放到权威数据层，让完整事务可重试，并对锁等待、长快照和冲突率持续观测。
