## 从两个最简单的命令开始理解数据库

这一章试图回答一个看似朴素的问题：调用 `put(key, value)` 之后，数据库到底把数据放到哪里？调用 `get(key)` 时，又怎样把它找回来？

先不用 SQL、事务和分布式，一个最小 KV 数据库可以这样写：

```bash
#!/bin/bash

db_set () {
  echo "$1,$2" >> database
}

db_get () {
  grep "^$1," database | tail -n 1 | cut -d, -f2-
}
```

`db_set` 只做 append（追加），所以写入顺序、简单而且高效；同一个 key 更新时不覆盖旧值，而是追加新记录。`db_get` 从头扫描并取最后一次出现的值，所以读复杂度是 O(n)。

这段玩具代码已经包含真实存储引擎的两条主线：

1. **日志是可靠写入的基础形态**：顺序追加容易恢复，也能保留变更历史。
2. **索引用额外空间和写入成本换查询速度**：没有索引时，读取必须扫描全部数据。

后面的 LSM-Tree、B-Tree、列存和物化视图，本质上都是在不同访问模式下重新安排数据与索引。

## Hash Index：最短路径是记住每个 key 在哪

最简单的索引是在内存维护一个 hash map：

```text
key -> 最新记录在文件中的 byte offset
```

每次 append 后更新 offset；读取先查 hash map，再跳到文件对应位置。只要所有 key 能放进内存，点查接近 O(1)。这类思路适合 value 更新频繁、key 数量有限的场景。

但日志会无限增长，因此需要 segment（分段）：当前段达到阈值后关闭，开启新段；后台把旧段 compact/merge，只保留每个 key 的最新值。

```text
写入：active segment
       k1=v3
       k8=v2

旧段：seg-01  seg-02  seg-03
          \      |      /
             merge
               ↓
         compacted seg
```

实现并不只是一句“后台合并”：

- 删除需要写 tombstone，合并时才能真正丢弃旧值。
- 崩溃后内存 hash map 丢失，可扫描段重建，或保存索引快照缩短恢复。
- 写到一半的记录需要 checksum/长度字段识别并截断。
- 多线程读写要保证切段和合并期间文件引用仍然有效。
- compacted segment 必须先写临时文件，完整 fsync 后再原子替换，避免暴露半成品。

### 为什么追加写不等于浪费

Append-only 看似保留重复值，但顺序写通常比随机覆盖更适合磁盘，也简化崩溃恢复。合并可以在后台批量回收空间。不可变段还让并发读无需担心原地修改，并且旧文件可被安全备份。

真正的代价是内存索引必须容纳全部 key；range scan 也不自然，因为 hash 不保序。这推动了下一步：让磁盘段按 key 排序。

## SSTable 与 LSM-Tree：把有序性变成能力

**SSTable（Sorted String Table）** 要求每个 segment 内记录按 key 排序，并且每个 key 在一个段中只出现一次。

有序段带来三个重要能力：

1. 合并多个段可以像归并排序一样顺序扫描，不需要把全部数据载入内存。
2. 内存索引只需保存 sparse index，例如每隔若干 key 记录一个 offset。
3. 相邻 key 位于相邻磁盘块，天然支持 range scan 和 block compression。

### Memtable：写入如何保持有序

随机写入不会天然按 key 排序，所以先写入内存有序结构，常见实现是红黑树、AVL Tree 或 Skip List。这个结构叫 **memtable**。

```text
Put
 │
 ├──> WAL（先保证崩溃恢复）
 │
 └──> Memtable（内存中按 key 排序）
          │ 达到阈值后冻结
          ▼
       SSTable（顺序刷盘）
          │
          ▼
      Background Compaction
```

读取按“新到旧”查找：active memtable、frozen memtable、较新的 SSTable、较旧层级。找到第一个值或 tombstone 即可。

WAL（Write-Ahead Log）不能因为 SSTable 已有日志语义就省掉：memtable 在内存中，机器崩溃会丢失尚未 flush 的写。WAL 通常只用于恢复，flush 完成后对应部分可删除。

### Compaction：LSM 的成本中心

如果每次读都查所有 SSTable，读放大会不断增长。Compaction 合并段、清理被覆盖的旧版本和 tombstone，并维持层级组织。

常见策略：

- **Size-tiered compaction**：把大小相近的 SSTable 合并成更大文件。写放大较低，但同一 key range 可能存在多个文件，读放大和空间放大较高。
- **Leveled compaction**：每层限制大小，同一层的 key range 尽量不重叠。读路径更可预测，但一个文件可能反复与下一层多个文件合并，写放大更高。

生产上最重要的不是背策略名字，而是监控 compaction debt：

| 指标 | 说明 | 恶化时的表现 |
| --- | --- | --- |
| L0 file count | 尚未整理的最年轻文件数量 | 点查要打开更多文件，最终 write stall |
| Pending compaction bytes | 为恢复层级约束还需重写的数据 | 后台债务正在增长 |
| Read amplification | 一次逻辑读触碰的文件/块数 | P99 读延迟升高 |
| Write amplification | 每个逻辑字节实际写盘次数 | SSD 带宽和寿命受压 |
| Tombstone age | 删除标记滞留时间 | 空间不释放，过早清理又可能数据复活 |

Bloom Filter 可以快速判断“某个 SSTable 肯定没有这个 key”，减少无效磁盘读取；block cache 缓存热点块。但二者不能治愈无限增长的 compaction debt。

## B-Tree：把磁盘组织成可原地更新的页

B-Tree 把数据库拆成固定大小 page（常见几 KB），每个页保存一段 key range 和指向子页的引用。查找从 root page 开始，逐层缩小范围，直到 leaf page。

```text
                 [root]
           /        |        \
      < k100    k100-k500    > k500
       / \         / \          / \
    leaf leaf    leaf leaf     leaf leaf
```

每个叶页只要没有满，可以原地更新。页满时分裂为两个页，并更新父页；树高度保持平衡，因此查询复杂度为 O(log n)，且大多数数据库三四层就能覆盖很大的数据量。

### WAL、页分裂和并发

原地更新带来崩溃风险：若页只写了一半，树可能损坏。因此 B-Tree 通常也有 WAL，先把预期修改追加到日志，再修改数据页；恢复时重放 WAL。

页分裂会同时修改多个页，并发线程还可能同时访问这些页，因此需要 latch 等轻量锁保护内部结构。Copy-on-write B-Tree 则写新页并原子切换 root，简化快照和并发读，但会产生额外写入。

### B-Tree 的常见优化

- 页只存 key 前缀，提升 branching factor，降低树高。
- 把相邻 leaf page 放在连续磁盘区域，帮助 range scan。
- 增加 sibling pointer，让顺序扫描无需反复回 root。
- 根据工作负载调整 fill factor，为未来插入预留空间。

## LSM 与 B-Tree 不是“谁更快”的问题

| 维度 | LSM-Tree | B-Tree |
| --- | --- | --- |
| 写入形态 | WAL + 内存排序 + 批量顺序刷盘 | WAL + 页内/页分裂更新 |
| 点查 | 可能检查多个层，依赖 Bloom/cache | 路径稳定，通常少量页读取 |
| 范围扫描 | SSTable 有序，适合合并扫描 | 叶页有序并通过 sibling 遍历 |
| 写放大 | 来自多轮 compaction | 来自 WAL、整页写和页分裂 |
| 读放大 | 文件/层级重叠时较高 | 树高决定，通常稳定 |
| 空间放大 | 多版本和 compaction 暂存 | 页预留空间与碎片 |
| 尾延迟风险 | compaction 与前台争 I/O、write stall | 页锁竞争、随机 I/O、checkpoint |

LSM 往往能提供较高写吞吐，因为顺序批量写减少随机 I/O；压缩后的 SSTable 也可能更省空间。B-Tree 点查与更新延迟通常更可预测，而且一个 key 的最新值只存在一个主要位置。

但结果高度依赖存储介质、缓存、数据大小、压实策略和实现质量。正确的选型方式是用真实 workload 回答：

- 点查、范围扫、写入和删除各占多少？
- key/value 大小及压缩率如何？
- 可接受的 P99 和恢复时间是多少？
- 热点是否能放入内存？
- 后台 compaction/checkpoint 与线上流量如何隔离？

## 不止主键索引：索引服务的是查询，不是表

### Secondary Index

主键唯一定位记录；secondary index 允许按非主键字段查找。索引值可能对应多条记录，因此通常保存记录 ID 列表或指针。

索引可以是：

- **Clustered index**：行数据直接按索引顺序存储。
- **Nonclustered index**：索引保存指向 heap/主键的引用。
- **Covering index**：在索引中附带查询所需的额外列，避免回表。

每个额外索引都会增加写放大、存储和迁移成本。线上“查询慢就加索引”若没有预算，最终会让写入、备份和 schema 变更越来越重。

### Multi-column 与特殊索引

复合索引的列顺序决定可使用的查询前缀。例如 `(tenant_id, created_at)` 适合某租户内按时间范围读取，但不能高效支持只按 `created_at` 查所有租户。

全文索引处理词项、前缀和相似度；内存数据库利用数据全驻内存实现不同结构；事务外的分析系统则需要另一套完全不同的布局。

## OLTP 与 OLAP：访问模式决定存储布局

**OLTP（Online Transaction Processing）** 通常面向用户：请求量大，但每次只读写少量记录，通过 key 或索引定位，延迟要求低。

**OLAP（Online Analytical Processing）** 面向分析：单次查询扫描大量行，却只使用少数列，常做聚合和 join，对吞吐与总执行时间更敏感。

把复杂分析直接压在交易主库上会制造资源争抢和查询计划风险。因此数据仓库通常从 OLTP 系统抽取事实，形成独立的分析副本。它不是“更大的业务数据库”，而是围绕分析访问模式重新组织的数据系统。

### Star Schema：事实与维度

星型模型中心是 fact table，例如销售事件；周围是 dimension table，例如日期、商品、门店、客户和促销。

```text
             dim_date
                 |
dim_store -- fact_sales -- dim_product
                 |
            dim_customer
```

事实表可能有万亿行但每行相对窄；维度表行数少但字段丰富。Snowflake schema 进一步规范化维度，但 star schema 常因分析者更容易使用而受到偏好。

## Column-oriented Storage：只读取查询真正需要的列

行存把一行所有字段放在一起，适合一次读取/修改完整记录。分析查询常扫描海量行但只需要 4-5 列；若表有 100 列，行存会加载大量无用数据。

列存把同一列的值连续保存：

```text
date_key:    20260101, 20260101, 20260102, ...
product_sk:  31,       68,       31,       ...
quantity:    2,        1,        5,        ...
price:       9.9,      19.0,     9.9,      ...
```

所有列必须保持相同的行顺序，才能用第 k 个位置重建一行。查询只加载过滤、join 和聚合需要的列，大幅降低 I/O。

### 压缩与向量化

同一列通常类型相同、重复值多，适合 run-length、dictionary、delta 和 bitmap 编码。低基数列可以为每个取值保存 bitmap：

```text
WHERE product IN (30, 68, 69)
=> bitmap(30) OR bitmap(68) OR bitmap(69)
```

```text
WHERE product = 31 AND store = 3
=> bitmap(product=31) AND bitmap(store=3)
```

压缩不仅节省磁盘，也让更多值进入 CPU cache。查询引擎可以对一批紧凑值执行 SIMD/向量化操作，减少逐行函数调用和分支判断。

> [!note] Column family 不等于列式存储
> Cassandra/HBase 的 column family 主要是一种逻辑分组；同一行的列仍常一起存储。它和分析数据库“同一列连续存放、可独立读取和压缩”的 columnar layout 不是一回事。

### 排序、写入与多个副本

列存可以按常用过滤列排序。例如按 `date_key, product_sk` 排序后，时间范围查询只扫描相关区域，相同值连续出现也提高压缩率。

每列不能独立排序，否则第 k 个位置不再属于同一行；排序单位仍是整行，只是物理上按列保存。不同副本还可以采用不同 sort order，为不同查询提供类似多组 clustered index 的效果。

压缩列不适合原地插入。常见方案再次借鉴 LSM：新写先进入内存/小型行存 delta，积累后批量合并进列式文件；查询同时读取主列存与 delta，由引擎合并结果。

### Materialized View 与 Data Cube

如果同一聚合被频繁查询，可以把结果预计算为 materialized view。Data cube 则按多个维度保存聚合，使“昨天每家门店总销售额”无需扫描原始事实。

预计算用写入和存储换查询速度，也牺牲灵活性。若 cube 没有“价格区间”维度，就无法回答“高于 100 元的商品占比”。因此仓库通常保留尽可能多的原始事实，把物化聚合当加速层，而非唯一数据。

## 工程现实：存储引擎需要前后台资源治理

一个数据库 P50 很好、P99 周期性抖动，常不是查询算法突然改变，而是后台工作与前台争夺资源：

- LSM compaction 占满磁盘带宽。
- B-Tree checkpoint/WAL flush 制造写入尖峰。
- 全表备份或索引构建污染 page cache。
- 大查询把热 OLTP 页从缓存淘汰。
- 副本追赶、再均衡与线上复制共用网络。

因此容量规划不能只给用户流量留空间，还要给恢复和维护留预算。一个只在“无故障、无迁移、无备份”时满足 SLO 的集群，并没有真正满足生产容量。

建议把资源分为三个预算：

```text
前台请求预算 + 必要后台维护预算 + 故障恢复余量 <= 可持续资源
```

遇到 compaction debt 时，盲目降低后台优先级可能暂时改善写延迟，却让文件越来越多，最终触发更长 write stall；盲目加速 compaction 又可能压垮前台读。正确方案是 admission control、分层优先级、限速与足够容量共同工作。

## 常见理解偏差

1. **顺序写永远快于随机写**：结果受 SSD、fsync、并发、放大和实现影响，不能脱离 workload。
2. **LSM 写快、B-Tree 读快就够了**：尾延迟、范围查询、恢复、空间放大和运维成熟度同样重要。
3. **Bloom Filter 能解决读放大**：它只能跳过不存在 key 的文件，无法替代健康 compaction。
4. **列存只是在文件上转置表格**：真正收益还来自列裁剪、编码、排序、CPU cache 和向量化执行。
5. **索引越多查询越快**：索引同时增加写入、存储、缓存、备份和迁移成本。
6. **OLAP 副本可以直接读主库**：资源隔离与不同存储布局往往比“少维护一个系统”更重要。

## 本章小结

- Append-only log 让写入与恢复简单，索引则用额外成本换读取性能。
- SSTable/LSM 通过内存排序、不可变文件和 compaction 获得高写吞吐；B-Tree 通过页和 WAL 提供稳定点查与原地更新。
- 评价存储引擎要同时看读、写和空间放大，以及后台维护对尾延迟的影响。
- OLTP 与 OLAP 的访问模式不同，因此行存、列存、排序、压缩和物化聚合各有明确适用边界。
