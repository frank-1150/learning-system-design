## 复制真正复制的不是数据，而是变化的顺序

进入 Part II，原书把问题从单机提升到多机。分布式数据库常见的动机有三个：用更多机器承载负载；在机器、网络或机房故障时继续服务；让数据靠近全球用户以降低延迟。实现这些目标有两种基本手段：**replication（复制）**把同一份数据放到多个节点，**partitioning（分区）**把不同数据放到不同节点。本章先假设每台机器都能放下完整数据，只讨论复制；下一章才把两者组合起来。

静态数据的复制只是拷贝文件。真正困难的是数据持续变化时，如何让多个副本处理相同的变化：

```text
客户端写入 W1、W2
       │
       ├─ 节点 A 观察到 W1 → W2
       ├─ 节点 B 因网络延迟只看到 W1
       └─ 节点 C 先收到重传的 W2，稍后才收到 W1
```

系统必须回答：谁可以接受写？写到几个副本才向客户端确认？节点恢复后如何追上？两个地方同时修改同一对象时，哪个结果正确？因此复制协议本质上是在分布式环境里管理**变化的顺序、确认边界和冲突语义**。

本章讨论三类主流模型：single-leader、multi-leader 和 leaderless。它们不是产品标签，而是三组不同的权衡。选型时不要只问“支不支持多副本”，而应问写路径如何工作，以及故障时承诺会怎样变化。

> 本章讲解对应 DDIA Chapter 5，原书印刷页 151–191（当前拆分 PDF 第 7–47 页）；开头的 Part II 导言对应原书印刷页 145–149。下文是按原书论证链重新组织的中文精讲，并补充现代生产系统的实现与观测方法，不是逐段翻译。

## Single-leader：用唯一写入顺序降低复杂度

> 原书依据：Leaders and Followers、Synchronous Versus Asynchronous Replication、Setting Up New Followers、Handling Node Outages 与 Implementation of Replication Logs，印刷页 152–160。

在 **leader-based replication（基于领导者的复制）**中，一个副本是 leader（primary），其余是 follower（secondary/read replica）：

1. 客户端把写请求发给 leader。
2. leader 先把变化写入本地，并按确定顺序追加到 replication log。
3. follower 拉取或接收日志，按相同顺序应用到自己的本地状态。
4. 读请求可由 leader 或 follower 承担，写请求只能由 leader接受。

```text
                         ┌── log ──> Follower A ──> read
write ──> Leader ────────┤
            │            └── log ──> Follower B ──> read
            └── read
```

这个模型的重要价值不是 leader 这台机器更强，而是它为所有写提供一个串行化位置。对同一个 key 的两次更新先后抵达 leader 后，follower 只需重放相同顺序。代价则是 leader 成为写入路径的协调点，故障切换也必须谨慎建立“新的唯一 leader”。

### 同步与异步：确认成功究竟代表什么

leader 把日志发给 follower 时，可以等待 follower 确认，也可以不等。

- **Synchronous replication**：leader 只有在同步 follower 持久化写入后才确认。优点是 leader 突然丢失时，至少另一个副本确定拥有这次写；缺点是同步副本不可用或跨地域网络变慢时，写延迟和可用性直接受影响。
- **Asynchronous replication**：leader 本地完成就确认，复制在后台发生。写延迟低，单个 follower 变慢不阻塞写入；但 leader 在日志尚未传走时永久损坏，已经向客户端确认的数据仍可能丢失。

工程中常见 **semi-synchronous（半同步）**：多个 follower 中至少一个同步，其余异步。这样单个同步 follower 故障时，可以把另一个已追上的 follower 提升为同步角色，避免所有副本都阻塞写。注意，“三副本”本身没有说明耐久性；关键是确认时写到了哪里。

跨地域部署尤其需要明确确认边界。若用户在美国西岸，leader 也在西岸，而同步副本在欧洲，每次写至少承担一次洲际 RTT。把欧洲副本改为异步能改善延迟，却改变了区域灾难时的 RPO。这里没有免费方案，产品必须在**写入延迟、可用性、可接受数据损失**之间做显式选择。

> [!warning] `200 OK` 不天然等于“绝不会丢”
> 应把 durability contract 写清楚：确认代表仅进入 leader 内存、写入 leader WAL、一个可用区持久化，还是跨可用区/跨地域持久化。否则故障复盘时，应用与数据库团队会使用完全不同的“成功”定义。

### 新 follower 不能简单复制正在变化的数据目录

给运行中的数据库增加副本时，直接复制磁盘文件可能得到不一致快照：复制过程中数据仍在变化，各文件来自不同时间点。可靠流程通常是：

1. 在不阻塞长期写入的情况下取得 leader 的一致性 snapshot。
2. 记录 snapshot 对应的 replication log 位置，例如 PostgreSQL LSN 或 MySQL binlog position。
3. 把 snapshot 传到新 follower 并恢复。
4. 从记录的位置开始追赶随后产生的日志。
5. 追到足够接近 leader 后，才让它承担读流量或成为候选 leader。

工程现实中，bootstrap 往往比协议图更棘手。传输数 TB 快照会占满 leader 磁盘带宽和网络；追赶期间新日志增长速度若超过回放速度，副本永远无法上线。因此应从备份或对象存储播种、限速传输，并观测 snapshot 年龄、剩余日志量、apply throughput 和预计追平时间。

### follower 故障：从 checkpoint 继续追赶

每个 follower 通常在本地保存已处理日志的位置。重启后，它向 leader 请求断点之后的变化并继续重放。只要 leader 仍保留所需日志，恢复相对直接。若落后太久、日志已被截断，则只能重新做 snapshot。

这解释了一个容易忽略的容量关系：日志保留时间必须覆盖可接受的最长 follower 中断时间，以及备份和恢复耗时。为了省磁盘把 WAL 只保留一小时，可能让两小时网络故障退化成数小时全量重建。

### leader 故障：failover 是风险最高的普通操作

leader 不可用时，系统需要 failover：判断旧 leader 已失败、选择新 leader、让客户端改写新地址，并确保旧 leader 回来后降级为 follower。

每一步都有不确定性。节点“没有响应”可能只是网络隔离或长时间 GC pause；超时太短会频繁误切，太长则延长不可写时间。新 leader 通常选择复制进度最靠前的 follower，但若复制是异步的，它仍可能缺少旧 leader 已确认的写。旧 leader 恢复后，系统可能丢弃它未出现在新 leader 上的日志，这会产生数据丢失，甚至让外部系统中的自增 ID 与数据库重新生成的 ID 冲突。

最危险的是 **split brain**：两个节点都相信自己是 leader 并接受写。仅靠心跳不足以消除它，因为处于网络分区两侧的节点都可能健康。生产系统常使用多数派租约、共识系统或 **fencing token** 保证旧 leader 即使继续运行，也无法再修改受保护资源。自动 failover 不是一个开关，而是一套需要定期演练的状态机。

## Replication log：传输 SQL、行变化还是存储引擎日志

“复制日志”可以位于不同抽象层，选择会影响兼容性和正确性。

### Statement-based replication

把 leader 执行的 SQL 语句发给 follower。它直观且紧凑，但非确定性函数（`NOW()`、`RAND()`）、自增序列、触发器和依赖当前行顺序的语句可能在各副本得到不同结果。数据库可以限制或改写不安全语句，但边界复杂。

### Write-ahead log shipping

直接复制存储引擎的 WAL。B-Tree 页修改或 LSM/SSTable 变化会在物理日志中精确重放，副本与 leader 字节级接近。但日志与存储格式强耦合，滚动升级可能受限：新旧版本若不理解彼此 WAL，就不能自由共存。

### Logical row-based log

使用与存储引擎解耦的逻辑日志，例如“表 T 中主键 42 的列 x 从 A 改成 B”。它通常比 SQL 更确定，又比物理 WAL 更适合作为外部数据变化流。Change Data Capture 可以用它驱动搜索索引、缓存、审计和数据仓库，但下游仍需处理重复、断点、schema 演化和事务边界。

### Trigger-based replication

触发器允许应用在数据变化时执行自定义逻辑，将变化写入独立表或外部系统。灵活性最高，适合只复制部分数据或接入旧系统；代价是额外开销、更复杂的故障语义，以及业务逻辑藏在数据库中后更难测试。若用 trigger/outbox 产生事件，必须让业务写与 outbox 写处于同一事务，并让消费者幂等。

## Replication lag：eventual consistency 不是一个完整承诺

> 原书依据：Problems with Replication Lag，印刷页 161–167。

异步 follower 扩展了读取吞吐，却带来 **replication lag（复制延迟）**。在负载平稳时 lag 可能只有毫秒；遇到大事务、网络抖动、锁等待或 follower I/O 饱和，它可能突然增长到分钟乃至小时。“最终一致”只说停止写入后副本最终会收敛，没有说明收敛多久，也没有描述用户会看到哪些异常。

原书把常用的读体验拆成三种更精确的保证。

### Read-your-writes：至少看到自己的修改

用户提交评论后刷新页面，若读请求落到尚未追上的 follower，评论会像消失一样。**read-after-write consistency** 保证用户总能看到自己刚写入的内容，但不要求立即看到其他人的写。

实现方式可以是：用户自己的可编辑数据读 leader，公开数据读 follower；写后短时间把该用户读请求粘到 leader；写响应返回 commit position，之后只读 replay position 不小于它的副本。最后一种更精确，却要求跨设备会话携带水位，并处理 follower 永远追不上的超时与回退。

仅用墙上时钟判断“写后一分钟读 leader”容易受时钟偏差和长 lag 影响。更可靠的是使用单调递增的日志位置或版本号。

### Monotonic reads：时间不要倒流

第一次读落到较新的 follower A，第二次读落到较旧的 follower B，用户会先看到新评论，再看到它消失。**monotonic reads** 保证同一用户后续不会看到比此前更旧的状态。

常见实现是 session affinity：按用户 ID 哈希到固定副本。若该副本不可用，切换时仍需携带用户已见的最低版本，或回退到 leader，否则故障切换本身会破坏单调性。

### Consistent prefix reads：因果顺序不要颠倒

观察者先看到回答、稍后才看到问题，说明两个因果相关的写被以相反顺序读取。单分区 leader 通常提供统一顺序；分区数据库中，相关记录若进入不同分区，各分区复制延迟不同，就可能颠倒。

**consistent prefix reads** 保证如果一组写存在先后关系，读取不会看到后果而遗漏原因。可以让因果相关的写落在同一分区，或显式跟踪 causal dependency；代价是分区自由度、元数据或等待时间。

这些保证揭示了一个工程原则：不要用一个模糊的“允许 eventual consistency”结束设计。应按用户旅程写明哪些读取必须 read-your-writes，哪些需要 monotonic reads，哪些跨实体关系需要因果前缀，超时时回 leader、等待还是显示“同步中”。

## Multi-leader：多个写入口换来显式冲突

> 原书依据：Multi-Leader Replication，印刷页 168–176。

single-leader 的自然扩展是让多个副本都能接受写，每个 leader 再把写异步复制给其他 leader。多主并非单主的无痛高可用版本；它把 leader 故障和网络延迟转化为**并发写冲突**。

### 多数据中心

每个数据中心设置本地 leader，中心内同步或快速复制，中心间异步复制。用户写本地机房，减少跨洲延迟；一个数据中心失效时，其他中心仍能接受写；中心间网络中断时，各地可独立运行。

代价是同一记录可能在不同中心同时变化。数据库的自增键、唯一约束、触发器和外键可能只在本地成立，跨中心合并后才发现冲突。网络恢复不等于状态自动正确，冲突解决必须成为数据模型的一部分。

### 离线客户端与协同编辑

手机、笔记应用或日历可以把每台设备视为本地 leader：离线时继续写，联网后异步同步。协同编辑也允许多个参与者低延迟更新同一文档。这些场景无法靠“只允许一个在线 leader”获得良好体验，因此更需要 operation transformation、CRDT 或领域级合并规则，而不是简单覆盖整个对象。

### 如何处理写冲突

两个 leader 分别把标题改成 A 和 B；因为冲突通常只在异步复制到达后才被发现，无法像单机事务那样立即让第二个写 abort。系统必须最终收敛，否则各副本会永久分叉。

可选策略包括：

- **Avoidance**：把同一用户或实体的所有写路由到固定 home leader。简单有效，但 home region 故障或用户迁移时仍会暴露冲突。
- **Last write wins**：给每个写附时间戳并选“最新”。它能收敛，却可能静默丢弃一个成功写；物理时钟偏差还会让“最后”不可靠。
- **确定性 winner**：按 leader ID、版本等规则选择一个值。保证收敛不等于保留意图。
- **合并值**：保留集合并集、最大值、追加日志，或将冲突值作为 siblings 交给应用合并。
- **自定义 resolution**：写入时在后台运行冲突处理，或读取时把多个版本返回给客户端。处理逻辑必须确定、可重试，并对旧版本代码兼容。

原书提醒：某些冲突没有通用答案。例如两个会议室预订若分别在两个 leader 通过，合并字段无法恢复“不可重复预订”的不变量。对于强唯一性、余额不可为负等约束，应把相关写协调到一个权威位置，或使用共识/事务，而不是寄希望于事后合并。

### 复制拓扑决定故障与循环

多主可以采用 circular、star 或 all-to-all topology。环形和星形连接少，但单点或单链路故障会阻断部分传播；全连接容错更好，却可能让同一写沿多条路径到达，并出现不同写在不同节点上顺序不一致。

每个写需要唯一 ID 和来源信息，节点看到已处理 ID 时不能再次转发，否则会无限循环。拓扑还应观测每个 peer 的发送/接收水位，而不是只看“复制线程正在运行”。

## Leaderless：让客户端与多个副本协作

> 原书依据：Leaderless Replication，印刷页 177–191。

Dynamo 风格系统去掉 leader。客户端或 coordinator 同时把写发到多个副本，并行从多个副本读，再根据版本选择或合并结果。节点故障时无需选主，写路径也没有唯一中心，但冲突检测被带到了每次读写中。

假设一个 key 有 `n` 个副本，成功写至少等待 `w` 个，读取至少询问 `r` 个。若：

```text
w + r > n
```

那么在理想前提下，读集合与最近成功写集合至少相交一个节点。例如 `n=3, w=2, r=2`。这常被称为 quorum read/write。

但公式不是“强一致性证明”。它依赖严格 quorum、正确识别新旧版本、写集合仍属于该 key 的指定副本等前提。并发写可能各自只到达部分节点；读与写并发时，不保证读到哪个值；节点替换、时钟式 LWW、故障恢复和 sloppy quorum 都会改变结果。即使 `w+r>n`，系统通常也不是线性一致的。

### 节点宕机时如何补齐

- **Read repair**：读请求发现某个副本返回旧版本，在返回结果同时或之后把新版本写回旧副本。它只修复经常被读到的 key。
- **Anti-entropy**：后台进程持续比较副本数据，例如用 Merkle tree 缩小差异范围，并复制缺失值。它覆盖冷数据，但收敛时间取决于扫描和修复速度。

生产中需要同时监控两个过程。只做 read repair 会让长期不读的合规数据永远分叉；只做低频 anti-entropy，则热 key 的错误窗口过长。可观测指标应包括版本分歧率、repair backlog、Merkle range mismatch、最大对象年龄，而不仅是节点存活。

### Sloppy quorum 与 hinted handoff

严格 quorum 只在该 key 指定的 `n` 个 home nodes 中计数。若网络分区导致其中多数不可达，可以选择拒绝写，保持更强的读取相交保证；也可以使用 **sloppy quorum**，把写暂存到附近但不属于 home set 的健康节点，并留下 hint。home node 恢复后，暂存节点通过 **hinted handoff** 把写送回。

这提高了写可用性，却意味着 `w+r>n` 的集合相交暂时不成立：读 home nodes 可能完全看不到暂存写。sloppy quorum 的承诺更接近“先接受，之后收敛”，不应被描述为严格 quorum。hint 还必须有磁盘持久化、过期策略和背压，否则长期故障会挤爆代偿节点。

多数据中心 leaderless 系统通常把每个写并行送到本地若干副本，并异步送往远端。它绕过选主，却没有绕过 WAN 延迟、冲突或跨地域 RPO。

## 并发写：没有共同顺序时，系统必须保留因果关系

无主复制中，两个客户端可同时更新同一个 key。网络到达顺序不同，各副本可能最终值不同。若简单使用 last-write-wins，以客户端时间戳最大者覆盖，其余已确认写会被丢弃。LWW 适合“只保留一个值且丢写可接受”的缓存类场景；对购物车、权限集合或用户创作内容往往过于粗暴。

更准确的模型是 **happens-before（先发生于）**：若操作 B 知道 A 的结果并在其基础上发生，则 A happens-before B，B 可以覆盖 A；如果 A 与 B 都不知道对方，则它们是 concurrent，不能仅靠到达顺序声称谁更新。

一个简化流程是：

1. 服务器为每次写分配版本，并记录它基于哪些旧版本。
2. 客户端读取时拿到值及 version context。
3. 客户端写回时携带该 context。
4. 服务器可以删除被 context 覆盖的旧值，但必须保留未被覆盖的并发 siblings。
5. 读到多个 siblings 的客户端按领域规则合并，再把合并结果写回。

例如两个设备都从购物车 `{牛奶}` 出发，一个加入面包，一个加入鸡蛋。两个加入是并发的，正确合并可能是集合并集 `{牛奶, 面包, 鸡蛋}`。但删除操作不能直接从集合消失，否则旧副本稍后重放会让商品“复活”；需要 tombstone 或带因果信息的 remove 操作。CRDT 正是把这类合并规则形式化，但它同样需要明确业务语义与元数据成本。

### Version vector 不是一个标量版本号

单节点可用递增版本判断新旧，多副本则需要知道每个 replica 贡献了哪些更新。**version vector** 为每个副本保存计数，例如：

```text
V1 = {A: 3, B: 1}
V2 = {A: 2, B: 2}
```

若 Vx 每个分量都不小于 Vy，且至少一个更大，则 Vx 因果上晚于 Vy；若一边在 A 更大、另一边在 B 更大，它们不可比较，表示并发。version vector 允许系统区分“可安全覆盖的旧值”与“必须保留的并发值”。它与单个副本内部事件的 version number 配合使用，不应与 vector clock 概念随意混为一谈。

节点成员频繁变化会让 vector 元数据膨胀，因此真实系统可能截断、按虚拟节点聚合或使用 dotted version vector。任何压缩都要说明是否会把因果关系误判为并发，或反过来造成丢写。

## 把复制设计落实到工程契约

评审一个复制方案时，至少把下面问题写成可验证答案：

| 维度 | 需要明确的契约 |
| --- | --- |
| 写入确认 | 写到内存、WAL、几个 AZ、几个 region 才成功 |
| 读取路由 | leader、任意 follower、session 粘滞还是 quorum |
| 新鲜度 | read-your-writes、monotonic、consistent prefix 或最大 lag |
| 故障切换 | 谁判断故障、如何选主、如何 fencing、允许多少 RPO/RTO |
| 冲突 | 如何检测并发、合并规则、是否允许 LWW 丢写 |
| 修复 | snapshot、日志保留、read repair、anti-entropy 的时限 |
| 观测 | commit/apply position、字节与时间 lag、冲突率、repair backlog |

复制延迟最好同时用**日志距离**和**墙钟时间**表达。仅看“落后 10 MB”无法判断严重性：低流量时可能代表数小时，高流量时可能只有几秒。还需区分 receive lag、flush lag 与 apply lag，才能判断瓶颈在网络、磁盘还是回放锁等待。

## 常见理解偏差

1. **副本越多，数据越安全**：若写成功只落在一个异步 leader，五个 follower 也可能来不及收到。
2. **自动 failover 总能提高可用性**：错误故障判断和缺少 fencing 会把短暂抖动升级成 split brain。
3. **eventual consistency 就是什么都不保证**：可以按业务增加 read-your-writes、monotonic reads 和 causal prefix。
4. **`w + r > n` 等于线性一致**：并发、sloppy quorum、故障切换和版本选择仍会破坏直觉。
5. **LWW 是冲突解决**：它只是确定性丢弃一个值，是否正确取决于业务能否接受丢写。
6. **复制延迟只影响“旧一点”**：它会导致自己的写消失、时间倒流和因果颠倒，是不同的用户语义。
7. **网络恢复后副本自然一致**：没有 repair/anti-entropy 和可观测水位，分歧可能长期存在。

## 本章小结

- single-leader 用唯一写入顺序简化冲突，但同步/异步确认和 failover 决定真实耐久性。
- 异步读副本需要把 eventual consistency 拆成可验证的会话与因果保证。
- multi-leader 适合多地域、离线与协作写入，但冲突必须由数据模型明确解决。
- leaderless 通过 quorum、repair 与版本信息工作；`w+r>n` 有严格前提，不等于强一致性。
- 判断并发需要 happens-before 与 version vector；“最后到达”或“墙钟最新”不代表业务上正确。
- 复制的生产质量取决于确认契约、日志保留、fencing、修复速度和可观测性，而不只是副本数量。
