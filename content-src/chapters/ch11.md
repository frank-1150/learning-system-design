## 从 bounded data 到永不结束的计算

批处理面对的是有边界输入：文件集最终会读完，任务可以排序全部记录、产出文件并宣布完成。真实业务事件却持续出现。用户明天仍会下单，设备下一秒仍会报告温度，数据库下一次写入仍会改变搜索索引。若每天把事件切成一个 batch，派生结果最多落后一天；把切片缩短到一分钟只能降低延迟，不能改变“数据本质上没有结尾”。

**Stream processing** 是对 unbounded data 的增量处理。Event 是“某件已经发生的事”的不可变记录，通常包含业务标识、payload、schema version、事件时间和追踪信息。Producer 生成一次 event，多个 consumer 可以各自处理；topic/stream 则把同类事件组织起来。

```text
业务事实发生
   │
   ▼
event log ──> 风控 ──> 告警
   ├────────> 搜索索引
   ├────────> 实时指标
   └────────> 数仓/推荐
```

这一章不是教某个 Kafka API，而是建立一条统一主线：消息如何传输与保存，数据库变化如何成为流，流如何累积成状态，以及在乱序、迟到和失败下怎样让派生状态仍可解释、可重放。

## Messaging systems：先问慢消费者和故障

评价消息系统时，两个问题比吞吐数字更重要：

1. Producer 比 consumer 快时怎么办：drop、buffer，还是 backpressure？
2. Producer、broker 或 consumer 崩溃时，哪些 message 可能丢失或重复？

丢周期性温度采样也许可以接受；丢“支付完成”会留下永久错误。把 message 写磁盘并复制能提高 durability，却增加确认延迟与成本。语义必须由业务后果选择，不能用“我们是实时系统”替代。

### Direct messaging：低延迟，但生命周期强耦合

Producer 可直接用 UDP multicast、ZeroMQ、HTTP/RPC 或 webhook 把事件发给 consumer。路径短、组件少，适合行情、遥测或在线回调。但 consumer 离线期间谁保存消息？Producer 崩溃后谁继续重试？新增 consumer 如何补历史？这些责任通常回到应用。

UDP 行情可在应用协议中用 sequence number 检测 gap，并向恢复服务请求缺包；webhook 需要签名、幂等 key、持久 delivery attempt、退避与 dead-letter。所谓 direct 并不表示没有 broker 的职责，只是这些职责散落在 producer 和 consumer 中。

### Message broker：把时间解耦交给中间层

Broker 接收并保存 message，producer 等 broker 确认后即可返回；consumer 在未来某时处理。它让双方不必同时在线，并集中承担 durability、retry、subscription 与 flow control。

传统 queue broker 常在成功 ack 后删除消息，工作集通常较短。数据库则默认保留数据直到显式删除，并支持索引与查询。Broker 擅长通知变化，不擅长任意历史查询；数据库擅长当前快照查询，却通常不会主动告诉旧查询“结果已变化”。Log-based broker 正是在二者之间建立桥梁。

### Fan-out 与 load balancing 是两层不同语义

- **Fan-out**：每个独立订阅都收到全部事件，例如搜索、风控、数仓各自消费订单流。
- **Load balancing**：同一订阅组内每条事件只交给一个 worker，多个实例共同扩吞吐。

实践中通常组合：每个 consumer group 获得完整 topic；组内实例分担 partitions。不要用两个竞争 consumer 假装两个独立业务订阅，否则其中一方会随机拿不到消息。

### Delivery semantics 从 ack 时机产生

Consumer 崩溃可能发生在任意位置：

```text
receive ─> business write ─> ack
                  ▲ crash here：写成功但 ack 丢失，message 会重投
```

- **At-most-once**：先 ack 再处理，崩溃可能丢，但不由 broker 重投。
- **At-least-once**：处理成功后 ack，失败会重投，不丢但可能重复。
- **Exactly-once effect**：即使内部重试，外部可见效果仿佛只执行一次；需要原子提交或幂等，而不是一个消费参数。

Redelivery 还会破坏顺序。Consumer 2 处理 `m3` 时崩溃，Consumer 1 可能已处理 `m4`，随后才收到重投的 `m3`。若事件存在因果关系，要么让同一 key 固定进入同一有序 partition 并串行处理，要么在状态层使用 version/sequence 拒绝倒退。

## Partitioned log：把消息变成可重读数据

Append-only log 把每条 event 追加到尾部，consumer 顺序读取。为了扩展吞吐，一个 topic 切成多个 partitions；每个 partition 是独立有序日志，event 获得单调递增 offset。系统通常只保证 partition 内 total order，不保证不同 partition 的全局顺序。

```text
topic: orders
P0: [42 created][43 paid][44 shipped]   key=order-7
P1: [18 created][19 cancelled]          key=order-9
```

Partition key 决定语义边界。按 `order_id` hash 可确保同一订单事件有序；按随机 key 虽更均匀，却要求下游处理乱序；按 `tenant_id` 便于租户级状态，却可能产生大租户热点。增加 partition 数还可能改变 key mapping，若顺序边界不能跨迁移，要设计 epoch 或显式路由版本。

### Offset 把 acknowledgment 压缩成一个位置

顺序消费时，`offset < committed_offset` 可视为已处理，broker 不必保存每条 ack。Consumer group 为每个 partition 记录 offset；实例故障后，partition 交给新实例，从最后提交位置继续。

Offset 提交太早可能丢处理；提交太晚会重复。将 output/state 与 offset 同事务提交是 exactly-once 的基础之一。把 event 提交给线程池再立刻前移 offset 很危险：offset 50 的任务完成，不代表 49 已完成。若需要 partition 内并发，必须跟踪连续完成水位，而不能提交最大已完成 offset。

### Consumer group 的并行度与 head-of-line blocking

一个 partition 同一时刻通常只分给组内一个 consumer，所以有效并行度不超过 partition 数。单条慢事件会阻塞后续同 partition 事件。可增加 partitions、把重任务异步拆到另一 topic、或按 key 做细粒度 worker mailbox；但把同 partition 随意并行会丢失顺序和简洁 offset 语义。

Rebalance 期间 ownership 会变化。旧 consumer 可能仍运行，因此状态与 sink 写入应带 generation/epoch fencing；cooperative rebalance、合理 session timeout 和静态 membership 可减少频繁暂停，但无法代替 correctness。

### Retention 是一个有截止时间的恢复承诺

Log 以 segments 保存，超过时间或容量后删除旧段。消费不删除事件，使多个 group 可独立 fan-out，并允许回退 offset 重放。代价是慢 consumer 若落后超过 retention，就永久失去历史。

因此 lag 不只是性能指标，而是**数据丢失倒计时**：

```text
time_to_data_loss ≈ remaining_retained_bytes / current_ingress_bytes_per_sec
```

至少监控 per-partition lag records/bytes/time、最老未消费事件年龄、消费速率与生产速率、预计追平时间。只看 group 总 lag 会隐藏单个 hot partition。Retention 要覆盖最长可接受 outage、修复与 replay 时间，并留容量余量。

### Replay 是能力，也是一场受控迁移

Log 读取是非破坏性的，consumer 可从昨天 offset 重跑新代码并写入新 view。这让修 bug、构建新索引、审计历史比传统 destructive queue 容易。

但 replay 会带来生产级问题：历史吞吐远高于实时流量，可能压垮 sink；旧 schema 和已删除维表版本可能无法解析；event-time timer 会集中触发；外部邮件绝不能重发。安全做法是新 consumer group + 新 output namespace，限制速率，先 shadow compare，再原子切换读取；对于不可重复 side effect，回放管道必须显式禁用或由幂等 ledger 拦截。

## Backpressure：不要等 queue 撑爆才叫它流控

消费者落后时有三种根本选择：丢、缓存、让上游慢下来。TCP 的小 buffer 满后阻塞 sender 是局部 backpressure；log broker 用磁盘提供大而有限 buffer，producer 可以暂时不受 consumer 影响。但 buffer 只是把问题延后，不会消除长期速率差。

一条流处理 pipeline 的 backpressure 应逐 stage 传播：sink 慢使下游 operator 停止拉取，input consumer 降低 fetch，必要时 producer 限流。若每层都用无界内存 queue“解耦”，最终表现是 OOM 和漫长 GC，而不是弹性。

工程上要区分 transient burst 与 sustained overload：

- Burst 可用 buffer 吸收，容量按峰值持续时间估算。
- Sustained overload 必须增并行度、优化每条成本、降采样/舍弃低价值数据，或向 producer 施加 admission control。
- Poison event 应进入隔离队列，不能无限阻塞一个 partition；同时保存原事件、错误、代码/schema 版本和重试次数。

关键指标包括 operator busy/backpressured time、mailbox/queue depth、records in/out、serialization time、state I/O、checkpoint duration，以及 source-to-sink end-to-end latency。

## Databases and streams：状态变化本来就是一条流

数据库 replication log 是 write events 的有序流。Follower 按相同顺序应用确定变更，得到相同状态。把这个思想开放给外部 consumer，就能让数据库成为 cache、search、warehouse 等派生系统的 leader。

### Dual writes 为什么会永久分叉

应用先写数据库再写搜索，两个 concurrent clients 的顺序可能相反：数据库最终是 B，搜索最终是 A；即使没有错误也会不一致。若其中一次写失败，系统又面临跨系统 atomic commit。

```text
Client 1: DB=A ─────────────> Search=A
Client 2:      DB=B -> Search=B

DB order: A,B；Search order: B,A
```

“失败时重试第二步”只处理部分失败，不能解决并发重排；给每条写加 version 可让 sink 拒绝旧版本，但仍要保证每个已提交 DB change 最终进入传播流。

## Change Data Capture：让派生系统 follow 主库

CDC 从数据库的 WAL/binlog/replication log 捕获 row change，转成稳定 event，交给下游按同一顺序应用。搜索索引、缓存和数仓不再与主库并列接收应用写，而是消费主库已经提交的事实。

一条成熟 CDC 流至少包含 table/key、operation、before/after 或新值、transaction/commit position、schema 信息和 source metadata。Delete 需要 tombstone，否则 compaction 或 sink 不知道应删除旧记录。

### Initial snapshot 与增量必须无缝衔接

新 consumer 不能只从“现在”读 log，否则缺历史；也不能先随意 dump 全表再读 log，否则 snapshot 期间的写可能漏掉或重复。协议需要一致切点：

1. 记录 source log position `L0`，或开启与 snapshot 协调的捕获。
2. 在一致 snapshot 中扫描当前表，事件携带 snapshot 标记。
3. 从 `L0` 之后读取增量，按 key/version 去重或排序。
4. 完成 catch-up 后发布 watermark，才把新 sink 切入服务。

大表 snapshot 会持续很久，需分 chunk、限速并保存进度；schema change 与 DDL 也要与 data event 有确定顺序。若 WAL retention 小于 snapshot/catch-up 时间，source 会提前删除所需日志，任务必须重来。

### Log compaction 把 change stream 变成可恢复快照

按 primary key compact 的 log 保留每个 key 最新 value，并用 tombstone 表示删除。新 consumer 先读取 compacted history 重建当前 state，再追尾实时变化。Compaction 不等于只保留一条物理记录的瞬时操作；旧版本在后台清理前仍可能存在，tombstone 也必须保留足够久让所有副本和 consumer 看见。

### Outbox 解决“业务提交但没发事件”

若数据库本身没有合适 CDC 语义，应用可在同一个本地事务中写业务表和 outbox row，由 relay/CDC 发布 outbox：

```text
BEGIN
  UPDATE orders ...;
  INSERT outbox(event_id, aggregate_id, payload) ...;
COMMIT
```

这保证业务变更与“待发布事实”原子出现。Relay 仍可能重复发布，因此 event_id 和 sink 幂等不可省。不要把“事务提交后调用 broker”叫 outbox，那仍有进程 crash 的丢事件窗口。

## Event sourcing：保存业务事实，不只是行变化

CDC 通常由现有 mutable database 派生，事件描述“row 从 A 变 B”。Event sourcing 则由应用显式把领域事件作为 system of record，例如 `SeatHeld`、`PaymentAuthorized`、`ReservationCancelled`。当前状态是按顺序 fold 这些事实的 materialized view。

**Command** 是还可拒绝的请求；验证成功、持久提交后才成为 **event**。Consumer 看到 event 时不能说“这个座位其实不能订”并悄悄丢掉，因为其他 consumer 可能已经接受这个事实。需要强约束的验证必须在 command 转 event 前串行化/事务化；或用 `ReservationRequested`、`ReservationConfirmed/Rejected` 建模异步流程。

CDC latest row 可按 key compact，因为新值覆盖旧值；domain events 往往不能这样做。`MoneyDeposited` 与 `MoneyWithdrawn` 都参与余额推导，后来事件不使先前事实失真。为了加速恢复，可周期保存 snapshot `(aggregate_version, state)`，再从该 version 继续 replay；snapshot 是缓存，不是丢弃审计历史的理由。

Event schema 演化是长期承诺。Consumer 重放十年前 event 时仍要解析旧版本；可维护 upcaster 把旧 payload 转成当前内部模型，或让 handler 支持多个版本。不要在不保留原义的情况下原地重写历史，否则审计与可复现性会变化。

## State、stream 与 immutability 是同一数据的两种视图

Mutable state 是历史 event 随时间累积的结果；change stream 是 state 的变化序列。余额是 credit/debit 的 fold，当前购物车是 add/remove 的 fold，搜索索引是商品变更的投影。

```text
state(t) = fold(events <= t)
change stream ≈ state 随时间的差分
```

不可变 log 带来：可审计、可重算、可从同一事实派生多个 read models、可让新旧 view 并行比较。CQRS 将写入事实与查询表示分开，允许搜索按全文索引、Feed 按读优化 timeline、分析按列式聚合，而不强迫 system of record 同时适配所有访问模式。

代价也要正视：异步 view 存在 read-your-write lag；高 churn 历史会膨胀；GDPR 删除、密钥泄露或错误 PII 可能要求真正删除。追加 `UserDeleted` 不会让旧 event、backup、SSD block 和下游 copy 中的 PII 消失。实践中需数据最小化、PII tokenization/crypto-shredding、全链路 deletion workflow 和可审计 completion，而不是用 immutability 拒绝合规需求。

## Stream processing 的四类用途

1. **CEP/pattern detection**：长期保存规则，让事件经过规则，识别欺诈序列、设备异常等复杂模式。
2. **Stream analytics**：窗口计数、rolling average、percentile、趋势检测；Bloom filter、HyperLogLog 等近似算法是内存优化，不代表 stream 天生不准确。
3. **Materialized view maintenance**：消费 changelog，持续维护搜索、缓存、聚合表和应用 read model。
4. **Search on streams**：存储用户查询，让新文档逐一匹配，例如房源订阅或媒体监控；这是“query 固定，data 流过”，与传统搜索相反。

最终输出可以是另一个 stream、数据库/index，或直接面向人的告警。不同 sink 的可重放性差异巨大：写派生表可以覆盖重建，发邮件和执行交易则是不可撤销 side effect，必须独立设计审批、dedupe 和审计。

## Event time：处理“事情何时发生”，而不是“机器何时看见”

Stream job 重启后会快速处理 backlog。若按 processing time 统计 QPS，恢复瞬间会制造一个并不存在的业务尖峰。Event time 来自事件发生时间，重放同一输入应归入同一业务窗口；processing time 是 operator 实际看到事件的时间，适合运维 latency，而非历史语义。

还应区分 ingestion time（broker 接收时间）。移动设备可能离线数天且本地时钟错误：event time 接近用户行为但不可信，ingestion time 可信却很晚。可同时保存 device timestamp、server receive timestamp、clock offset/sequence，按业务选择并标注质量。

### Window 类型

- **Tumbling**：不重叠固定窗口，如每分钟订单量。
- **Hopping/sliding**：固定长度、按较短步长滑动，如每 10 秒计算过去 5 分钟。
- **Session**：同 key 事件间隔小于阈值归为会话，间隔过大关闭。

窗口越长，state 越大；滑动步长越小，更新成本越高。可用 pane 预聚合减少重复计算，但要保持合并函数正确。

### Watermark 是完成度估计，不是真相

系统无法证明某个 event-time window 再也没有迟到事件，除非 source 提供可证明的 progress。Watermark 通常表示“预计不会再看到 timestamp ≤ W 的正常事件”，由各 source partition 的进度合并；最慢或空闲 partition 可能拖住全局 watermark。

窗口在 watermark 越过 end 后触发，但仍需 late-data policy：

- 允许 lateness 内更新结果，发 correction/upsert/retraction。
- 超过阈值进入 late side output 或丢弃，并监控数量与业务影响。
- Sink 必须理解 changelog：同一 window 的后续更新不能简单 append 成两份最终结果。

Watermark 设得激进，结果快但修正多；保守则准确但延迟和 state retention 高。它是产品延迟与完整性的显式权衡。

## Stream joins：无界输入迫使系统保存时间相关状态

### Stream-stream window join

搜索曝光与点击按 session/request ID join。点击可能不发生，也可能先于曝光到达。Processor 为两侧维护有限窗口 state，任一事件到达时查另一侧；匹配则输出，窗口到期仍无点击则输出未点击。

窗口决定业务定义：1 小时后的点击算不算本次曝光？过短漏匹配，过长占 state 且延迟 negative result。Duplicate event 还会产生笛卡尔重复，需 event ID dedupe 或明确多对多语义。

### Stream-table join / enrichment

活动 event 按 user ID 查 profile。逐条远程查数据库延迟高且会把 stream 峰值传给 OLTP；更常见是通过 CDC 在 operator 本地维护 table replica，再本地 join。

关键问题是**哪个时间版本的 profile**。若只保留最新 profile，重放旧活动会被今天的国家、套餐或税率 enrichment，结果不可复现。需要 as-of join 时，event 应携带 dimension version，或 table 保存 effective-from/to 的历史版本。

### Table-table join / materialized view

Tweets 与 follows 两张变化表持续 join，维护每个用户 timeline。Tweet 新增/删除、follow/unfollow 都会产生 view 更新。它本质是 SQL join 的增量维护，而非一次查询。

不同 input partitions/streams 没有共同全序，接近同时的变化可能以不同 interleaving 到达，导致 nondeterministic result。用业务 version、effective time、单一 ordered log 或 temporal table 明确时间语义，才能让 replay 稳定。

## Fault tolerance：exactly-once 必须沿数据流逐段成立

流永不结束，不能等 job 完成后一次发布输出。两种常见恢复方法：

- **Microbatch**：把短时间输入作为小 batch，成功后提交。
- **Checkpoint/barrier**：周期性为所有 operator state 和 source position 形成一致快照，不强制业务窗口等于 checkpoint 周期。

恢复时从最近 checkpoint 还原 state，并 replay 之后 input。Input event 实际可能执行多次；所谓 exactly-once 更准确是 **effectively-once visible effect**。

### Checkpoint 的一致切面

Checkpoint 不能只是“每个 task 随便存一次内存”。Barrier 沿 input 流传播，operator 在一个逻辑边界捕获 state；多 input operator 需对齐 barrier，或记录未对齐的 in-flight buffers。完成 checkpoint 后才可提交对应 source offsets。

需监控 checkpoint duration、alignment time、state bytes、失败率、last successful age 和 restore time。Checkpoint 太频繁增加 I/O 与 barrier 开销；太稀疏则失败后 replay 多、恢复慢。State backend snapshot 也要验证真实 restore，而非只看上传成功。

### 外部 sink 是语义边界

框架可以回滚自己管理的 state 和 downstream messages，却无法撤回已发邮件或普通数据库写。要让 source offset、operator state、output 一致，有三条主要路径：

1. Sink 与 stream framework 参与受限 atomic transaction/two-phase commit，checkpoint 完成才 commit staged output。
2. Sink 操作天然或人为幂等，replay 得到同一效果。
3. 输出先写入框架管理的 durable log，再由独立幂等 delivery service 执行外部 side effect。

### 幂等 sink 的具体做法

`SET key=value` 在值确定时可幂等，`INCREMENT` 不可。常用去重键为 `(source_topic, partition, offset)` 或业务 `event_id`，在 sink transaction 中原子记录已处理 marker 与业务变更：

```sql
BEGIN;
INSERT INTO processed_events(event_id) VALUES (?) ON CONFLICT DO NOTHING;
-- 只有 insert 成功才执行业务 mutation
UPDATE ...;
COMMIT;
```

或者 sink row 保存 `last_applied_version`，仅接受更大 version。前提是 replay 顺序确定、processing deterministic、没有另一 writer 绕过 version 规则。Dedupe table 有 retention 问题：若 marker 先过期而 source 仍可 replay，旧 event 会再次生效。

对于金额不要简单把 offset 当全局版本：offset 只在 topic partition 内可比。业务 aggregate 必须稳定映射到同 partition，或使用 aggregate sequence。Failover 时还需 fencing generation，阻止被认为死亡的旧 task 继续写 sink。

### Rebuilding state

Window、join table、dedupe set 都是 state。可以：

- 存远端 replicated database，简单耐久但每条 RPC 慢。
- 本地 state + checkpoint 到 durable storage，正常路径快。
- 本地 state changes 写 compacted changelog，新实例 replay 重建。
- 对短窗口完全从 retained input replay。

选择取决于 state 大小、更新速率、允许恢复时间和网络/磁盘性能。恢复计划必须验证 retention 足够、schema 可读、checkpoint 与代码兼容，并预估 bootstrap 对 broker/sink 的压力。

| Event contract | ID、key、schema version、event/ingest time、敏感字段、顺序边界 |
| Delivery | 丢失/重复/乱序语义、ack 点、retry/DLQ、idempotency key |
| Capacity | 峰值 ingress、partition skew、retention、lag 到数据丢失时间 |
| State | 窗口/join/dedupe 大小、TTL、checkpoint、restore RTO |
| Time | event-time 来源、watermark、allowed lateness、correction/retraction |
| Sink | transaction 或幂等协议、外部副作用、fencing、replay 模式 |
| Operations | backpressure、lag、checkpoint、late data、schema/poison event、对账 |

## 常见理解偏差

1. **用了 broker 就不会丢消息**：durability、replication、producer ack 与 retention 配置共同决定。
2. **Partitioned log 有全局顺序**：通常只有 partition 内 order，key 选择决定因果边界。
3. **提交 offset 等于业务处理成功**：除非两者原子，否则早提交会丢、晚提交会重。
4. **Buffer 足够大就不需要 backpressure**：长期输入大于输出最终必填满任何有限 buffer。
5. **CDC 就是定时查 updated_at**：轮询会漏同时间戳、删除和事务顺序，也难无缝接 snapshot。
6. **Event sourcing 等于保存数据库 row diff**：前者记录领域事实与意图，后者通常记录物理状态变化。
7. **Immutable 表示永远不能删除**：合规和泄密处置仍要求全链路删除设计。
8. **Watermark 证明窗口完整**：它是 progress estimate，迟到策略与 correction 仍不可省。
9. **Checkpoint 开启后端到端 exactly-once**：它只覆盖框架控制的 state/output，外部 sink 另算。
10. **幂等就是用 UUID**：必须在最终资源原子检查并把 dedupe marker 与副作用一起提交。
11. **Replay 只是把 offset 调回去**：容量、旧 schema、历史维表与不可逆副作用都可能改变结果。

## 本章小结

- Stream 是持续可用的 unbounded event sequence；消息系统必须明确慢消费者、durability、ack 和 redelivery。
- Partitioned log 用 offset、consumer group、retention 和 replay 结合数据库持久性与消息通知；顺序只在 partition 内成立。
- CDC 让搜索、缓存和数仓 follow 主库提交顺序；一致 snapshot、增量切点、tombstone 与 schema 演化决定能否可靠重建。
- Event sourcing 把 accepted domain event 作为事实，state 是其 materialized view；command 在成为 event 前必须完成验证。
- Event time、window、watermark 和 late-data policy 共同定义实时结果的时间语义。
- Stream-stream、stream-table 和 table-table join 都需要有界或持久 state，并明确历史版本与跨流顺序。
- Checkpoint 让内部 state 可一致恢复；端到端 exactly-once 仍要求 source position、state 与 sink 通过事务、幂等和 fencing 连接起来。
- Backpressure、lag、checkpoint、replay 与对账是流系统日常正确性的一部分，不只是运维细节。
