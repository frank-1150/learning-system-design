## Part III 的起点：从请求响应转向派生数据

前两部分主要讨论 record system：存储、复制、分区、事务与一致性。Part III 转向 **derived data（派生数据）**：搜索索引、推荐特征、指标汇总、缓存和数据仓库表都不是事实的唯一来源，而是由另一份数据计算出来的视图。

派生数据的关键优势是可以重建。若搜索索引损坏，只要权威事件或数据库还在，就能重新计算；但“理论上能重建”不代表流程可靠。你仍需回答：输入快照在哪里、算法和 schema 是哪个版本、输出如何原子切换、重跑会不会重复写、副作用是否幂等。

数据处理系统大致有三种形态：

- **Online service**：等待请求，尽快返回，性能看 latency 与 availability。
- **Batch processing**：读取有限但可能巨大的输入，运行一段时间产出结果，性能看 throughput 与完成时间。
- **Stream processing**：输入无界，事件持续到达，尽快更新结果；第十一章展开。

批处理像“自动化的大规模离线程序”。它没有用户在连接上等待，却承担数据规模、长链依赖和部分机器故障。第十章从单机 Unix 工具出发，解释 MapReduce/HDFS，再走向 dataflow engine、图迭代与高级声明式 API。

## Unix tools：几十年前已经出现的批处理组合模型

假设 Web server 每个请求写一行 access log，现在要找访问量最高的五个 URL：

```bash
cat access.log |
  awk '{print $7}' |
  sort |
  uniq -c |
  sort -r -n |
  head -n 5
```

每个工具只做一件事：`awk` 取字段，`sort` 排序，`uniq -c` 按连续相同 key 聚合，第二次 `sort` 排名。通过 stdin/stdout pipe，工具彼此不知道对方实现，却组成完整 workflow。

### Chain of commands 与自定义程序

也可以写一个内存 hash map 统计 URL，最后取 top 5。数据装得下内存时，自定义程序可能更快；Unix pipeline 的排序需要 `O(n log n)` 比较。但 `sort` 能自动 spill to disk，并行利用多个 CPU；内存 map 若遇到高基数输入可能 OOM。

选择不是“命令行一定优雅”。pipeline 的优势是少代码、可观察中间结果、成熟工具自动处理超内存数据；自定义程序更易表达复杂解析、近似算法和专门数据结构。工程判断要看基数、数据大小、错误处理与可维护性。

### Unix philosophy 的四个支点

原书从 Unix 传统提炼出对现代 data system 仍重要的思想：

1. 每个程序做好一件事。
2. 期望一个程序的输出成为另一个程序的输入。
3. 尽早试验，不因原型笨拙就犹豫重写。
4. 用工具自动化重复劳动。

它能组合，靠的不只是“小程序”，还有三个接口性质。

### Uniform interface：字节流是最小公分母

stdin/stdout 是顺序 byte stream。程序不需要针对每个上游写 adapter；只要约定记录边界和字段格式就能连接。文本行便于人工检查，但不是必要条件，binary stream 同样可 pipe。

统一接口会牺牲类型丰富度，却大幅降低 wiring 成本。这与网络世界的 HTTP、数据库世界的 relation 有相似作用：一个足够通用的接口让独立工具组合，而不是每对组件定制集成。

### Separation of logic and wiring

程序实现转换逻辑，shell 管输入文件、输出去向和 pipeline 拓扑。同一个 `awk` 不需知道输入来自文件、网络还是上个进程；也不需知道输出被打印、压缩还是继续聚合。这种逻辑与编排分离，后来也出现在 dataflow DAG 和 workflow scheduler 中。

### Transparency：文件让实验与排障可见

可以保留每一步输出，用 `head` 抽查，替换 pipeline 中一个命令后重跑。输入不可变时，命令是 deterministic function，就容易复现。Unix 工具也不完美：只有单机；程序可能对输入格式作隐含假设；复杂 quoting 和错误传播难维护。但其组合原则贯穿后续批处理架构。

## MapReduce 与 HDFS：把 Unix 模型扩展到集群

Hadoop 的 MapReduce 可理解为跨机器的 `map | sort | group | reduce`，输入输出通常位于 distributed filesystem，例如 HDFS。HDFS 把大文件拆成 block，复制到多台 data node；name node 维护文件与 block 元数据。计算 scheduler 尽量让 task 在拥有输入副本的机器上执行，即 **move computation to data**，降低跨机架网络流量。

这种设计面向 commodity hardware：单机随时可能失败，任务重跑是正常机制。HDFS 文件偏好大块顺序读写，不是低延迟随机更新数据库。

## MapReduce job 怎样执行

用户提供 `mapper` 与 `reducer`：

```text
input files
   │ split
   ▼
map tasks: record -> (key, value)
   │ partition by key + local sort
   ▼        shuffle over network
reduce tasks: key -> all values for key
   │
   ▼
output files
```

1. framework 按 HDFS block/file split 创建多个 map task。
2. mapper 解析记录，可发出零到多个 key-value pair。
3. map side 按目标 reducer 分区并按 key 排序，通常先写本地磁盘。
4. reducer 从所有 mapper 拉取自己的 partition，这个数据搬运称为 shuffle。
5. 同 key 的值连续到达 reducer，reduce function 聚合并写 HDFS。

mapper/reducer 通常是无状态函数。framework 负责调度、重试、排序、shuffle 和输出。map task 失败可在另一节点重跑；reduce task 的输入来自已完成 mapper，失败也可重新拉取并计算。

### Workflow：一个 job 往往不够

复杂分析由多个 job 串联：job A 输出目录成为 job B 输入。早期 Hadoop 的 workflow 通过目录路径隐式连接，Oozie、Azkaban、Airflow 等 scheduler 决定依赖、重试和时间窗口。MapReduce 本身不会自动理解整个 DAG，因此每个 stage 都把中间结果完整写入 HDFS。

物化中间文件带来恢复和审计优势，也产生序列化、磁盘和网络开销。后面的 dataflow engine 正是从这里改进。

## Reduce-side join：把相同 join key 的记录送到同一处

批处理经常把 activity event 与 user profile 关联。日志可能只含 `user_id`，而分析需要年龄、地区等属性。逐条向在线数据库查询会把批处理吞吐转成海量随机请求，拖垮生产数据库，也无法保证同一快照。

Reduce-side sort-merge join 把两个输入都送进 MapReduce：

```text
activity mapper: user_id -> ("event", event)
profile mapper:  user_id -> ("user", profile)
                       │ shuffle/group by user_id
                       ▼
reducer: [profile, event, event...] -> enriched events
```

mapper 给记录打 source tag，partitioner 确保相同 `user_id` 到同一 reducer；secondary sort 可让 profile 先到，reducer 不必把所有 event 缓存在内存。核心不是 SQL join 语法，而是 **bringing related data together in the same place**。

### Grouping 是 join 的近亲

`GROUP BY user_id`、倒排索引、每用户 sessionization 都让同 key 数据汇聚。若按 key 排序，reducer 一次处理一个 group，内存占用取决于单个最大 group，而不是全量数据。

### Skew 与 hot key

若一个 celebrity 用户有数亿事件，hash partition 并不会平均：一个 reducer 承担巨大 group，其他 reducer 已完成，整个 job 等最后一个 straggler。平均每 partition 大小掩盖了 key frequency distribution。

常见策略：

- 先抽样识别 hot key。
- 将 hot key 加随机 salt 分散到多个 reducer，下一 stage 再合并部分聚合。
- 对 join 使用 skewed join：把 hot-key 侧分片，把另一侧对应小记录复制到各分片。
- 单独处理极端 key，再与普通结果 union。

salting 改变数据流语义，并非无成本。不可结合的非交换运算可能无法两阶段聚合；复制小表也要计算内存与网络。

## Map-side join：输入预先满足条件时绕过 shuffle

Reduce-side join 通用但昂贵，因为两侧都经历 shuffle、sort 与网络。若输入结构已知，可以在 mapper 本地完成。

### Broadcast hash join

一侧足够小，加载到每个 mapper 内存 hash table；mapper 流式扫描大表并查小表。它避免大表 shuffle，但“小”必须按每 worker 可用内存判断，且每个 worker 都复制一份。

分布式 cache 或 framework broadcast variable 可分发小表。若维表更新，要确保所有 task 使用同一 version，不能一部分加载新快照、一部分旧快照。

### Partitioned hash join

若两侧用相同 key、相同 partition 数和相同 hash function 预分区，则对应 partition 可在同一个 mapper join。小的一侧 partition 建 hash table，另一侧流式 probe。Hive 称 bucketed map join。

这个优化对 metadata 要求严格：bucket count、hash algorithm、key normalization 任一不一致都会漏匹配。文件“看起来按同一字段分区”不等于 co-partitioned。

### Map-side merge join

如果两侧不仅按同一 key 分区，还在 partition 内按 key 排序，mapper 可像 merge sort 一样顺序扫描两侧，以有限内存 join。输入准备成本更高，但重复查询可复用这种布局。

Map-side join 把成本前移到数据布局。它适合稳定、可复用的 pipeline；临时分析中，reduce-side join 的通用性往往更重要。

## Batch workflow 的输出：派生视图而不是逐条副作用

批处理最终常构建搜索索引、推荐模型、KV lookup file 或数据仓库表。原书强调，直接从 mapper/reducer 对在线数据库逐条写入通常很危险：

- task 重试会重复写，副作用必须幂等。
- 大量并发 worker 可能压垮在线服务。
- 半个 job 成功后，读者会看到新旧结果混杂。
- job 失败后难以判断哪些记录已生效。

更稳妥的模式是让 task 写 immutable file，整个 job 成功后再批量加载或切换版本。

### 构建搜索索引

mapper/reducer 可按 index partition 生成 Lucene segment。每个 reducer 产出一个完整 partition 文件；成功后把整套索引复制到 serving node，最后切换 alias/manifest。旧索引在切换前继续服务，失败的新版本直接丢弃。

### 构建 KV 文件

SSTable、HFile 等按 key 排序的 immutable file 可由 batch job 直接生成，再由 serving database bulk-load。这比逐条 `PUT` 更高效，也使输出可回滚。Voldemort 的只读 store 是原书示例：生成新版本目录，分发完成后原子切换。

### 输出哲学：纯函数、不可变输入、显式发布

理想 batch task 像纯函数：给定相同输入与代码版本，产生相同输出；不修改输入，没有隐藏外部副作用。task 失败可任意重跑，framework 实现看起来接近 exactly-once output，底层实际是 **at-least-once execution + deterministic output/commit protocol**。

输出目录应带 run/version，task 先写临时文件；task attempt 成功后由 coordinator 选择一个 attempt；整个 workflow 验证完成后，原子更新 manifest/metadata pointer。消费者只读已发布版本。

## Hadoop 与 distributed database：目标不同导致架构不同

Hadoop 常被拿来与 massively parallel processing（MPP）database 比较。两者都把计算分到多机，但传统侧重点不同。

### Storage diversity

数据库要求输入先按 schema 导入自己的存储格式，随后能用索引与统计信息高效查询；HDFS 接受任意文件，先落地、后解释，即 schema-on-read。原始数据即使暂时不能解析也可保存，适合数据湖和探索式处理，但垃圾输入不会因为存进 HDFS 自动变好。

### Processing model diversity

MPP database 以 declarative SQL 为中心，optimizer 决定 join 顺序和执行计划；MapReduce 允许任意 mapper/reducer 代码，能处理文本、图像、模型训练等非关系转换，但手写过程逻辑更冗长，也更难优化。

后来两侧相互吸收：Hive/Spark SQL 给 Hadoop 带来声明式查询；数据库支持 UDF、外部表与半结构化数据。重点不是阵营，而是 workload 是否需要严格 schema、交互查询、任意代码与开放文件格式。

### Designing for frequent faults

MPP 查询常在几秒到几分钟内完成，节点失败可能 abort 整个 query 再重跑；MapReduce job 可运行数小时，遇到机器故障概率更高，所以以 task 为单位重试并物化 stage。MapReduce 也可能启动同一 task 的 speculative execution，采用先完成者结果来缓解慢机器。

但 speculative execution 对真正的数据 skew 帮助有限：同一个 hot partition 在任何机器上都慢；若 task 有外部副作用，重复 attempt 还会制造灾难。需要先区分 hardware straggler、resource contention、GC pause 与 skew。

## Dataflow engine：把固定 map/reduce 变成任意 DAG

Spark、Tez、Flink 等将计算表示为 operator DAG：map、filter、join、group、aggregate 可以任意组合，而不必每两步都套一个 MapReduce job。

```text
read ─> filter ─> parse ─┬─> join ─> aggregate ─> write
                         └─> quality metrics
```

operator 的输出可直接通过内存 buffer 或网络送给下游；只有 shuffle boundary、显式 checkpoint 或缓存需要物化。scheduler 能看见整个 DAG，于是可把相邻 operator fuse 在同一 task，减少中间文件和启动开销。

### Intermediate state 与容错

MapReduce 每 stage 写 HDFS，中间状态耐久、恢复简单但昂贵。Dataflow engine 常通过 lineage 记住 partition 如何由输入转换；丢失 partition 时，从最近 durable input/checkpoint 重算必要路径。

重算是否便宜取决于 lineage 长度、shuffle 输出是否仍可用和计算成本。Spark 可缓存 RDD/DataFrame，Flink batch/dataflow 也有不同 blocking/pipelined exchange。不是“内存计算不落盘”，而是 framework 按 DAG 与故障模型选择哪些中间状态值得 materialize。

### Pipelining 的代价

减少物化能显著降低 latency 和 I/O，却扩大 failure recomputation 范围；上游与下游同时运行时，backpressure、buffer 和资源调度更复杂。一个下游慢 operator 可占住整个 pipeline。批处理 engine 仍需要 spill、shuffle service、checkpoint 和 stage retry。

## 图与迭代处理：一次 DAG 不够表达反馈回路

PageRank、连通分量、推荐传播等算法反复沿 edge 更新 vertex，直到收敛。朴素 MapReduce 每次迭代都读写完整图并启动新 job，I/O 和调度成本高。

Pregel 使用 **bulk synchronous parallel（BSP）** 模型：每个 vertex 在一个 superstep 中接收上轮消息、更新状态、向其他 vertex 发消息；barrier 后进入下一 superstep。若所有 vertex 都 inactive 且无消息则结束。

```text
superstep N:   vertex compute + send messages
                       │ barrier
superstep N+1: consume messages + update
```

vertex 按 ID 分区，edge 跨 partition 时消息走网络。图结构若存在超级节点，也会产生 skew；可切分高 degree vertex、组合消息或使用更适合的 partitioning。

容错通常通过 checkpoint vertex state，失败后回到最近 superstep；也有 lineage/message logging 方案。图算法的迭代依赖让低级容错与 partition strategy 对性能影响很大。

## High-level API：声明“要什么”，让 optimizer 决定“怎么做”

直接写 mapper/reducer 要手动管理 tag、secondary sort 和 join strategy。Hive、Pig、Cascading 以及后来 Spark SQL/DataFrame 提供 relational/declarative API。查询 optimizer 可根据统计信息选择 broadcast、shuffle hash 或 sort-merge join，裁剪列与谓词，并融合 operator。

声明式 API 不只是少写代码：它给系统看见整体意图的机会。手写 UDF 是 optimizer 的黑盒，可能阻断 predicate pushdown、列裁剪和代码生成；能用内建 expression 时通常更可优化。

### Specialization 仍有价值

批处理还需要 machine learning、图算法、空间处理等专门领域 API。高级库把算法表达为 dataflow operator，同时复用底层 scheduler、shuffle 与 fault tolerance。SQL 不必包办所有计算，但统一执行底座能减少每个领域重复造分布式基础设施。

## 工程延伸：让批处理可重跑、可发布、可追责

以下是由原书输出哲学延伸的生产实践。

### 幂等输出不等于覆盖同一路径

稳定输入与代码版本应生成唯一 `run_id`，输出写入新路径：

```text
/features/date=2026-07-11/run=abc123/part-000...
```

每个 task 使用 attempt 临时路径，commit 后不可变。直接覆盖正式目录可能让读者在覆盖过程中看到混合版本；object storage 的 rename 也未必是原子操作。更可靠的是写完整版本，再原子更新小型 manifest/catalog pointer。

数据库 sink 使用自然唯一键或 `(job_version, entity_id)`，配合 upsert；但 upsert 只能解决重复，不能保证全量版本同时可见。需要 staging table + metadata swap，或让查询显式按 published version 读取。

### 数据快照与代码必须共同版本化

可复现 run 至少记录：输入 URI/partition 与 snapshot ID、schema version、代码 commit、参数、依赖版本、时区、输出 manifest、质量检查结果。只记“2026-07-11 的数据”可能仍不确定，因为迟到分区后来被补写。

### Straggler 与 skew 的诊断顺序

观察 task duration 分布、输入 bytes、record count、shuffle read/write、spill、GC、CPU 与 remote fetch wait：

- 输入显著更大：partition/key skew。
- 输入相近但 CPU 高：数据内容或算法热点。
- CPU 低、fetch wait 高：网络/shuffle service。
- 同 task 换机器后恢复：节点问题，speculation 有效。
- 所有 task 同时变慢：共享存储、配额或下游瓶颈。

盲目增加 worker 不会拆开单个 hot key，反而可能增加 shuffle 连接和小文件。

### 数据质量是发布条件，不是事后 dashboard

发布前校验 row count、主键唯一、null/范围、referential integrity、分布漂移、金额守恒以及与上一版本差异。失败版本保留诊断信息但不更新 pointer。对于大表可组合精确 constraint 与抽样统计；关键财务不变量应精确对账。

## 常见误区

1. **Batch 就是不实时、所以不重要**：账单、索引重建、模型训练和合规删除都依赖批处理正确性。
2. **MapReduce 保证业务 exactly once**：framework 可选择一次 task 输出，但外部 API/数据库副作用仍可能重复。
3. **hash partition 会自然均匀**：key 频率重尾时，单个 hot key 决定整个 job 完成时间。
4. **Map-side join 总更快**：它依赖小表大小或严格 co-partition/sort metadata；准备与维护布局也有成本。
5. **Dataflow engine 全在内存**：shuffle、spill、checkpoint 和故障恢复仍会使用磁盘/远端存储。
6. **Speculative execution 能解决所有 straggler**：数据 skew 在另一机器重跑依然慢。
7. **写完所有文件就能被读取**：缺少原子发布时，下游会看到不完整版本。
8. **Schema-on-read 不需要治理**：它把解析时点推迟，并没有消除 schema、质量或兼容性责任。
9. **高级 API 一定优化得最好**：opaque UDF、错误统计信息或极端 skew 仍需理解物理执行计划。

## 本章小结

- Unix pipe 通过统一字节流、逻辑与 wiring 分离、透明中间结果，奠定了可组合批处理思想。
- MapReduce/HDFS 把函数移到数据附近，用 shuffle/grouping 连接相关记录，并以 task 重试容忍机器故障。
- Reduce-side join 通用但 shuffle 昂贵；map-side join 用 broadcast 或预分区/排序换取更少网络。
- Batch 输出应被视为可重建派生视图：不可变、幂等、版本化，整套结果验证后原子发布。
- Hadoop 与 MPP database 的差异源于 storage、processing model 和故障时长假设，现代系统正在互相吸收。
- Dataflow engine 用任意 DAG、pipeline 与 lineage 减少中间物化；图迭代和高级 API 在同一执行基础上提供专门抽象。
- 生产性能常由 hot key、straggler、shuffle 与小文件决定；生产正确性则取决于输入版本、质量 gate 和发布协议。
