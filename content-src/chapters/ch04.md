## 这一章真正要解决的问题：变化发生时，旧世界不能立刻消失

第四章讨论的表面主题是 encoding（编码）与 schema，真正主题却是 **evolution（演化）**：当数据结构改变时，系统怎样允许多个版本同时工作。

产品变化通常会带来数据变化：新增字段、拆分记录、改变枚举、引入新的事件类型。代码却不可能在同一瞬间全部升级。服务端通常滚动发布，十个实例里可能六个运行新版本、四个仍是旧版本；移动端用户可能几个月不升级；数据库里则同时存在五年前与刚写入的数据。真实系统更像这样：

```text
旧数据 ─┐                    ┌─ 旧服务实例
新数据 ─┼─ 数据库 / 消息流 ─┼─ 新服务实例
未来数据 ┘                    └─ 尚未升级的客户端
```

因此必须同时区分两个方向：

| 兼容性 | 含义 | 谁面对未知信息 |
| --- | --- | --- |
| Backward compatibility（向后兼容） | 新代码能读旧代码写的数据 | 新 reader 面对旧数据，通常较容易 |
| Forward compatibility（向前兼容） | 旧代码能读新代码写的数据 | 旧 reader 面对未来新增内容，通常更难 |

“向前/向后”说的是数据版本的时间方向，不是请求方向。一个实用记忆法是：**reader 能否容忍 writer 与自己版本不同**。本章会先比较几种字节格式，再把兼容性放进数据库、服务调用和异步消息三种数据流中检验。

> [!note] 原书与工程延伸
> 本章对语言原生格式、JSON/XML、Thrift、Protocol Buffers、Avro，以及数据库、REST/RPC、消息队列的论证依照 DDIA Chapter 4（印刷页 111-142）。Schema Registry、expand/contract 发布流程、兼容性 CI 和 poison message 隔离属于基于本章原则的工程延伸，会明确标出。

## Encoding：内存对象不能直接穿过进程边界

程序在内存中使用 object、struct、hash table 和 tree。这些结构为 CPU 操作优化，里面可能有指针；指针只在当前进程地址空间有意义。写入磁盘或发送网络时，数据必须成为自包含的 byte sequence。

- **Encoding / serialization / marshalling**：内存表示变成字节。
- **Decoding / parsing / deserialization / unmarshalling**：字节恢复成内存表示。

编码不等于加密，也不等于压缩；“serialization”在第七章还表示事务的可串行化，为避免混淆，原书在本章使用 encoding。格式选择也不只是“少几个字节”：它同时决定跨语言能力、安全边界、类型表达、兼容规则和调试体验。

### 语言原生序列化：方便恰恰掩盖了长期契约

Java `Serializable`、Python `pickle`、Ruby `Marshal` 能几乎不写额外代码就保存对象，但原书指出四类深层问题：

1. 字节表示与语言乃至具体类绑定，跨语言困难，长期数据也被当前实现锁定。
2. decoder 为恢复任意对象可能实例化任意 class。对不可信输入反序列化，可能升级为远程代码执行。
3. versioning 往往是事后补丁，而持久数据偏偏比代码活得更久。
4. 大小和编解码性能通常不是设计重点。

所以它们更适合短暂、封闭、可信的场景，例如同一进程版本生成并立即消费的缓存；不适合公共 API、长期数据库记录或跨团队事件。即便输入来自“内网”，也不应默认可信：队列权限误配、备份污染或被攻陷的上游都可能把恶意字节送到 decoder。

## JSON、XML、CSV：互操作性经常比完美更重要

文本格式广泛、跨语言且可人工查看。JSON 因浏览器原生支持和相对简单而流行；XML 的 schema 与命名空间能力更强，但也更复杂；CSV 适合扁平表格，却几乎把所有类型与结构解释交给应用。

它们的麻烦不只在“冗长”：

### 数字不是一个无歧义的类型

XML/CSV 若无外部 schema，无法区分数字 `42` 与字符串 `"42"`。JSON 区分 number 和 string，却不区分整数与浮点，也不规定精度。JavaScript Number 使用 IEEE-754 double，超过 `2^53 - 1` 的整数不能保证精确：

```json
{
  "orderId": 9223372036854775807,
  "orderIdText": "9223372036854775807"
}
```

原书以 Twitter 同时返回数值 ID 和字符串 ID 为例。工程上，64-bit ID、金额和高精度小数跨 JSON 边界时，应显式约定十进制字符串或受控 decimal 类型，不能让各语言“自行理解”。金额尤其不能用 binary floating point 随意运算。

### 文本对 bytes 的支持是绕行

JSON/XML 擅长 Unicode text，却没有自然的原始 byte string。Base64 可以承载图片摘要、密钥材料等二进制值，但体积约增加三分之一，还必须通过 schema 或字段约定说明“这是 Base64，而不是普通文本”。

### 可选 schema 不等于真实契约

XML Schema 和 JSON Schema 能提供丰富约束，但团队未必采用；不用 schema 时，数值精度、日期、null、缺失字段和 Base64 的解释会散落在业务代码中。CSV 更弱：列意义、转义、换行、表头和新增列都可能因 parser 不同而漂移。

尽管如此，原书没有得出“文本格式不好”的结论。组织之间达成一致往往比节省几十个字节更难；可读、普及、工具丰富本身就是重大优势。对外接口和人工数据交换中，JSON/CSV 常常是合理选择。

### Binary JSON 为什么提升有限

MessagePack、BSON 等把 JSON 数据模型编码成二进制，可以增加整数/bytes 类型并降低解析成本。但没有 schema 时，每条记录仍要带 `userName`、`favoriteNumber` 等字段名。原书示例的紧凑 JSON 为 81 bytes，MessagePack 仍约 66 bytes；收益是否值得牺牲可读性，需要结合数据规模与性能实测，而不是看到“binary”就假设快很多。

## Thrift 与 Protocol Buffers：用 field tag 把名字移出数据

Thrift 和 Protocol Buffers（Protobuf）都先写 IDL schema，再生成各语言类型与 codec。schema 给字段分配数字 tag：

```proto
message Person {
  string user_name = 1;
  optional int64 favorite_number = 2;
  repeated string interests = 3;
}
```

字节中保存 tag、wire type 与 value，不再重复完整字段名。Thrift 有 BinaryProtocol 和更紧凑的 CompactProtocol；Protobuf 使用相近的 tagged-field 思路。变长整数让小整数占更少空间，但负数若没有 zigzag 等专门编码可能很浪费。这里的重点不是背每一位，而是理解 decoder 依赖的是 **tag 与 wire type**，字段的人类名字主要存在于 schema 和生成代码中。

### Field tag 是持久身份，不能随意复用

字段名可以改，因为字节中通常没有名字；tag 一旦发布就成为协议身份：

- 新增字段时分配新 tag，并让 reader 对缺失字段使用 optional/default 语义。
- 删除字段后保留/封存旧 tag，绝不把同一数字给另一个含义。
- 旧 reader 遇到未知 tag 时跳过它，于是能读取新 writer 的数据。
- 新 reader 遇到旧数据缺少新字段时，使用默认值或“未设置”状态。

因此，把新字段改成 required 会破坏旧数据；删除 required 字段也会让旧 reader 失败。原书使用的版本里 Protobuf 有 required/optional 区分，现代 proto3 的默认实践更偏向避免 required，但原则不变：**schema 演化不能突然创造历史数据无法满足的不变量**。

改变 datatype 只有在 wire representation 兼容时才可能安全。`int32` 扩成 `int64`，旧 reader 读取超范围新值时会截断或溢出；反方向则可能读取成功，但新 writer 仍不能贸然写大值。Thrift 的 list 元素类型也有声明；Protobuf `repeated` 的演化在某些情况下允许旧单值与新列表互读，但旧代码只会看到列表最后一个元素，语义并不等价。

> [!warning] “能 decode”不等于“业务兼容”
> 把 `status=2` 新解释成 `REFUNDED`，旧代码虽然能读出整数 2，却可能走默认分支；把秒改成毫秒，wire type 完全相同，结果仍错一千倍。格式库只能检查结构，无法替你保证单位、枚举和不变量。

## Avro：让 writer schema 与 reader schema 显式相遇

Avro 同样使用 schema，但编码结果不包含 field tag，也不包含字段名。值按 schema 中的字段顺序紧凑排列，因此 reader 不能只拿一串 bytes 猜结构；它必须知道写入时使用的 **writer's schema**，并把它与自己期待的 **reader's schema** 做 resolution。

```text
bytes + writer schema + reader schema
                │
                ▼
       按字段名匹配与类型提升
       忽略 reader 不需要的字段
       为 writer 缺少的字段补默认值
```

writer 与 reader schema 不必完全相同，只需 compatible。字段顺序可不同，因为 resolution 按名字匹配；reader 新增字段需要 default，才能读取旧 writer 没写该字段的数据；reader 删除字段时可忽略 writer 多出的值。字段改名可以借 alias 建立匹配。

Avro 用 union 表达 `null` 或多种可能类型。新增分支通常向后兼容，但是否向前兼容取决于旧 reader 能否理解新分支。默认值也不是数据库里的自动回填：它是 schema resolution 时 reader 在字段缺失情况下构造的值。

### decoder 到哪里找 writer schema

原书列出三种典型答案：

1. **大文件**：文件头包含一次 writer schema，后面大量记录共享它，适合 object container file。
2. **数据库记录**：每条记录带一个 schema version，数据库维护版本到 schema 的映射。
3. **网络连接**：连接建立时协商 schema，随后连接生命周期内复用。

这使 Avro 很适合动态生成 schema。例如关系数据库表结构变化后，可以从表 schema 自动生成 Avro schema，导出记录；列增加时生成新的 writer schema，旧 reader 仍可根据 resolution 读取。它避免为每个字段维护永久 tag，也因此适合 schema 经常由外部系统生成的场景。

### Code generation 不是所有语言都同样需要

Thrift/Protobuf 常通过 codegen 生成静态类型类，IDE 补全和编译期检查很好；Avro 也支持 codegen，但可直接把 schema 当运行时对象使用。对 Java 等静态语言，生成类型很自然；对动态语言，生成 class 可能收益较小。原书还提醒：数据库行天然是动态 schema，ORM/codegen 的再生成周期可能降低探索式查询的便利。

## 为什么 schema 值得付出成本

binary schema 常被误解为“难看、没人能读的专有格式”。原书总结了它相对语言原生序列化和无 schema JSON binary 的优势：

- 比在每条记录中重复字段名更紧凑。
- schema 是可维护的文档，且必须与实际解码一致。
- schema database 可以在部署前检查 forward/backward compatibility。
- 静态语言可以从 schema 生成类型，在编译期发现错误。

它不是万能答案。面向人类调试、开放生态或请求量较小时，JSON 的可见性可能更值钱；高吞吐内部日志和长期事件契约，则更可能从 Protobuf/Avro 获益。真正重要的是把“数据会演化”作为格式的一等设计目标。

## 数据流一：数据库是给未来代码发送消息

写入数据库的进程编码数据，之后读取的进程解码数据。两者可能隔几分钟，也可能隔十年。因此数据库中的值可以视为一封发给未来自己的消息。

滚动发布期间，新代码可能写入新字段，随后旧实例读取、修改同一记录并写回。即使旧实例会忽略未知字段，如果它采用“decode 成旧 object → 修改 → 整体重新 encode”，未知字段也可能被悄悄丢掉：

```text
新版本写：{name, email, preferred_locale}
旧版本读：{name, email}                 # 不认识 locale
旧版本写：{name, updated_email}         # locale 消失
```

因此 forward-compatible decoder 只是第一步，更新路径还需保留未知字段、使用局部字段更新，或保证旧 writer 不覆盖新表示。数据库 schema 本身也可能限制兼容窗口。

### 历史数据不会因 schema 改变自动重写

许多数据库允许新增 nullable/default 字段而不重写所有旧记录；读取时像是存在默认值，磁盘上的旧字节仍保持原样。LinkedIn 的 Espresso 用 Avro 存储就是原书例子之一。迁移应区分：

- **logical migration**：新 reader 能解释旧记录。
- **physical rewrite/backfill**：后台逐步把旧记录转换成新表示。

后者可能制造大量写放大、复制延迟和缓存失效，不能因为 DDL 很快就假设迁移免费。

备份与数据仓库导出则常在同一批次把不同年代的记录解码后，统一用最新 schema 编码。Avro object container、Parquet 一类分析格式适合这种 immutable snapshot，后续第十章会继续讨论。

## 数据流二：服务调用把兼容性变成组织契约

面向服务架构中，一个服务拥有自己的数据，并通过网络 API 提供能力。服务器是 client 的下游，但数据编码上双方轮流成为 writer/reader：请求由 client 写、server 读；响应反过来。

### REST 是设计哲学，SOAP 是带标准体系的协议路线

REST 通常利用 HTTP 的资源、方法、状态码、缓存和内容协商，常配 JSON；它不是单一线协议。OpenAPI 可描述 endpoint 与 schema，curl、浏览器、代理和抓包工具让调试较直接。

SOAP 通常使用 XML 与 WSDL，围绕类型、操作以及 WS-* 扩展形成更完整但更复杂的工具生态。WSDL codegen 能让静态语言调用看似本地方法，却也可能让协议细节难以观察。原书并非简单宣布 REST 胜出，而是在对比开放、易调试的路线和更强标准/codegen 的路线。

### RPC 的根本错觉：远程调用不是本地函数

RPC 试图让网络请求看起来像函数调用。这个抽象会泄漏：

- 本地调用成功或抛错通常明确；网络超时意味着“可能没执行，也可能已执行但响应丢了”。
- 本地调用延迟较稳定；网络调用有排队、拥塞、重试和长尾。
- 重试可能让操作执行两次，除非请求天然 idempotent 或带去重键。
- 参数跨语言编码，数字、集合和对象模型并非完全一致。
- client/server 可独立升级，版本组合远多于单进程库。

现代 gRPC、Finagle 等 RPC 框架改善了 service discovery、deadline、codec 和可观测性，却没有取消网络的不确定性。API 兼容策略取决于组织关系：公司内部可以协调升级；公开 API 无法控制 client；跨组织 API 还要长期维护文档与版本。

原书建议 server 先升级、client 后升级的兼容路径：新 server 兼容旧 request（backward），旧 client 能理解新 response（forward）。现实中发布顺序可能反转，所以成熟契约通常要求双向兼容，并在遥测确认旧版本消失后再删除字段。

## 数据流三：消息传递把时间进一步解耦

异步消息系统处于数据库和 RPC 之间：消息不像 RPC 那样要求接收方在线，也不像数据库那样主要由写方自己稍后读取。producer 把消息交给 broker，broker 缓冲并投递给一个或多个 consumer。

它带来几种优势：接收方过载或不可用时可缓冲；进程崩溃后可重投；producer 不需知道 consumer 地址；一条消息可 fan-out；发送者与接收者在时间上解耦。代价是响应通常要另建通道，而且重复、乱序、积压与重放成为正常情况。

兼容性尤其重要：队列里可能积压旧消息，日志型 broker 还可能重放数月前的数据。consumer 独立部署时，新旧 schema 的组合比同步 RPC 更复杂。部署新 producer 前，应确认所有存量 consumer 能跳过未知字段；部署新 consumer 时，应能处理历史消息。若系统允许任意时间重放，那么 compatibility window 实际上等于 retention period，而不是“滚动发布的半小时”。

### Distributed actor：位置透明仍不能抹去版本问题

Actor model 把逻辑封装在单线程 actor 中，通过异步消息通信；分布式 actor 框架可能让本地与远程 actor 使用相似 API。原书提醒，这种 location transparency 仍有类似 RPC 的风险：远程消息有延迟、丢失和版本差异。Akka、Orleans、Erlang OTP 等框架的编码策略不同，但消息 schema 的 forward/backward compatibility 仍需显式设计。

## 工程延伸：把 schema 演化做成可执行发布流程

下面不是原书逐段内容，而是把本章兼容原则落到现代事件平台和微服务发布中的做法。

### Schema Registry 与 CI gate

集中 registry 保存 `subject + version → schema`，producer 在 payload 或 header 中携带 schema ID。CI 在合并前把候选 schema 与历史版本比较：

- backward：新 consumer 能读旧消息。
- forward：旧 consumer 能读新消息。
- full：两个方向都满足。
- transitive：不只与上一版比，还与所有保留版本比。

只比较最近一版可能漏错：v3 与 v2 兼容，但删除了 v1 reader 仍依赖的语义。结构检查还应配合 contract test，覆盖枚举未知值、缺失/显式 null、大整数、默认值和单位。

### Expand / migrate / contract

危险迁移常把“改 schema、改 writer、改 reader、删旧字段”放在一次发布里。更稳妥的步骤是：

1. **Expand**：先让所有 reader 同时接受旧表示和新表示。
2. **Introduce**：writer 开始写新字段；必要时 dual-write，并监控两种表示一致性。
3. **Migrate**：回填历史记录或等待消息超过 retention，记录迁移水位。
4. **Switch**：reader 以新字段为准，旧字段只用于回退。
5. **Contract**：确认旧 client/consumer 为零后停止旧写，最后封存字段/tag。

每一步都需要独立回滚。dual-write 若分两次提交可能不一致，数据库内可用事务/outbox；跨系统则需要对账和可重放来源，而不是假设两个写总会同时成功。

### 处理 poison message 与未知枚举

单条无法 decode 的消息不应让整个 partition 永久卡住。消费端要记录 schema ID、offset、错误类型和原始 payload 的受控引用，经过有限重试后送入隔离队列，并告警。隔离不能成为静默丢数据：必须有修复、重放和审计路径。

枚举应保留 `UNKNOWN` 分支并携带原始值。旧 consumer 看到未来枚举时可降级，而不是崩溃或把它误映射成某个已有业务状态。对支付、权限等关键状态，`UNKNOWN` 通常应 fail closed，并进入人工或延迟处理。

## 边界条件与常见误区

1. **加 optional 字段永远安全**：如果默认值改变业务含义，或旧 writer 整体覆写记录导致未知字段丢失，仍不安全。
2. **换成 Protobuf 就自动兼容**：复用 tag、改变单位、收窄数值范围、旧代码不处理未知枚举都可破坏系统。
3. **默认值等于数据已回填**：reader 看到的逻辑默认值不代表存储中存在该字段，聚合、CDC 和其他 reader 可能看到不同结果。
4. **HTTP 200 代表 RPC 成功且只执行一次**：超时和重试要求 idempotency key、状态查询或去重记录。
5. **消息消费成功就能删除旧 schema**：只要保留历史日志、备份或灾备重放，旧 writer schema 仍是解码依赖。
6. **二进制一定显著节省成本**：字段名是否内嵌、压缩、批大小、CPU 和可观测性共同决定收益，应基准测试完整链路。
7. **schema 管的是结构，业务语义可以口头约定**：单位、时区、枚举、字段来源与隐私等级也是契约，应进入文档和测试。

## 本章小结

- 编码格式定义的不只是字节布局，也是跨语言、安全、类型和演化契约。
- Forward compatibility 要求旧 reader 对未来新增内容有容忍策略，通常比 backward compatibility 更难。
- Thrift/Protobuf 依靠稳定 field tag；Avro 通过 writer schema 与 reader schema resolution 实现紧凑编码和演化。
- 数据库、服务调用与消息系统具有不同的版本共存周期；越异步、越可重放，兼容窗口通常越长。
- “能 decode”只是结构兼容，单位、枚举、默认值、不变量和更新方式决定业务是否真的兼容。
- 工程上应以 registry、兼容性 CI、expand/migrate/contract、遥测和重放演练，把兼容性从约定变成发布能力。
