## 数据模型不是存储格式，而是思考方式

这一章的核心句子可以翻译为：**数据模型也许是软件开发中影响最深远的部分，因为它不仅决定软件怎么写，还决定我们怎么思考问题。**

一个应用通常经历多层模型转换：

```text
现实世界
  ↓ 业务对象与规则
应用对象 / API
  ↓ ORM、序列化、查询
关系表 / 文档 / 图
  ↓ 存储引擎编码
磁盘页、日志、索引
  ↓ 硬件抽象
字节、内存与 I/O
```

每层都隐藏下层复杂度，并提供一套新的词汇。选择关系、文档或图模型，实际是在选择哪些关系可以自然表达、哪些约束由系统替你保证、哪些查询需要额外付费。

## Relational Model 与 Document Model

### 关系模型为什么能统治几十年

关系模型把数据组织为 relation（关系，SQL 中通常表现为 table）和 tuple（元组，通常表现为 row）。它最重要的价值不是“数据看起来像表格”，而是把业务数据的逻辑结构与底层存储实现分离。

早期数据库常要求应用沿着内部指针或预定义访问路径导航；关系模型让应用声明“想要哪些结果”，优化器决定 join 顺序、索引和访问路径。业务查询变化时，不必重写磁盘上的指针网络。

这也是关系模型的耐久性来源：它能自然表示多种业务数据，并通过规范化、主外键、唯一约束和事务维护共享事实。

### NoSQL 的出现不是一个单一技术原因

原书将 NoSQL 兴起归纳为多股力量：比关系库更高的扩展需求、对免费开源方案的偏好、关系模型不自然的特殊查询，以及对动态 schema 的需求。

“NoSQL”后来常被解释成 **Not Only SQL**，这比“不要 SQL”更有用。真正的问题不是阵营，而是：你的数据是否围绕一个聚合读取？关系是否频繁变化？是否需要跨对象约束？查询模式是否可预先知道？

### Object-relational mismatch：应用对象和表之间的摩擦

面向对象代码通常把一份简历表示成嵌套对象：一个人包含多个职位、教育经历和联系方式。关系模型有几种表达方式：

1. 把每类子项拆到独立表，用外键关联用户。
2. 用数据库支持的结构化类型或 JSON/XML 列。
3. 把整个对象编码成 JSON/BLOB，但数据库难以查询内部字段。

文档模型可以把这类 **one-to-many tree（一对多树状聚合）** 存成一份自包含文档：

```json
{
  "user_id": 251,
  "name": "Ada",
  "positions": [
    {"company": "Example", "title": "Engineer", "start": "2022-01"}
  ],
  "education": [
    {"school": "Example University", "degree": "MSc"}
  ]
}
```

读取整份简历时，文档模型具有良好的 **locality（局部性）**：一次读取拿到完整聚合，不需要多表 join，也更接近 API 返回结构。

但不要把“对象嵌套”误解成“所有业务都适合文档”。一旦 company、school 或 region 需要成为共享实体，问题就从 one-to-many 变成 many-to-one 或 many-to-many。

### 为什么 ID 比重复字符串更重要

假设每份用户文档都保存：

```json
{"region": "Greater Seattle Area"}
```

如果产品要统一改名或支持多语言，就必须更新大量文档。若保存稳定 ID：

```json
{"region_id": "us-wa-seattle"}
```

显示名称只在一处维护。ID 对人没有语义，因此不因文案变化而改变；这就是 normalization（规范化）消除写入重复的价值。

代价是读取时需要解析引用。在关系库里是 join；文档库若缺乏 join，应用需要额外请求、批量查询或把显示字段反规范化回文档。工程现实中，这通常不是“一次选型结束”，而是明确哪个系统保存事实，哪些字段是允许短暂陈旧的派生副本。

## 历史为什么会重复：层次模型、网络模型与关系模型

文档数据库看起来很现代，但“把数据存成嵌套树”与早期 hierarchical model 有相似之处。树对 one-to-many 很自然；many-to-many 则会遇到两难：复制数据造成一致性问题，或保存引用并在应用层手工解析。

早期 network model 允许记录有多个父节点，但应用必须沿 access path 导航。关系模型的突破是让查询优化器选择路径，而不是把路径固化在业务代码中。

这个历史提醒我们：**模型的表达能力之外，还要看查询如何获得结果。** 一个格式能保存引用，不等于系统能高效、灵活地解析这些引用。

## 关系与文档的真实权衡

### 文档模型的优势

- 聚合边界清晰时，读写局部性好。
- schema 随对象类型变化更自然。
- API 对象与持久化形态接近，转换层较薄。
- 不需要跨文档 join 的工作负载通常容易水平分区。

### 文档模型的代价

- 文档内部嵌套项不能独立引用时，重复和大文档更新会增加。
- many-to-many 关系通常需要应用 join 或额外派生索引。
- “schema-less” 不代表没有 schema；schema 只是从写入时检查转移到读取代码与数据治理。
- 局部更新、并发修改和跨文档约束可能更难。

### 关系模型的优势

- join 让新的关联查询不必预先把所有路径编码到数据中。
- 约束和事务能在共享事实上维护不变量。
- 规范化适合频繁变化、被多处引用的实体。
- 声明式 SQL 为优化器和工具生态留下空间。

### 关系模型的代价

- 树状聚合需要 join、ORM 或多次查询组装。
- schema 迁移需要兼容发布流程。
- 跨分片 join 和事务会显著增加分布式复杂度。
- 若 ORM 隐藏了查询形态，容易出现 N+1 和不可预测访问。

| 数据形态 | 通常更自然的起点 | 仍需确认 |
| --- | --- | --- |
| 用户配置、商品详情、内容正文 | 文档/聚合 | 是否会出现共享引用和跨聚合约束 |
| 订单、库存、账户、权限 | 关系 | 事务边界是否跨分片 |
| 社交关系、反欺诈网络、权限继承 | 图 | 遍历深度、更新率和结果规模 |
| 搜索结果 | 派生文档索引 | 新鲜度、删除和重建策略 |

## Query Languages：声明“要什么”，还是指挥“怎么做”

### Imperative 与 Declarative

命令式代码告诉机器执行步骤：遍历、判断、追加、排序。声明式查询描述结果需要满足的条件，让查询引擎决定执行策略。

```sql
SELECT animal
FROM animals
WHERE family = 'Sharks';
```

这条 SQL 没有指定全表扫描还是索引，也没有指定并行度。正因为接口没有暴露执行顺序，数据库才能在不改变查询语义的情况下加入新索引、重排 join、并行扫描或利用列存。

原书用 Web 上的 CSS/XSL 对比 JavaScript DOM 操作来说明：声明式规则通常更短，也给底层实现留下优化空间。工程上同样适用于基础设施 API：声明“期望副本数为 3”比脚本化“启动 A、等待、启动 B”更容易持续 reconcile。

### MapReduce 位于两者之间

MapReduce 查询接口允许用户提供 map/reduce 函数，表达能力更接近命令式，但框架仍控制分区、调度和容错。它展示了一种常见权衡：自定义函数增加表达力，却减少优化器理解和重写计算的能力。

现代数据系统也经常提供这种分层：常见查询走声明式 DSL；无法表达的部分使用 UDF。UDF 应被视为逃生舱，而不是默认接口，因为它会影响优化、隔离和可观测性。

## Graph-like Data Models：当关系本身成为主角

如果应用主要是 one-to-many，树形文档足够自然；当 many-to-many 和多跳关系占主导，图模型更合适。

### Property Graph

属性图由 vertex（顶点）和 edge（边）组成，两者都可以带属性：

```text
(Person: Alice) -[WORKS_AT {since: 2022}]-> (Company: Example)
(Person: Alice) -[LIVES_IN]-> (City: Seattle)
(City: Seattle) -[IN]-> (Country: US)
```

关键点是边不仅是外键，它有方向、类型和自己的业务属性。顶点可以连接任意其他顶点，因此模型不需要提前规定所有关系层级。

Cypher 这类声明式图查询可以表达模式匹配：找出住在某地区、且与某人存在若干跳关系的候选人。关系 SQL 也能递归查询图，但表达复杂度和优化能力依实现而异。

### Triple-store 与 SPARQL

Triple-store 把事实统一表示成 `(subject, predicate, object)`：

```text
(Alice, worksAt, Example)
(Alice, livesIn, Seattle)
(Seattle, within, Washington)
```

当 object 是另一个实体时，它就是一条边；当 object 是字符串或数字时，它是属性。RDF/SPARQL 生态强调跨数据集统一语义，但工程落地仍需处理标识、schema 治理和查询成本，不能因为格式统一就假设语义自动统一。

### Datalog：规则递归与数据分离

Datalog 使用规则组合查询，例如“如果 A 在 B 内，B 在 C 内，则 A 在 C 内”。它适合表达递归关系和规则推导。代价是对多数应用开发者不如 SQL/Cypher 直观，执行计划也需要专门优化。

## 工程现实：一个系统经常需要多个读模型

多模型并不意味着多个系统都能独立接受业务写入。一个更稳健的边界是：

```text
业务命令
   ↓
关系事实库（订单、SKU、商家、约束）
   ↓ CDC / Outbox
   ├──> 文档搜索索引（商品详情与全文检索）
   ├──> 图投影（搭配、兼容、推荐关系）
   └──> 列式仓库（分析）
```

这里关系库是 system of record；其他模型是可重建的 derived views。它们可以为查询反规范化，但必须记录来源版本、暴露新鲜度并支持删除传播。

最危险的是无协议 dual write：应用先写数据库，再写搜索；任何一步失败都可能产生永久分叉。更可靠的做法是让事实写入与 outbox 事件处于同一事务，再异步构建投影，并用对账修复漏更新。

## 选模型时应先回答的七个问题

1. 最频繁的读写路径是什么，而不是对象叫什么？
2. 一对多聚合是否总作为整体读取和修改？
3. 是否存在频繁变化、被大量对象共享引用的实体？
4. 哪些不变量必须由同一事务维护？
5. 新查询出现时，能否新增索引/投影，还是必须重写所有数据？
6. 派生数据允许多旧，如何发现并修复漂移？
7. 删除、权限和 schema 变更如何传播到所有模型？

## 常见理解偏差

- **Schema-less = 没有 schema**：读取者仍假设字段类型和含义，只是错误更晚暴露。
- **文档库不需要 join**：共享引用仍然存在，join 只是被移动到写入时或应用层。
- **规范化一定慢**：是否慢取决于访问路径、索引、工作集和缓存，不是范式名称。
- **图数据库适合所有有关联的数据**：一层主键/外键查询通常不需要图引擎；图的价值在动态多跳遍历。
- **Polyglot persistence 天然先进**：每多一个模型，就多一条同步、回放、删除、监控和运维链路。

## 本章小结

- 数据模型决定自然表达的关系、查询方式和演化成本。
- 文档模型擅长局部性强的树状聚合；关系模型擅长共享事实、约束和灵活 join；图模型擅长关系本身及多跳遍历。
- 声明式查询的价值是把执行策略交给优化器，为未来优化留下空间。
- 生产系统可以有多个模型，但应明确单一事实源，并把其他模型建设成可重建、可观测的派生视图。
