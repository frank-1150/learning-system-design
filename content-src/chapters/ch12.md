## 终章真正关心的是：数据系统最终为谁、为什么而存在

前十一章大多在描述数据系统“现在如何工作”：存储引擎、编码、复制、分区、事务、共识、批处理和流处理。终章转向规范性问题：这些机制应该如何组合，才能构造更可靠、正确、可演化，并最终对人有益的应用。

这不是“未来数据库产品预测”。Kleppmann 的核心判断是：没有一个工具适合所有访问模式，复杂应用必然组合多个专用系统。真正的设计对象因此不再是某个数据库，而是**贯穿组织的数据流**：输入从哪里进入，谁决定顺序，哪些状态是权威事实，哪些是可重建的派生视图，发生错误如何检测与修复，用户如何控制与自己相关的数据。

```text
用户意图
   │
   ▼
权威写入 / Event Log
   ├──> OLTP 当前状态
   ├──> Search Index
   ├──> Cache / Serving View
   ├──> Warehouse / Metrics
   └──> ML Features / Notifications
            │
            └── lineage、watermark、对账、删除传播
```

图里最重要的不是组件名称，而是箭头方向。只要多个系统都能被应用随意直写，系统就失去了唯一的变化顺序，也很难回答“以谁为准”。

## Data integration：一个应用天然需要多种数据表示

同样是“存储并再次查找数据”，不同负载适合 B-Tree、LSM-Tree、列式存储或内存索引；复制可以单主、多主或无主；查询可以事务点查、全文搜索、图遍历或大规模聚合。所谓通用数据库也有其优化方向，不可能在所有工作负载上都最好。

实际应用因此会组合：

- OLTP 数据库保存订单、用户等权威业务状态；
- 搜索引擎支持关键词、相关性和倒排检索；
- 缓存与反规范化表优化热点读取；
- 仓库、湖或批流引擎提供分析与报表；
- 推荐、分类、风控系统生成模型与派生特征；
- 消息系统触发通知和跨服务工作流。

“应该用哪个数据库”只是局部问题。系统级问题是：同一个业务事实出现在六个系统里后，它们如何保持可解释的关系？

### Derived data：区分事实与可重建视图

原书把系统分成两类角色：

- **System of record** 保存权威输入和事实。若出现矛盾，它是裁决来源。
- **Derived data system** 通过确定性或可解释的转换，从其他数据得到缓存、索引、物化视图或模型。

派生不等于“不重要”。搜索索引坏了会影响收入，风控特征错了会误伤用户；区别在于它理论上可以从输入重新计算。这个性质带来强大的恢复路径：修复代码后 replay，或从 snapshot/backfill 重建，而不必人工猜测最终值。

设计时应为每份状态回答：

1. 它的权威输入是什么？
2. 谁唯一有权更新它？
3. 转换逻辑和 schema 是哪个版本？
4. 可以从什么 checkpoint 重放？
5. 删除、订正和权限变化如何传播？

若这些问题没有答案，“数据平台”只是很多互相矛盾的副本。

### Reasoning about dataflows：单写入口比双写更重要

应用直接先写数据库、再写搜索索引，看起来简单，却有两个根本故障：一边成功另一边失败；两个客户端的并发写在两套系统里以相反顺序生效。重试只能处理部分丢失，不能恢复统一顺序。

更稳健的模式是先让一个系统决定变化顺序，再从该顺序派生其他表示：

```text
Command ──事务──> OLTP + Outbox
                       │
                    CDC Log（唯一顺序）
                       │
            ┌──────────┼──────────┐
            ▼          ▼          ▼
          Search     Cache      Warehouse
```

CDC 与 event sourcing 的具体选择不如“只有一个权威输入顺序”重要。消费者采用 deterministic transformation，并以 event ID、source offset 或业务幂等键去重，就能在失败后重复处理而收敛。

> [!note] 工程延伸：CDC 不是魔法事务
> CDC connector 需要正确处理 snapshot 与增量交界、事务边界、schema change、日志保留、分区重排和 poison event。业务表写与 outbox 必须在同一数据库事务；否则“写订单成功但没有事件”的 dual-write 缺口仍然存在。

### Derived data 与 distributed transaction 的取舍

两种方法都试图让多个系统保持关系，但机制不同：

| 维度 | Distributed transaction | Log-based derived data |
| --- | --- | --- |
| 顺序 | 锁与协调 | 日志位置/事件顺序 |
| 原子性 | 2PC/atomic commit | 可重试转换与幂等 |
| 可见时间 | 通常提交后同步可见 | 通常异步，有 lag |
| 故障形态 | prepared 状态可能阻塞 | backlog 增长、视图陈旧 |
| 重建 | 依赖各参与者状态 | 可从输入 replay |

原书并不是说“事务已死”。受控环境中，分布式事务能提供有价值的线性一致与 read-your-writes。它批评的是 XA 的故障和性能特征，以及跨异构系统普遍部署的困难。日志派生更松耦合、更易演化，但不能用“最终一致”掩盖用户刚写完却看不到、约束短暂失效等产品问题。

解决方式是按路径提供最小必要保证：写响应返回 source offset，后续读取等待视图 watermark 追过该 offset；或写后暂时从权威库读取；对必须同步成立的不变量留在同一事务/一致分区里。

## Batch + stream：差别主要是输入是否有边界

批处理读取一个 bounded dataset，计算完成并输出；流处理面对 unbounded input，持续维护结果。两者的底层相似性比产品分类更重要：都消费不可变输入，通过函数产生派生输出。

### Lambda architecture 的经验与代价

Lambda architecture 曾建议同时维护：批处理层从完整历史生成准确视图，流处理层生成低延迟近似视图，查询时合并。它用 batch 重算修正 stream 错误，但要求同一业务逻辑实现两次，两套结果常因边界、窗口和版本差异而不一致。

更理想的方向是同一逻辑既支持历史重放，也支持持续增量：

- 流引擎把历史文件或保留日志视作有限输入；
- 批任务按小批次/分区增量运行；
- 转换逻辑共享语义，执行方式按延迟与成本选择；
- backfill 与实时流有明确切换水位，避免重复或缺口。

今天常说的 batch、micro-batch、streaming 是延迟连续谱，不应让 API 名称决定正确性。真正要定义的是 event time、processing time、watermark、迟到数据策略、状态保留和重放结果是否确定。

### Maintaining derived state 是共同目标

搜索索引、物化视图、推荐特征和缓存都可看成 `state = fold(events)`。批处理从头 fold 全历史，流处理从 checkpoint 继续 fold 新事件。若转换是幂等、版本化且输入可重放，部署新逻辑可以：

1. 在新 namespace 从历史 backfill。
2. 记录 backfill 截止 offset。
3. 从该 offset 接上实时增量。
4. 并行比较旧、新视图。
5. 切换读流量，保留回滚窗口。

这比就地修改一个不可解释的派生库更安全。

## Unbundling databases：把数据库内部能力变成可组合数据流

数据库内部已经包含许多组件：WAL、索引、查询引擎、物化视图、复制、缓存和并发控制。它们在一个产品中通过内部接口紧密组合。**Unbundling** 的设想是：如果可靠的日志与流处理能够连接不同存储引擎，应用就可以像组合 Unix 工具一样，按需求组合数据系统，同时保留统一的数据流语义。

这不是鼓励“每个功能一个新数据库”。拆开组件会增加网络、部署、schema、权限、故障恢复和所有权成本。合理目标是让专业工具通过明确、可观测、可重放的接口组合，而不是隐藏在脆弱的脚本双写中。

### Composing storage：索引是数据库内置的派生视图

数据库的 secondary index 本质上由主表派生。写事务保证主表与索引按同一顺序变化；崩溃恢复会重放 WAL。把搜索引擎看作“跨系统 secondary index”，就能提出同样的问题：更新顺序在哪里？索引落后多少？能否重建？切换 schema 时如何并存？

同理，缓存是 materialized view，分析 cube 是聚合视图，ML feature 是带时间语义的派生列。它们不应拥有脱离来源的随意写入口。

### Application around dataflow：数据变化触发后续动作

传统应用把数据库当被动容器，由 request handler 主动读写。数据流设计则让变化成为一等输入：一个服务发布事实，其他服务订阅并维护自己的状态。它带来松耦合和独立演化，但事件 schema 成为公共接口，需要兼容策略、所有权和契约测试。

事件最好描述已经发生的领域事实，如 `OrderPaid`，而不是要求某个具体消费者执行的内部命令。消费者必须接受至少一次投递、重复、延迟和重放；副作用需要 inbox/dedup，外部 API 需要幂等键。事件不能随意删除或改变含义，否则历史重放会得到不同世界。

### Observing derived state：读路径也是数据流的一部分

写入是事件流，读取也可以看作客户端对派生状态的观察。若服务端维护状态并把变化推送给客户端，网页不必不断轮询；spreadsheet 的公式、响应式 UI 和数据库 materialized view 其实共享“输入变化自动传播”的模型。

原书设想数据库、缓存、搜索、客户端之间的更新可以更加增量化。工程上这对应 subscription、change feed、WebSocket/SSE 和 incremental view maintenance。但推送连接会断、消息可能合并或丢失，因此协议必须包含 snapshot + resume token：

```text
读取 snapshot @ revision 120
订阅 changes after 120
连接断开，最后确认 revision 147
从 148 恢复；若历史已 compact，则重新 snapshot
```

前端本地状态不能因“实时”而成为新的权威源；revision gap 必须触发重新同步。

## Aiming for correctness：Exactly-once 不能只靠一个组件声明

数据系统会丢请求、重复消息、超时、部分失败。仅仅在内部组件提供事务，不代表用户意图端到端只生效一次。客户端发出请求后连接断开，不知道服务端是否提交；若重试付款，可能重复扣款。这是跨越客户端、网络、服务、数据库的边界，必须端到端解决。

### End-to-end argument

底层可靠机制能减少错误，但最终正确性往往只能在知道业务语义的端点验证。TCP 去重数据包不能防止 HTTP 客户端重试整个请求；消息 broker 的 exactly-once 不能自动去重消费者对第三方支付 API 的调用。

常用端到端模式是 **idempotency key**：

1. 客户端为一次业务意图生成稳定 request ID。
2. 所有重试复用同一个 ID。
3. 服务在权威数据库用唯一约束原子记录 ID 与结果。
4. 已存在则返回第一次结果，不重复执行副作用。
5. 下游继续传播同一业务 ID，并各自在自己的效果边界去重。

去重记录的 TTL 必须覆盖最大重试、离线和消息保留窗口。若一天后删除 key，但客户端一周后恢复重试，“exactly once”承诺就有时间边界。对支付等长期不可重复动作，应以业务对象/支付意图作为永久身份，而非短期缓存键。

### 约束必须放在能看到所有竞争者的位置

唯一用户名、同一座位不可售两次、账户不能超额扣款，都要求竞争操作经过共同仲裁。唯一性通常需要 consensus，但可以通过 partitioning 缩小范围：按 username 哈希后，所有相同名字进入同一分区/leader；不需要让全球所有用户名共享一个锁。

在日志系统中，可以把 `claim(resourceId, owner)` 按 resourceId 路由到同一 partition，并让第一个合法事件获胜。消费者按日志顺序维护 claim 状态。关键点是**确认写成功前**必须由权威分区判定，而不是先让多个区域都成功、事后选一个 winner。

很多约束可以重构为可局部执行：给每个仓库预分配库存份额；给 region 发放唯一 ID 范围；使用 escrow 额度。这样牺牲部分资源利用率，换取分区期间的本地可用性。无法局部化的不变量就应诚实支付协调成本。

### Timeliness 与 integrity 是不同维度

- **Timeliness（及时性）**：用户观察状态是否足够新。违反时通常是暂时陈旧，等待追赶可恢复。
- **Integrity（完整性）**：数据是否无丢失、无矛盾、无重复副作用，并满足不变量。违反后不会仅靠等待自动修好。

异步派生常牺牲 timeliness，不必牺牲 integrity。搜索结果晚 30 秒出现可能可接受；订单永久漏索引或重复退款不可接受。把两者混成“consistency”会导致错误取舍。

工程契约应分别写：

```text
Timeliness SLO: 99.9% 的商品变更在 60 秒内进入搜索
Integrity invariant: 每个已确认商品版本最终恰好对应一个可检索状态，删除后不可复活
```

若业务必须 read-your-writes，可让客户端携带写入 offset，读取等待 search watermark ≥ offset；超时则显示“处理中”或回源数据库。不要为了低延迟返回确定错误的“找不到”。

### Trust, but verify：正确性要靠持续核验

事务、复制和 exactly-once 都是实现承诺，软件与硬件仍可能有 bug。磁盘静默损坏、错误 schema migration、CDC connector 跳过记录、消费者代码缺陷都会让系统在“所有组件绿色”时产生错误状态。

因此需要独立校验：

- 对源表与派生视图按 partition 比较 count、checksum、版本水位；
- 从不可变事实重算关键余额，与在线 materialized state 对账；
- 对资金使用 double-entry ledger，验证借贷总额守恒；
- 定期恢复备份并重放，而不是只检查备份任务成功；
- 对 repair 产生审计事件，保留修复前后值与责任链；
- 用不同实现路径计算高价值结果，减少共因 bug。

> [!note] 工程延伸：Reconciliation 是产品能力
> 对账不应只是事故脚本。每个 pipeline 都应有 source watermark、sink watermark、差异表、自动重试与人工处置队列。差异修复要幂等且可审计，避免“修复任务”反而覆盖更新数据。

### Data lineage：知道一个结果由什么产生

> [!note] 工程延伸
> 原书强调 dataflow、派生关系和可重算性；现代平台通常用 lineage 把这些关系显式记录。Lineage 至少连接 dataset、field、job、code version、schema version 和 source offsets。

当某个税率计算 bug 存在三天，lineage 应回答：哪些订单、报表、模型特征和外发文件受影响？从哪个 input range 用哪个新版本重算？没有 lineage，团队只能按表名猜影响面；有 lineage 才能做定向 backfill 和证明修复完成。

Lineage 元数据本身也会不完整。应从编排器、查询计划、schema registry 和运行时事件自动采集，并用“关键数据产品必须有 owner、freshness、lineage”的策略持续检查，而不是依赖人工 wiki。

## Privacy deletion：删除不是一条 SQL，而是反向数据流

> [!note] 工程延伸
> 原书深入讨论隐私、追踪与用户控制；下面把这些原则落到 CDC、派生视图和 ML 系统中的删除工作流。

当用户要求删除数据，权威库执行 `DELETE` 只是起点。数据可能已进入缓存、搜索、warehouse、对象存储、日志、备份、特征库、训练集和第三方出口。删除必须沿 lineage 传播：

1. 创建不可复用的 deletion request ID 和 subject identity 范围。
2. 权威系统写入 tombstone/删除事件，阻止旧更新复活。
3. 每个派生系统幂等执行删除并回报 completion evidence。
4. 协调器按系统清单和 deadline 跟踪，超时告警并升级人工处置。
5. 对不可逐条修改的 append-only 日志采用加密删除、分段重写或受控保留策略。
6. 备份在恢复时重新应用删除 ledger，不能让旧备份复活已删除主体。
7. 对训练产物评估是否需要重新训练、machine unlearning 或文档化的保留例外。

删除事件不能携带已删除的完整敏感内容，否则“删除流水线”本身制造新副本。使用稳定内部 subject ID，并把身份映射权限降到最小。审计应证明某 request 在哪些系统何时完成，而不重新暴露内容。

关键 race 是删除与并发写：一个延迟事件可能在 tombstone 后到达并恢复用户数据。消费者必须按 subject/version 比较，规定 deletion epoch 之后哪些事件合法；重新注册应创建新 identity generation，不能让旧 generation 的事件污染新账户。

## Doing the right thing：技术正确不等于社会上正确

系统可能完全满足 schema、事务与 SLO，却仍然伤害人。终章把视角从“系统是否按设计运行”提升到“这个设计是否应该存在、权力如何分配、错误由谁承担”。

### Predictive analytics 的边界

预测系统根据过去数据推断未来行为，可用于推荐、欺诈检测、信用、招聘或保险。但统计相关不等于因果，更不等于对单个人的确定事实。模型输出应被视为带误差和适用条件的估计，而不是客观真相。

训练指标与业务目标之间会错位。优化点击率可能奖励耸动内容；优化拦截率会让正常用户承担误报；只看整体 AUC 会隐藏小群体上极差的召回。上线前要定义错误成本由谁承担，并为高影响决策提供人工复核、申诉、解释和纠正入口。

模型还会改变它所预测的世界：被拒绝贷款的人不会产生还款结果，导致反馈数据选择性缺失；推荐系统曝光什么，下一轮训练就更容易看到什么。离线准确率无法捕获这种 feedback loop。

### Bias and discrimination

历史数据记录的是历史制度与行为，不是自然法则。即使删除敏感字段，邮编、学校、消费和社交关系也可能成为 proxy。偏差可能来自采样、标签、测量、缺失数据、目标函数和部署环境漂移。

公平没有一个可同时满足所有场景的单一指标。不同 base rate 下，校准、各群体误报率相等和召回率相等可能互相冲突。选择指标是价值判断，应由领域专家、受影响群体、法律与产品共同决定，而不是交给模型工程师凭默认阈值处理。

工程上至少要保留：训练数据 datasheet、feature lineage、模型版本、分群误差、decision log、override 与 appeal outcome。监控不仅看模型延迟，还要看长期 outcome 与群体影响。

### Privacy 与 tracking：收集能力不等于使用许可

行为追踪能个性化产品、测量实验，也能形成对个人极细的画像。商业激励常让客户/广告主利益优先于实际用户，数据在用户不可见的组织间流动。所谓“匿名数据”也可能通过稀有行为、位置轨迹或外部数据重新识别。

有意义的 consent 应具体、可撤回且不靠暗黑模式；用途改变应重新评估，而不是用一份宽泛条款永久授权。隐私设计包括：

- Data minimization：只收完成明确目的所需的数据。
- Purpose limitation：为目的 A 收集的数据不能默认用于高风险目的 B。
- Retention limit：按字段和用途设置删除期限，不是“以后可能有用”就永久保存。
- Access control：按最小权限隔离原始身份、行为事件与聚合结果。
- User agency：允许访问、更正、导出、撤回和删除，并给出可理解反馈。
- Differential privacy、aggregation 或 on-device computation：在适合场景减少中央明文数据。

安全与隐私相关但不等价。数据被严格加密、从未泄露，仍可能因过度收集或不当决策侵犯隐私；反过来，合法用途的数据若访问控制薄弱，也会造成安全事故。

## 把本章转成一套架构评审问题

| 主题 | 评审问题 |
| --- | --- |
| Authority | 哪个系统决定事实？是否存在绕过它的双写入口？ |
| Ordering | 并发变化在哪里排序？跨 partition 的因果如何传播？ |
| Replay | 输入保留多久？代码/schema 版本如何重算？ |
| Timeliness | 每个视图的新鲜度 SLO 与 read-your-writes 路径是什么？ |
| Integrity | 哪些不变量必须同步成立？如何去重和 fencing？ |
| Verification | 谁与谁对账？差异如何自动修复和审计？ |
| Lineage | 一个错误版本影响哪些字段、报表、模型和出口？ |
| Deletion | tombstone 如何穿过日志、备份、特征和第三方？ |
| Human impact | 误判伤害谁？是否可解释、申诉、撤回和纠正？ |

这张表的顺序很重要：先确定权威输入和顺序，才谈工具；先定义不变量和用户承诺，才谈 exactly-once；先知道数据流向哪里，才可能谈隐私控制。

## 常见理解偏差

1. **用了 Kafka 就有 event-driven correctness**：日志只提供其范围内的顺序和保留；生产者双写、消费者副作用和 schema 演化仍需设计。
2. **所有数据都是 system of record**：没有权威/派生边界，就无法裁决冲突或安全重建。
3. **Eventual consistency 等于可以无限陈旧**：应为 timeliness 定义 SLO，并给关键路径提供水位等待或回源。
4. **Exactly-once 是 broker 的开关**：端到端副作用必须依赖稳定业务身份、幂等与结果记录。
5. **派生视图可重建，所以无需备份与监控**：输入可能过期，重建可能耗时数周，转换 bug 也可能重复生成错误。
6. **Batch 准确、stream 近似**：准确性取决于事件时间、迟到处理、幂等和转换逻辑，不由执行模式名字决定。
7. **2PC 与事件日志谁都能替代谁**：前者提供同步原子决定并可能阻塞，后者支持异步重试但需暴露中间状态。
8. **CDC 保证下游永远一致**：connector、schema、日志 retention 和消费者都可能出错，必须对账。
9. **删除主库记录就满足删除**：派生视图、日志、备份、模型和第三方都需要可证明的传播策略。
10. **去掉敏感字段就没有偏差或隐私风险**：proxy 特征和组合数据仍可歧视或重新识别。
11. **模型准确率高就能自动做高风险决定**：需评估群体误差、反馈循环、申诉与错误成本。
12. **Trust but verify 表示不信任团队**：独立核验是承认任何实现都会有 bug，并保护团队快速发现与修复。

## 本章小结

- 复杂应用必然组合专用工具；必须用权威输入、统一顺序和可重放转换管理派生数据。
- Batch 与 stream 都是从输入维护派生状态，核心差别是输入边界、延迟和增量执行方式。
- Unbundling 让日志、索引和视图跨工具组合，但接口、schema、故障与所有权成本不会消失。
- 端到端正确性依赖业务幂等键、局部化约束、及时性/完整性分离和独立对账。
- CDC、lineage、watermark 与 deletion workflow 让数据流可以观测、修复并回应隐私权利。
- 预测分析的准确率不能替代对偏差、反馈循环、申诉和人类影响的判断。
- 数据系统的最高目标不是保存数据本身，而是可靠地服务人的正当目的，并让人保有控制权。
