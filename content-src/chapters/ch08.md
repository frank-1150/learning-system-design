## 这一章真正要改变的直觉

单机程序给工程师一种非常舒服的抽象：一次 CPU 指令要么执行，要么机器崩溃；写入内存后，下一条指令能看到它；同一个程序在同样输入下大体有确定结果。分布式系统打破的不是某个 API，而是这种“世界有一个可直接观察的当前状态”的直觉。

多个节点之间没有共享内存，唯一的信息来源是网络消息。一条请求发出后没有响应，可能是：

1. 请求在途中丢失。
2. 请求在交换机、内核或接收方队列中等待。
3. 接收节点已经崩溃。
4. 接收进程正在 GC、抢不到 CPU 或等待磁盘。
5. 请求已经执行，但响应丢失。
6. 请求已经执行，响应仍在某处排队。

这些情形对发送方的观测完全一样：**暂时没有收到响应**。超时可以让调用方停止等待，却不能告诉它远端是否执行了操作。

```text
Client ── charge(order-7) ──X?──> Payment
Client <──── response ──────X?─── Payment

timeout 后：
- 没扣款？
- 已扣款但响应丢了？
- 请求稍后才到并扣款？
```

因此 Ch8 真正建立的是一套认识论：节点只能根据收到的有限消息作判断；沉默不是失败证明；本机“我还活着、我还是 leader、我的 lease 还有效”都可能与集群已经接受的现实不同。可靠算法必须声明自己在哪个 **system model（系统模型）** 下成立，并把不确定性限制在明确边界里。

## Partial failure：一半正常比全部崩溃更难

分布式系统的定义性难题是 **partial failure（部分失效）**：有些组件正常，有些组件损坏，还有一些只是非常慢。服务 A 能调用 B，B 却回不来；同一机架内通信正常，跨机架失败；数据库端已经提交，应用端仍认为请求失败。

全部崩溃反而容易理解：没有结果、从 checkpoint 重启。部分失效让不同参与者形成不同且都“符合各自观测”的判断，调用者还必须决定是否重试、降级、切主或等待。

### Cloud 与 supercomputer 选择了不同故障哲学

HPC 超级计算机常运行可停止的科学计算，节点和互连较专用。一个节点失败时，可以停止整个 job，修复后从 checkpoint 重跑。它把部分失效升级成整体失效，换取更简单、接近单机的模型。

互联网服务通常不能这样做：

- 在线请求要求全天低延迟，不能因一台机器维护而停止整个集群。
- 云上使用多租户通用硬件和 IP 网络，节点、宿主机与资源争用不受应用完全控制。
- 数千节点规模下，“某处总有东西坏着”比“偶尔全都健康”更现实。
- 跨地域部署为了贴近用户，通信必然经过更慢、更复杂的网络。
- 滚动发布、弹性替换和机房维护都要求服务在局部变化时继续工作。

所以 cloud 风格系统通常接受较不可靠的组件，再用复制、隔离和协议构造较可靠的整体。这并非能无限消除底层故障：TCP 可隐藏丢包、乱序和重复，却无法消除无限延迟；纠错码可纠正有限 bit error，却不能从完全淹没的信号中恢复数据。每层抽象都必须说明它隐藏了什么、仍向上泄露什么。

## Unreliable network：timeout 是策略，不是事实探测器

数据中心常见的 Ethernet/IP 是异步 packet network：协议不承诺消息一定到达，也不承诺最大到达时间。TCP 提供可靠字节流，但“可靠”只表示连接正常时通过确认与重传交付有序字节；它无法保证对端进程会及时处理，也无法在有限时间内证明对端永久死亡。

网络延迟来自很多队列叠加：交换机端口拥塞、TCP send/receive buffer、操作系统 run queue、虚拟机 steal time、应用线程池、磁盘 I/O 和对端负载。一个 RPC 的“网络超时”可能根本不是网线问题，而是消息已经到达、迟迟没有轮到业务线程。

### Timeout 怎么选都存在代价

Timeout 太短：正常慢请求被当成失败，触发重试和 failover；额外负载又让系统更慢，形成 retry storm。Timeout 太长：真实故障发现慢，用户等待久，连接和线程等资源被占住。

不存在从理论上推导出的“正确 3 秒”。实践中应从端到端 SLO、正常延迟分布、恢复目标和重试预算出发：

```text
用户总预算 2s
├─ edge + auth          150ms
├─ service A            250ms
├─ dependency B         500ms（含最多一次受控重试）
├─ database             400ms
└─ 排队与安全余量        700ms
```

每层都独立设置 2 秒会让最坏路径远超用户预算。Deadline 应向下传播，依赖收到的是“还剩多少时间”，而不是重新开始自己的完整 timeout。

### 重试必须先回答“第一次可能成功了吗”

读通常可以重试；写则必须区分语义。支付、发券和创建订单应携带稳定 idempotency key，由服务端记录该 key 对应的状态和响应。仅在客户端“尽量不重试”不够，因为网关、SDK、消息队列也可能重放。

```text
POST /charges
Idempotency-Key: order-7-payment-v1

首次到达：原子地写 key -> PROCESSING/COMPLETED
重复到达：返回同一结果，不再次产生副作用
```

指数退避、jitter、并发上限与 retry budget 用来保护系统容量；它们解决的是放大问题，不解决重复副作用。Exactly-once 体验通常来自 at-least-once delivery 加幂等状态机，而不是网络恰好只传一次。

### Failure detector 输出的是怀疑程度

心跳超时无法区分节点故障与网络故障。固定阈值只会给出二元判断；phi accrual 一类 detector 根据历史到达间隔输出 suspicion level，让上层按风险选择动作。即便如此，它也没有获得真相，只是在更好地量化证据。

工程上应把“怀疑”与“执行破坏性动作”分开：暂停向节点发新流量可以较激进；删除其副本、选出可写新 leader 或重建大量数据则需要 quorum、epoch 与资源护栏。

## Synchronous 与 asynchronous network：确定性要用利用率购买

传统电话电路在建立连接时预留端到端带宽，因此没有与其他流量动态争抢造成的排队，能够给出 bounded delay。它是 synchronous network 的典型直觉。TCP connection 并不是电路：连接空闲时几乎不占带宽，活跃时机会式使用可用容量。

Packet switching 适合网页、文件与 RPC 这种突发流量。所有租户动态共享链路，资源利用率高、单位成本低，但高利用率必然带来可变排队。CPU 时间片、共享磁盘和虚拟机也有同样权衡：静态预留能提供更强上界，却会让空闲份额无法被别人利用。

借助专用硬件、admission control、QoS、实时操作系统和保守容量，可以建立确定延迟上界。但大多数通用云服务选择统计 SLO 而不是 hard real-time guarantee。不要把日常观测到的 P99 当作协议上界；一次拓扑重收敛、bufferbloat 或宿主机暂停就可突破它。

> [!note] unbounded 不等于一定无限慢
> 异步模型说的是算法不能假设一个永不被突破的固定上界。生产网络绝大多数时间仍然很快，因此工程上常使用“部分同步”模型：平时遵守经验边界，异常时允许任意突破。

## Clocks：先区分 duration 与 point in time

机器至少有两类时钟，它们不能互换。

| 时钟 | 用途 | 关键性质 | 不适合 |
| --- | --- | --- | --- |
| monotonic clock | duration、timeout、latency | 同一进程视角持续向前，绝对值无意义 | 跨机器比较、日历时间 |
| wall/time-of-day clock | 发布时间、日志日期、日程 | 试图对应 UTC，可由 NTP 校准 | 可靠测量耗时、直接排序因果事件 |

测量一个 RPC 用 monotonic clock：`elapsed = mono_end - mono_start`。用 wall clock 相减可能遇到 NTP step 导致负延迟或突然变长。反过来，给用户展示“北京时间几点”必须使用 wall clock；monotonic 的数值可能只是开机以来 tick，跨重启没有意义。

### NTP 校准不等于完全同步

每台机器的石英振荡器会 drift，温度也会影响频率。NTP 通过网络与时间源比较，通常用 slewing 缓慢调整速度，有时在偏差过大时 step 跳变。准确度受到网络往返时间、非对称延迟、上游服务器错误、防火墙、虚拟化暂停和 leap second 策略影响。

“时间戳有纳秒字段”只代表 API resolution，不代表 accuracy。一个更诚实的物理时钟读数是区间：

```text
TrueTime.now() -> [earliest, latest]
真实时间位于区间内，但不知道确切是哪一点
```

若事件 A 的最晚可能时间仍早于 B 的最早可能时间，才能凭物理时钟断言 A 在 B 前；区间重叠则无法确定。Spanner 用专门时间基础设施缩小 uncertainty，并在提交时等待不确定区间过去。不能只复制“用 timestamp 排序”这一步而忽略它为误差边界付出的硬件、监控与 commit wait。

### Wall clock 不能可靠表达因果顺序

事件 B 读取了 A 的结果再写入，因此 B 因果上一定在 A 之后；但 B 所在节点的 wall clock 可能慢几毫秒，让 B 的 timestamp 小于 A。用 last-write-wins 会错误丢掉 B。即使 NTP 平常误差很小，只要误差大于事件间隔，就可能违背因果。

逻辑时钟、版本向量或 consensus log index 表达的是 happened-before/协议顺序，不声称“现实世界几点”。要决定覆盖关系，优先使用能反映因果的版本；wall clock 可作为展示、保留策略或有明确容错的 tie-breaker，而不是偷偷承担正确性。

### 时间依赖必须被监控

若系统确实依赖 clock accuracy，就应监控每节点 offset、frequency error、上次成功 sync、上游 source 数量和 uncertainty；超过边界的节点应停止承担需要准确时间的角色。时钟故障最危险之处在于机器仍看起来正常，却静默地产生错误排序或过早过期。

## Process pause：代码两行之间可能过去一分钟

即使用 monotonic clock 正确判断 lease，在“检查有效”和“执行写入”之间，线程仍可能暂停：

```text
if lease.remaining() > 10s:   # 此刻有效
    [进程暂停 60s]
    write(resource)           # 此刻早已失效
```

暂停来源包括 stop-the-world GC、虚拟机 live migration、容器 CPU throttling、OS 调度与 steal time、同步磁盘/网络盘 I/O、page fault 和 swap、SIGSTOP、笔记本休眠等。进程恢复时从原指令继续，并不会自动收到“你已睡了一分钟”的异常。与此同时，其他节点已把它判死并选出新 leader。

这与线程并发的根本困难相似：代码在任意点都可被抢占。单机有 mutex 和共享原子变量；分布式节点无法读取共同内存，只能通过也可能延迟的消息协调。

减少 pause 很有价值：控制 heap、避免 swap、监控 GC/steal time、在 GC 前 drain traffic、滚动重启累积大量长寿命对象的进程。但这些是降低发生率和影响，不是正确性证明。协议仍需在任意 pause 后保持 safety。

## Majority 定义可接受的事实，而不是节点自我感觉

一个只能收不能发的节点会觉得自己工作正常，其他节点却收不到它的回复；一个暂停后恢复的旧 leader 仍保留内存中的任期和请求。分布式算法不能让单节点凭自我判断决定全局唯一角色，通常由 quorum 决定 leader、锁或提交状态。

多数 quorum 的核心性质是任意两个多数集合必相交。3 节点可容忍 1 个失效，5 节点可容忍 2 个；网络分裂后至多一侧拥有多数，因此协议可以让少数侧停止写入。这里“多数决定事实”不是说多数永远正确，而是协议选择多数的已提交历史作为后续参与者必须遵守的权威。

然而 quorum 只决定谁获得新权利，不能自动阻止旧持有者继续触碰外部资源。这个缺口正是 fencing 要解决的。

## Lease 与 fencing：从工程直觉推导安全边界

Lease 是会到期的 lock。它有两个主要用途：持有者故障后不需永久等待释放；正常期间减少每次操作都向协调服务确认的成本。但 **lease 本身不是外部资源的互斥保证**，因为旧持有者可能暂停到 lease 过期，然后恢复并继续执行。

### 一个正确性推导

假设 lock service 依次授予 client A、B：

```text
t0: A 获得 lease，token=33
t1: A pause
t2: lease 到期
t3: B 获得 lease，token=34，并写 storage
t4: A 恢复，携带旧 token=33 写 storage
```

若 storage 只相信“客户端说自己拿过锁”，A 与 B 都是合法持有者，文件可能被旧写覆盖。正确方案要求 lock service 每次授予都返回严格递增 fencing token；**最终被保护的 storage** 记录已接受的最大 token，拒绝更小 token。

```text
accept write(token) iff token >= max_token_seen
after accept: max_token_seen = token
```

这样 A 是否知道 lease 过期不再重要。即使它暂停一小时、网络延迟一小时，token 33 在 token 34 生效后也不能改变资源。

### Fencing 成立需要哪些条件

1. Token 必须唯一且按授予顺序单调增加；普通随机 UUID 只有唯一性，没有先后关系。
2. Resource 必须原子比较并更新 `max_token_seen`；先读后写的非原子实现仍有竞态。
3. 每一条有副作用的路径都必须携带并校验 token，包括后台任务和重试旁路。
4. Token 的作用域要明确：全局、tenant、partition 或具体 object。不同资源的 token 不应误作可比。
5. Resource 的持久状态必须保留已见 token；重启后忘记 fence 会重新接受旧写。
6. 若写还会调用下游，下游也需继承 fence 或通过同一受保护提交点产生不可逆结果。

ZooKeeper 的 zxid/cversion、consensus term 加 log index 或数据库递增 generation 可作为 token 来源。若外部资源无法校验 token，可把 generation 编入不可覆盖的对象名并通过一个支持 compare-and-set 的 pointer 发布；仅在文件名里放 token、但读者仍随意选文件，并没有完成 fencing。

### Lease 的时钟边界

若 lease 由同一 authority 按其时钟判断到期，客户端不应用自己的 wall clock 声称仍有效。客户端可用本地 monotonic clock 保守估计并提前停止工作，但 clock drift 和 pause 仍意味着“客户端检查”只是一种优化。Safety 应由 authority 的新授予规则和资源侧 fence 保证。

Lease duration 也不是越长越安全：长 lease 降低续租频率，却延长故障接管；短 lease 恢复快，却更易受 jitter、GC 和负载影响。应分别定义 renewal deadline、grace、takeover 条件，并做故障注入验证，而不是把 30 秒当魔法常量。

## Byzantine faults：失败与说谎是两个不同模型

本书大部分协议假设节点 **unreliable but honest**：它可能崩溃、暂停、丢失内存状态或持有过期信息，但响应时会按协议、按自己所知说真话。Byzantine fault 则允许节点发送任意矛盾或伪造信息，甚至主动欺骗不同参与者。

Fencing 能阻止诚实但过期的 client，却阻止不了恶意 client 直接伪造更大的 token，除非 token 还有不可伪造的认证、资源验证其 authority。BFT 协议适合参与组织互不信任、航空航天遭受任意硬件错误，或开放 P2P 网络没有中央 authority 的环境。它通常需要更高副本数、更多消息和更复杂的实现。

普通公司内部数据系统通常不因此直接上 BFT：

- 所有节点部署同一份有 bug 的二进制，多数投票不会消除共同模式故障。
- 攻击者若通过同一漏洞取得所有相同配置节点，副本数量也不会提供独立信任域。
- 身份认证、授权、TLS、密钥隔离、审计和输入校验往往更直接有效。

但可以低成本防御“弱形式说谎”：应用层 checksum、协议长度与范围校验、多个 NTP 源排除离群值、持久化数据校验和、消息 schema 校验。这些不是完整 BFT，却能把硬件损坏、配置错误和普通 bug 更早转成可见失败。

## System models：先声明世界，才能证明算法

系统模型把现实压缩成算法允许依赖的假设。按 timing 常见三类：

| 模型 | 网络、暂停、时钟假设 | 工程现实 |
| --- | --- | --- |
| synchronous | 都有已知固定上界 | 通用云环境通常不成立 |
| partially synchronous | 大多数时间有界，异常时可任意突破；最终恢复 | 多数生产系统最实用 |
| asynchronous | 不允许任何 timing 假设，甚至没有 timeout | 推理强但限制很大 |

按 node failure 常见三类：

- **Crash-stop**：节点一旦崩溃永不回来。
- **Crash-recovery**：节点可崩溃并在未知时间恢复；内存丢失，stable storage 保留。
- **Byzantine/arbitrary**：节点可以做任何事，包括撒谎。

实际系统最常用 partially synchronous + crash-recovery。但其中“stable storage 一定保存”仍是抽象：磁盘固件、误删、静默损坏或错误恢复可能让节点失忆。理论模型帮助我们缩小问题，不会替实现处理所有现实意外。工程上要同时做模型内证明、故障注入、checksum、备份恢复和运维逃生路径。

## Safety 与 liveness：故障时宁可停，也不能错

一个 fencing token 服务可要求：

- Uniqueness：不同授予不能得到相同 token。
- Monotonicity：后完成的授予得到更大的 token。
- Availability：未崩溃的请求者最终得到 token。

前两项是 **safety**：一旦违反，可以指出发生违反的具体操作，错误已无法抹去。第三项是 **liveness**：当前没有完成仍可能未来完成，定义中经常出现 eventually。

分布式算法通常要求 safety 在所有模型允许的故障下都成立，即使整个网络断开，也不能发出两个冲突的已提交结果；liveness 可以附条件，例如“只要多数节点可用且网络最终恢复，请求最终完成”。这解释了为什么少数分区应拒绝写：它牺牲暂时可用性，守住不会双主提交的 safety。

评审任何“高可用”设计时都应把二者分开问：

```text
Safety：网络永久断开时，系统绝不会做错什么？
Liveness：在哪些恢复条件下，一个正确请求最终会完成？
```

只说“保证一致性和可用性”没有给出可检验的性质；只做 happy-path 压测也无法证明 safety。

## 把本章落到生产设计

### 故障处理矩阵

| 观测 | 允许推断 | 推荐动作 | 不应直接做 |
| --- | --- | --- | --- |
| RPC timeout | 截止前无响应 | 幂等重试、查询操作状态、降级 | 断言远端未执行 |
| 心跳缺失 | 节点被怀疑不可达 | 摘流、收集 quorum 证据 | 单节点决定删除副本 |
| clock offset 超限 | 时间语义不可信 | 隔离时间敏感角色、修 NTP | 继续用 LWW 覆盖数据 |
| leader 恢复 | 进程还活着 | 重新确认 term/epoch | 沿用暂停前 lease 写入 |
| 网络恢复 | 消息可再次流动 | 对账、repair、清旧请求 | 假设双方状态自动相同 |

### 应监控与演练什么

- RPC latency histogram、timeout reason、retry amplification、in-flight 与 deadline exhausted。
- 各节点 clock offset、sync age、source health、monotonic/wall clock 使用审计。
- GC pause、scheduler delay、CPU steal、disk stall、event-loop lag。
- Leader term/epoch、lease renewal latency、fencing rejection、旧 leader 请求数。
- Network drop/delay/reorder、单向隔离、跨 AZ 与控制面/数据面分别失联。
- Quorum availability、少数侧行为、恢复后的 log divergence 与 repair 进度。

故障注入不能只 `kill -9`。干净 crash 是最好处理的一类；更有价值的是延迟响应、丢单向包、暂停进程、耗尽线程池、让时钟偏移、让磁盘极慢，以及让旧 leader 在新 leader 提交后恢复。验证点不是“服务最终绿了”，而是资金、唯一性、版本单调性等业务不变量从未破坏。

## 常见理解偏差

1. **Timeout 证明节点死了**：它只证明本地截止前没收到响应。
2. **TCP 可靠，所以 RPC 不会不确定**：TCP 无法让调用者知道断连前应用是否已提交。
3. **延迟有 P99，所以可以当最大值**：统计分位不是协议保证，异常可任意突破。
4. **NTP 后 timestamp 可以排序所有事件**：resolution 不等于 accuracy，物理时钟可能违背因果。
5. **Monotonic clock 解决分布式排序**：它只适合单机测 duration，跨机器值不可比较。
6. **GC 调优后可以依赖 lease 检查**：任何代码点仍可能 pause，正确性需要资源侧 fencing。
7. **Quorum 选了新 leader，旧 leader 就消失了**：旧进程可能恢复，必须以 term/token 拒绝旧写。
8. **UUID 可以当 fencing token**：唯一不代表可排序，资源无法知道哪个更新。
9. **BFT 能解决相同软件 bug**：相同实现和配置会产生相关故障，投票无法创造独立性。
10. **证明算法正确就等于实现可靠**：证明依赖模型假设，现实还包含磁盘损坏、误配置和实现 bug。

## 本章小结

- Partial failure 让不同节点拥有不同观测；远程调用无响应时，成功、失败和延迟不可区分。
- Packet network 和共享计算资源以更高利用率换取可变、无硬上界的延迟；timeout 是风险策略而非真相检测。
- Duration 应用 monotonic clock；wall clock 只近似现实时间，NTP 不能自动提供因果顺序。
- 进程可在任意指令处暂停并在旧状态上恢复，所以“我仍持有 lease”不能作为最终写权限。
- Quorum/term 决定新 authority，单调 fencing token 由最终资源校验，才能阻止复活的旧 owner。
- Byzantine fault 允许节点任意撒谎，和本书主要采用的诚实 crash/pause 模型不同；是否付出 BFT 成本取决于信任边界。
- 系统模型明确算法依赖的 timing 与 failure 假设；safety 必须始终守住，liveness 可以以多数存活和网络最终恢复为条件。
