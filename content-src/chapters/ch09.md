## 这一章要解决的不是“不出错”，而是出错后仍能达成正确决定

第八章建立了分布式系统的现实：网络会丢包、重复、乱序和任意延迟；时钟只能近似；进程会暂停、崩溃，节点无法仅凭超时区分“对方死了”和“链路慢了”。第九章不再罗列故障，而是寻找能把这些不确定性封装起来的通用抽象。

最重要的抽象是 **consensus（共识）**：多个节点即使遇到故障，也对某个决定达成一致。例如单主数据库故障后，所有存活节点必须同意谁是下一任 leader；两个节点同时认为自己是 leader，就会产生 split brain 和数据损坏。

但在直接讨论共识算法前，原书先追问三层问题：

1. 复制系统对读取能给出多强的一致性保证？
2. 没有可靠物理时钟时，如何描述事件的先后与因果？
3. 多个参与者如何原子提交事务，又如何在节点故障后继续做决定？

这条主线把 linearizability、causality、total order broadcast、2PC 与 consensus 连在一起。它们不是可以互换的 buzzword，而是保证范围、成本和故障模型各不相同的工具。

## Consistency guarantees：先说清“像一份数据”具体指什么

复制系统的节点在同一时刻看到不同值并不罕见。**Eventual consistency** 只承诺：停止写入、网络故障最终修复并等待一段未指定的时间后，副本最终收敛。称它为 convergence 也许更准确，因为它没有说明收敛前读会返回什么，也没有给出时间上界。

这种弱保证难用，是因为数据库表面上像一个变量，行为却不像单线程变量：刚赋值后可能读回旧值；两个客户端可能同时得到不同答案；故障和高并发下才出现的边界又很难靠普通测试复现。

更强保证使应用更容易正确，但从来不是免费的。它可能增加协调、延迟，或在网络分区期间降低可用性。还要区分两组常被混淆的概念：

- **Transaction isolation** 主要处理并发事务间的竞态，并定义多个对象上的事务效果。
- **Distributed consistency** 主要处理复制节点在延迟与故障下呈现什么状态。

Serializability 与 linearizability 可以组合成 strict serializability，但单独拥有其中一个并不自动得到另一个。

## Linearizability：让复制系统表现得像一个原子寄存器

**Linearizability（线性一致性）**制造“系统只有一份数据”的幻觉。每个读写操作看起来在调用与返回之间的某个瞬间原子生效；一旦写操作成功返回，之后开始的所有读都必须看到该值或更晚的值，不能回到旧副本。

它是一个 **recency guarantee（新鲜度保证）**，并且尊重真实时间先后。如果操作 A 已返回后操作 B 才开始，那么全局顺序中 A 必须在 B 之前。若 A 和 B 的执行区间重叠，它们可以按任一顺序线性化，只要所有客户端看到的解释一致。

```text
时间 ─────────────────────────────>
Client A:  write(x=1) [──────] OK
Client B:                         read(x) [──]  必须为 1 或更晚

Client C:       read(x) [──────────]
                与 write 重叠，可返回旧值或新值
```

判断一个历史是否线性一致，不是看各副本最后是否相同，而是能否把每个操作放到一个合法的线性化点，同时保留所有不重叠操作的真实时间顺序。某客户端已读到 `x=1` 后，另一个更晚的读又得到 `x=0`，就无法维持单副本幻觉。

### Linearizability 不等于 serializability

- **Serializability** 是事务隔离保证：多个事务的结果等价于某种串行顺序。它不要求该顺序尊重真实时间。
- **Linearizability** 通常描述单个寄存器/对象上的读写新鲜度，不自动把多对象事务组合成一个原子单位。
- **Strict serializability** 同时提供 serializability 和 linearizability，事务既像串行执行，也尊重现实中的先后。

例如一个数据库可以通过 snapshot isolation 让只读事务看到一致的旧快照，却不满足 linearizability；也可以对单 key 提供线性一致读写，但跨两个 key 的应用流程仍发生 write skew。

### 哪些场景真的依赖 linearizability

**Leader election 与锁。** 所有节点必须对锁的唯一持有者或当前 leader 达成一致。ZooKeeper、etcd 一类协调服务常提供线性一致操作，再用 fencing token 防止已过期持有者继续写外部资源。只有锁服务线性一致还不够，下游必须验证递增 token。

**约束与唯一性。** 用户名唯一、同一文件只能一个人持有写锁、库存不能重复售出，都需要所有竞争者对一个最新决定进行协调。不是所有约束都需要全局线性一致：电子邮件地址可按哈希分区，把每个值的唯一性局部化；某些库存也可以预先把额度分配给区域，从而减少全局协调。

**跨渠道依赖。** 用户上传文件后，消息队列通知转码器读取。如果队列消息先到，而文件存储的副本仍返回旧状态，转码器会误判文件不存在。此类问题可用线性一致存储，也可通过携带版本、从同一日志消费，或把对象创建与通知纳入明确的 happens-before 流程解决。

工程判断不应是“整个数据库要不要强一致”，而是找出真正依赖最新值的状态机：leader epoch、租约、幂等键、余额、不重复预订。把协调范围缩到这些对象，其他分析查询或派生视图可以容忍更弱保证。

### 常见系统如何实现或破坏 linearizability

- **单 leader 复制**：从 leader 读取且 leader 身份确定时可能线性一致；异步 follower 读通常不是。网络分区中的旧 leader 若仍接受请求，也会破坏保证，因此需要 quorum/consensus 与 fencing。
- **Consensus algorithm**：Raft/Paxos 类状态机复制通常能提供线性一致操作，但读取也必须确认当前 leader 仍持有有效权限，例如经过 quorum read/read index 或有效 lease。
- **Multi-leader**：多个写入口异步复制，本质上允许并发冲突，通常不线性一致。
- **Leaderless quorum**：即使 `w+r>n`，并发写、sloppy quorum、last-write-wins 和故障恢复仍可能产生非线性历史。Dynamo 风格 quorum 不能仅靠公式宣称 linearizable。
- **单 CPU 核/单机**：如果进程崩溃恢复和缓存层没有破坏顺序，原子指令或串行事件循环可以很自然地实现；“分布式”不是获得该保证的前提。

Linearizable compare-and-set（CAS）尤其重要。普通读后写之间存在竞态，CAS 只在当前值仍等于预期值时更新，是锁、leader epoch 和唯一注册的基本原语。

### 成本：网络分区时不能同时承诺正确响应与每个请求都有响应

假设两个数据中心之间断网。若两边都继续处理线性一致的同一对象写，它们可能做出互相冲突的决定；若坚持单一最新顺序，至少一边必须拒绝或等待无法联系 quorum 的请求。这里的取舍不是产品偏好，而来自节点无法确定另一侧状态。

#### CAP 的准确边界

“Consistency、Availability、Partition tolerance 三选二”是误导性口号。正式 CAP 结论的范围很窄：

- Consistency 指单对象的 linearizability，不泛指 ACID、一致快照或数据正确。
- Availability 指每个送达非故障节点的请求最终都获得非错误响应，没有现实 SLO 的延迟上界。
- Partition 是节点间部分消息永久丢失的特定网络故障，不是一个平时可自由关闭的功能。

结论是：**发生网络分区时，线性一致系统若无法联系必要节点，就不能同时对所有分区继续成功响应。** 正常无分区时并不存在“只能选 C 或 A”；现实系统还要讨论延迟、节点崩溃、数据丢失、退化模式、局部对象和不同操作的保证，而 CAP 对这些几乎没有帮助。

更实际的设计表达是：“region 间断网时，账户扣款拒绝写但允许带水位的旧读；商品浏览继续服务本地缓存；购物车接受并发更新并稍后合并。”这比给系统贴 CP/AP 标签信息量大得多。

即使网络没有被判定为 partition，线性一致通常也更慢，因为每次协调要等待网络与 quorum；无法可靠区分“慢”和“断”意味着尾延迟始终存在。强一致的成本是日常延迟与故障可用性的共同结果，不只在 CAP 故障发生时支付。

## Ordering and causality：顺序是分布式正确性的骨架

分布式系统没有天然全局时钟，但许多事件之间仍有真实逻辑依赖：先创建记录，才能更新它；先发出问题，才可能有回答；先读到配置版本 7，之后基于它的写就依赖版本 7。这就是 **causality（因果关系）**。

若事件 A 导致或影响 B，则 A happens-before B。happens-before 是 partial order：某些事件有先后，彼此独立的并发事件则不可比较。因果关系会出现在多个主题中：

- consistent prefix reads 防止先看到回答再看到问题；
- snapshot isolation 需要快照包含某事件时也包含它的因果前驱；
- write skew 是读取旧状态后并发决策造成的因果异常；
- fencing token 要保证后任的写排在前任之后；
- 消息系统必须在确认输入之后才发布依赖它的输出。

Linearizability 自动蕴含 causal consistency，因为它把所有操作排成尊重真实时间的 total order。反过来不成立：因果一致只要求因果相关操作顺序正确，并发操作可以在不同副本以不同顺序出现。

这使 causal consistency 成为重要中间点。它比 eventual consistency 强，能保留应用逻辑；又不必把所有独立操作都放到单一全局顺序中，因此在网络延迟环境下通常比 linearizability 更可用。原书指出，在允许网络延迟和分区的模型中，因果一致是不会因协调而牺牲可用性的最强一致模型之一。

### 捕获因果依赖的代价

系统需要知道一次操作读过哪些版本，才能判断它依赖什么。最直接的方法是携带完整依赖集合或 version vector，但跨大量节点和对象会产生元数据、传播与垃圾回收成本。实践中常通过 session token、分区内日志位置、hybrid logical clock 或限制事务范围做近似与压缩。

不要把“消息按 Kafka partition 有序”直接等同于因果一致。若一个业务动作从数据库提交，经 outbox 到多个 topic，再由消费者写入另一个数据库，因果链跨越多个日志；必须传播 correlation/version，或把需要顺序的事件路由到同一 key/partition。

## Sequence numbers：用逻辑时钟表达顺序

物理时间戳会受时钟偏差影响，不能可靠表示因果先后。系统可给操作分配 sequence number/version number：单 leader 每次递增计数器，就能形成与复制日志一致的全序。但多 leader 或无 leader 场景没有单一计数器，需要可扩展的逻辑顺序。

### Lamport timestamp

每个节点维护一个本地 counter：

1. 本地产生事件时 counter 递增。
2. 发送消息时携带 counter。
3. 收到远端值 `t` 后，把本地 counter 更新为 `max(local, t)`，再递增。
4. 时间戳使用 `(counter, nodeId)`；counter 相同时以唯一 nodeId 打破平局。

这样每两个 Lamport timestamp 都可比较，形成 total order；如果 A happens-before B，则 `L(A) < L(B)`。但逆命题不成立：`L(A) < L(B)` 不代表 A 导致 B，可能只是 nodeId 或计数规则给并发事件定了人为顺序。

这也是 Lamport timestamp 与 version vector 的关键区别：

- Lamport timestamp 紧凑地产生与因果兼容的 total order，但无法判断两个操作是否并发。
- Version vector 能判断一方支配另一方还是并发，但元数据随参与者增长，且只提供 partial order。

### 有序编号不自动解决唯一约束

若两个节点并发注册同一用户名，Lamport timestamp 可以事后决定谁排在前面，但节点收到写时并不知道是否还有一个更小时间戳的请求正在网络中。要安全地立即确认 winner，节点必须知道所有可能更早的操作都已到达，这需要 total-order broadcast 或其他协调。

因此“能给事件排序”和“能在请求返回前确认全局决定”是两种能力。雪花 ID、UUIDv7、数据库序列能提供唯一或大致有序标识，不自动提供 linearizability、因果跟踪或唯一业务约束。

## Total-order broadcast：让所有节点交付同一条日志

**Total-order broadcast（全序广播，也称 atomic broadcast）**要求：

- **Reliable delivery**：一条消息被交付给一个非故障节点后，最终会被所有非故障节点交付，消息不会丢失。
- **Totally ordered delivery**：所有节点严格按相同顺序交付消息。

它通常表现为 append-only log：节点可以在日志末尾追加消息，所有节点按相同 offset 应用。数据库复制、Kafka 分区、ZooKeeper zxid 和共识状态机都能看到类似形式，但具体故障保证与范围要看实现，不能因“有日志”就假定它是容错全序广播。

全序日志很适合构建：

- 状态机复制：每个副本按相同顺序执行确定性命令，得到相同状态。
- Serializable transaction：把事务提交/写集合按顺序验证。
- Fencing token：日志位置单调递增，旧持有者不能越过新持有者。
- 唯一约束：以 `claim(username)` 作为日志命令，第一个 claim 成功，后续失败。

### Total-order broadcast 与 linearizability 的关系

两者密切相关但不是同一表述。使用全序广播可以实现线性一致 CAS：把“若值未被占用则认领”广播到日志，所有节点按顺序执行，第一个操作获胜。反过来，若有线性一致的原子递增/CAS 寄存器，可以给消息分配无间隙序号并按序交付，从而构造全序广播。

区别在于时间模型：全序广播的顺序固定且异步交付，但默认不要求某客户端读到的结果立即反映刚完成的外部操作；linearizability 还要求尊重真实时间。要从复制日志提供线性一致读取，必须确认读取点包含此前完成的写，例如 read index、barrier/no-op、quorum read 或有效 leader lease。

## Atomic commit：一个事务不能在参与者之间“半成功”

单节点事务提交通常依靠 WAL：先持久化 commit record，再把内存页异步落盘；崩溃恢复看到 commit record 就重做，没有就回滚。分布式事务跨多个节点时，每个节点都可能成功或失败，且消息可能丢失。若 A 提交扣款而 B 回滚入账，系统违反原子性。

简单地让每个参与者各自提交不够；让协调者逐个发送 commit 也不够，因为协调者可能发到一半崩溃。原子提交需要所有参与者最终做同一个决定。

### Two-phase commit（2PC）

2PC 引入 coordinator（transaction manager）和多个 participants。

**Phase 1: prepare/vote**

1. 应用向 coordinator 请求提交。
2. coordinator 给事务分配全局 ID，向参与者发送 prepare。
3. 每个参与者检查约束、获取必要锁、把事务数据和 prepare record 持久化。
4. 参与者回答 yes 后承诺：即使重启，也有能力随后 commit；不能再单方面 abort。

**Phase 2: decision**

1. 只有所有参与者都 vote yes，coordinator 才把 commit decision 持久化；任一 no/超时则记录 abort。
2. coordinator 反复向参与者发送决定，直到确认。
3. 参与者持久化决定、提交或回滚，并释放锁。

决定的“point of no return”是 coordinator 的 commit record。一旦落盘，即使客户端超时或 coordinator 重启，也必须继续提交。参与者收到 commit 后不能因为应用取消而回滚。

### 2PC 的阻塞问题

参与者 prepare 后若与 coordinator 失联，它不知道全局决定：其他参与者可能 vote no，也可能所有人 yes 且 coordinator 已提交。它既不能安全 commit，也不能安全 abort，只能等待 coordinator 恢复，同时继续持有行锁和资源。这叫 **in-doubt transaction**。

长时间 in-doubt 会阻塞无关事务、耗尽连接和放大故障。运维有时被迫 heuristic decision（人工猜测提交或回滚），但这会故意破坏原子性，必须对账修复。因此生产上应监控 prepared transaction 数量、最老年龄、锁占用、coordinator log 恢复能力，并让 coordinator 状态存储拥有可靠备份。

### XA 与数据库外部事务管理器

**XA** 是异构事务处理的标准 API，允许 transaction coordinator 协调多个支持 XA 的数据库或消息 broker。应用通过 XA transaction ID 将各连接加入同一事务，驱动 prepare/commit。

它能在特定企业系统中提供强原子性，但常见成本包括：

- 参与者在网络往返期间持锁，吞吐与尾延迟恶化。
- coordinator 或应用进程恢复配置错误，会留下无法判定的 prepared transaction。
- 数据库、驱动、连接池与 broker 的语义细节必须全部正确。
- 跨组织、跨公网或长业务流程不适合持有数据库锁。

微服务系统常改用 saga、outbox/inbox、幂等消费者和补偿动作。这不是“与 2PC 等价的无锁原子提交”：它选择 eventual completion，并让中间状态可见。业务必须接受补偿可能失败、用户已观察到中间结果，以及不可逆外部动作需要人工处置。

### 2PC 不是 fault-tolerant consensus

2PC 在 coordinator 存活并可恢复时能决定原子提交，但 coordinator 故障会让 prepared participants 阻塞。它不让多个 coordinator 在故障下通过多数派继续决定，也不是 Raft/Paxos 意义上的容错共识。

可以用 consensus 复制 coordinator 的 decision log，使单个 coordinator 故障后其他节点接管，这类 nonblocking atomic commit 协议比经典 2PC 多了容错决定层。但“使用共识提交事务”仍需处理参与者 prepare、锁和事务恢复；共识并不会自动把跨资源业务操作变成原子。

## Fault-tolerant consensus：在故障下仍只决定一个值

共识问题通常要求一个或多个节点提出值，算法最终决定一个值，并满足：

- **Uniform agreement**：没有两个节点决定不同值。
- **Integrity**：每个节点最多决定一次。
- **Validity**：决定值必须由某个节点提出，不能凭空产生。
- **Termination**：所有未崩溃节点最终都决定某个值。

前三项是 **safety（安全性）**：任何时刻、任何允许的消息延迟下都不能发生坏事。Termination 是 **liveness（活性）**：在模型规定的条件最终满足后，好事最终发生。

### Safety 与 liveness 必须分开承诺

一个正确共识系统在网络分区、时钟跳变或 leader 混乱时可以暂停，但不能让两边决定不同值。通常表述为“safety 在所有非 Byzantine 网络延迟下保持；liveness 在最终存在足够长的同步窗口、quorum 可通信时恢复”。

若监控只看“集群还能写”，可能鼓励实现牺牲 safety；若只看“没有 split brain”，一个永远不做决定的系统也会通过。生产验收要同时验证：旧 epoch 写被 fencing、已提交日志不回退；以及故障恢复后在目标时间内选主并继续提交。

### FLP 与超时的角色

在完全异步网络中，只要允许一个节点崩溃，就不存在保证总能终止的确定性共识算法，因为无法区分崩溃与无限延迟。这不意味着共识不能实践，而是 termination 需要额外假设：超时、故障检测器、随机化，或“网络最终会稳定一段时间”。

超时帮助节点发起新一轮选举，但不证明旧 leader 已死。正确性来自 term/ballot、quorum 相交、日志规则和持久化状态；超时只推进 liveness。把“心跳超时”本身当作互斥证明会产生 split brain。

### Epoch、quorum 与 leader

Paxos、Raft、Zab、Viewstamped Replication 的表述不同，但常见结构是：

1. 时间被划分为递增 epoch/term/ballot，每个 epoch 最多一个被认可的 leader。
2. leader 必须得到多数 quorum 支持，两个多数集合必然相交。
3. 新 leader 不能覆盖已由 quorum 接受/提交的旧决定，选举规则确保它继承足够新的日志。
4. 决定写入复制日志，节点按相同顺序应用到确定性状态机。

多数派系统通常能容忍少于半数节点故障：3 节点容忍 1 个，5 节点容忍 2 个。增加到偶数节点常不会增加故障容忍度，却增加通信和运维成本。节点应分布在独立故障域，但跨地域 quorum 会把 WAN 延迟带入写路径。

共识算法一般假设 crash/recovery 或 omission faults，不防任意恶意/错误行为。Byzantine consensus 需要不同模型与更多副本，不能看到“quorum”就默认能抵抗磁盘返回坏数据、节点发出矛盾消息或证书被攻破。

### Consensus 与 total-order broadcast

反复运行共识，为日志每个位置决定下一条消息，就得到 total-order broadcast；反过来，在全序广播日志中投递提案，取第一个合法提案作为决定，可以解决单次共识。工程系统因此多以 replicated log 暴露共识结果，而不是让业务直接调用 `consensus(value)`。

需要注意：日志共识通常只覆盖一个 group/shard。跨多个 Raft group 的事务仍需要额外协议；“每个分片都强一致”不等于跨分片操作原子或全局可串行化。

## Membership and coordination：小数据也可能承担全系统正确性

ZooKeeper、etcd 等协调服务把共识包装成较易使用的接口。它们通常只存少量、变化不频繁的元数据，却支撑服务发现、leader election、配置、分片 ownership、分布式锁和故障转移。

### Linearizable atomic operations

CAS、create-if-absent 和递增版本允许多个节点竞争 leader 或锁，只有一个成功。客户端应把返回 revision/epoch 当 fencing token 传播到真正资源处；仅靠 ephemeral node 消失或 lease 超时，旧客户端仍可能在暂停恢复后继续执行。

### Operation ordering 与通知

全序 revision 可以给事件排序，watch 让客户端订阅配置或 membership 变化，避免持续轮询。但 watch 通常不是永久消息队列：连接断开可能需要从 revision 恢复，历史可能 compaction，回调可能重复。客户端必须先读取一致快照，再从对应 revision watch，并在 gap/compaction 时重新同步。

### Failure detection 与 ephemeral membership

协调服务通过 session/lease 将节点注册与存活关联。会话超时后删除 ephemeral entry，其他节点收到变化并重新分配任务。但超时只是协调服务撤销资格，不会停止隔离节点上的代码。仍需 fencing 防止 zombie worker 写入。

### Service discovery

服务实例把地址注册到目录，调用者查询或 watch。这里常不需要每次请求都线性一致读取；缓存目录能提高可用性。但 ownership、唯一 leader 等控制面决定必须通过强一致写入。把“控制面要强一致、数据面可缓存”分开，是降低协调服务压力的常见方式。

### Membership 本身也需要共识

共识算法常假设参与投票的节点集合已知，但集群扩缩容恰好要改变这个集合。如果不同节点同时使用旧配置和新配置，两个互不相交的“多数派”可能各自做决定。安全实现使用 joint consensus 或等价的联合配置过渡，确保旧新 quorum 在切换期间相交。

不要通过直接改配置文件、一次替换多数节点或复用旧节点身份来操作共识集群。成员变更是日志中的一致决定，应串行执行并验证新节点追上后再移除旧节点。

## 工程现实：怎样选择保证而不是选择口号

可以从不变量反推最小协调范围：

| 业务状态 | 常见保证 | 实现方向 |
| --- | --- | --- |
| leader/锁 ownership | Linearizable CAS + fencing | 共识协调服务、递增 epoch |
| 单账户余额 | 串行化或单 key 状态机 | 固定分区 leader、事务/CAS |
| 用户名唯一 | 每个名称线性一致 claim | 按名称分区，避免全局锁 |
| 社交 Feed | 因果/单调读取常已足够 | session token、日志水位 |
| 搜索索引 | 有界延迟 eventual consistency | CDC、watermark、回放修复 |
| 跨服务订单流程 | 明确中间状态与补偿 | saga + outbox + 幂等，不冒充原子 |

共识系统需要观察的不只是 CPU 与存活：

- current term、leader changes、election duration；
- commit index、applied index、follower lag；
- quorum availability、fsync latency、proposal latency；
- rejected stale epoch/fencing token 数量；
- snapshot/compaction 进度与日志增长；
- membership configuration version 与未完成变更；
- 2PC prepared/in-doubt 数量和最老事务年龄。

测试应使用 history-based verification，而不只是 API 成功率。Jepsen 类测试会在并发操作、网络分区、进程暂停和时钟异常下收集完整历史，再验证能否线性化。对于状态机，还要验证已提交前缀不会回退、不同副本同一 index 的命令相同。故障注入必须包含磁盘慢、fsync 失败、长 GC pause、双向不对称网络和滚动升级，而不是只 kill leader。

## 常见理解偏差

1. **Strong consistency 是一个精确定义**：它常被不同产品用来指 linearizability、session consistency 或仅仅“同步复制”，必须追问具体历史保证。
2. **Linearizable 就是 serializable**：前者是操作的新鲜度与真实时间顺序，后者是事务并发隔离。
3. **CAP 是平时三选二**：它只描述网络分区下 linearizability 与特定 availability 定义的冲突。
4. **`w+r>n` 就是 linearizable**：quorum 相交不处理并发、sloppy quorum、旧 leader 和错误版本决议。
5. **Lamport timestamp 能识别并发**：它把所有事件全序化；version vector 才能表示不可比较的并发关系。
6. **有递增 ID 就有全局顺序保证**：生成编号不代表节点已知所有更小操作，也不保证真实时间或完成顺序。
7. **2PC 是 consensus**：coordinator 故障后经典 2PC 会阻塞；容错共识能通过 quorum 继续决定。
8. **超时证明 leader 已死**：超时只是故障怀疑与 liveness 机制，安全性来自 epoch、quorum 和 fencing。
9. **共识集群能保护所有业务写**：只保护经过其日志的状态；旧 worker 写外部数据库仍需 token 校验。
10. **三个节点跨三个机房就永远高可用**：任一机房到另外两个的网络路径、维护叠加和共同依赖都可能失去 quorum。

## 本章小结

- Linearizability 提供尊重真实时间的单副本幻觉；它与 serializability 是不同维度。
- CAP 只精确描述网络分区下 linearizability 与特定 availability 的冲突，不能替代完整故障设计。
- Causality 是 partial order；Lamport timestamp 产生与因果兼容的 total order，却不能识别并发。
- Total-order broadcast 让节点交付同一日志，并与 consensus、线性一致 CAS 紧密等价。
- 2PC 解决原子提交，但 coordinator 故障可能阻塞；容错共识用 epoch、quorum 和日志保证只决定一个值。
- Safety 必须始终保持，liveness 依赖网络最终稳定和 quorum 可达；超时推进选举但不证明互斥。
- Membership、锁和服务发现依赖少量高价值元数据，必须配合 fencing、watch 恢复和安全成员变更。
