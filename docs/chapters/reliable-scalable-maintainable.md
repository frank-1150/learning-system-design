## 这一章真正要建立的坐标系

第一章不是一份“高可用最佳实践清单”。Kleppmann 先做了一件更基础的事：定义我们评价数据系统时使用的语言。

今天的应用大多不是 **compute-intensive（计算密集型）**，而是 **data-intensive（数据密集型）**。CPU 往往不是最难的问题，更难的是数据量、数据结构的复杂度、数据变化速度，以及多个系统之间如何保持正确关系。一个普通产品背后通常已经组合了数据库、缓存、搜索索引、消息系统和批处理任务：

```text
用户请求
   │
   ▼
应用服务 ──读──> 缓存 ──miss──> 主数据库
   │                         │
   ├──写──> 主数据库 ──变更──┼──> 搜索索引
   │                         └──> 分析/推荐流水线
   └──事件──> 消息日志 ─────────> 异步消费者
```

当应用把这些通用组件组合起来，并向外提供一个统一 API 时，应用本身就成为了一个新的、专用的数据系统。它必须回答：缓存什么时候失效？数据库写成功但索引更新失败怎么办？客户端能看到多旧的数据？灾难恢复会丢多少数据？从这一刻起，工程师不只是“调用数据库的人”，而是 **data system designer（数据系统设计者）**。

原书用三个目标统领后续所有章节：

| 原文术语 | 更准确的理解 | 不是 |
| --- | --- | --- |
| Reliability | 面对预期范围内的逆境，系统仍正确地服务 | “机器永不坏” |
| Scalability | 当特定负载参数增长时，有合理的应对路径 | “上了分布式就是可扩展” |
| Maintainability | 不同的人能长期安全、有效地运营和修改系统 | “代码有注释” |

## Reliability：可靠性从区分 Fault 和 Failure 开始

原书给可靠性的近似定义是：**即使出错，仍持续正确工作**。这里“正确”同时包含功能、性能、安全和对误用的容忍，而不仅是进程还活着。

### Fault 不是 Failure

- **Fault（故障）**：某个组件偏离规格，例如一个磁盘坏了、一个副本超时、一个进程返回坏数据。
- **Failure（失效）**：系统整体不再向用户提供承诺的服务，例如订单确认后丢失，或页面慢到等同不可用。

组件 fault 无法降到零。可靠性设计的重点，是让 fault 被吸收、隔离或恢复，不升级成用户可见的 failure。

```text
磁盘损坏（fault）
   ├─ 有副本 + 自动切换 ─> 用户无感
   └─ 无副本/恢复失败 ───> 数据不可用（failure）
```

这也是“故障演练”成立的原因。主动杀掉一个副本看起来是在制造 fault，实际上是在持续验证隔离、选主和恢复链路，降低真正 failure 的概率。演练的目标不是展示系统能重启，而是验证不变量：确认过的写是否还在、重试是否重复扣款、流量是否真的绕过故障节点。

### 硬件故障：独立随机故障相对好处理

磁盘、内存、电源、网卡都可能失败。单机时代常用 RAID、双电源和备机提高单台机器可靠性；机器数量增加后，即使每台机器很可靠，集群每天遇到某种硬件故障也会成为常态。

工程上的转变是：从“尽量让单机不坏”转向“允许整机随时退出”。软件复制、自动重调度和滚动升级让维护机器不必等于维护窗口。不过冗余不自动等于可靠：同机架副本、同时过期的证书、相同固件缺陷都可能让看似独立的副本一起失败。

### 软件错误：最危险的是相关性

硬盘故障通常近似独立，软件错误却会复制到每个实例。一个特殊输入、共享依赖变慢、资源泄漏或错误重试策略，可以让所有副本一起崩溃。这类 fault 的共同点是系统隐藏了一个“平时成立”的假设：

```text
假设：下游通常在 100ms 内返回
变化：下游延迟升到 3s
后果：线程池占满 → 请求排队 → 上游超时重试 → 负载继续放大
```

因此，多副本只能对付一部分故障。对系统性软件错误，还需要进程隔离、资源上限、超时预算、重试预算、持续自检和端到端不变量监控。例如消息系统不只监控“进程健康”，还应核对进入、确认、失败和积压消息之间是否守恒。

### 人为错误：把安全恢复设计进系统

“不要犯错”不是工程方案。真正可操作的可靠性设计包括：

1. 让正确操作成为默认路径，危险操作需要更强确认和更小作用域。
2. 把实验环境与生产故障域隔离，同时提供足够真实的测试数据和流量模型。
3. 支持渐进发布、快速回滚和可重算的派生数据。
4. 用审计日志记录谁在何时改变了什么，而不是只保留最终配置。
5. 监控关键假设，例如容量余量、复制水位、队列积压和错误预算。

这里有一个很实用的判断：**如果恢复主要依赖某位资深工程师记得一串命令，恢复能力就还没有成为系统能力。**

## Scalability：先描述负载，再讨论扩展

“这个架构很 scalable”几乎没有信息量。扩展性必须绑定两个问题：

1. 哪一个负载参数将如何增长？
2. 为了维持哪一个性能目标，需要增加或改变什么？

### Load parameters：找到真正驱动成本的变量

常见负载参数包括 QPS、读写比、活跃连接数、对象大小、缓存命中率和数据保留期。但最关键的参数有时不是平均值，而是分布的尾部。

原书用 Twitter 时间线说明这一点。假设平均发帖约 4,600 次/秒，读取首页时间线约 300,000 次/秒：

**方案 A：读时计算（fan-out on read）**

发帖只写一份；读取首页时查出关注列表，再拉取并合并每个被关注者的内容。写入便宜，读取昂贵。

**方案 B：写时扩散（fan-out on write）**

每个用户维护一份收件箱式时间线。发帖时把内容 ID 写入所有粉丝的时间线，读取变成简单点查。读取便宜，写入被放大。

如果每条内容平均投递给 75 个粉丝，那么：

```text
4,600 posts/s × 75 followers ≈ 345,000 timeline writes/s
```

但平均值隐藏了 celebrity 用户。一个拥有数千万粉丝的账户发一条内容，可能制造数千万次写入。真正的 load parameter 不是“发帖 QPS”，而是 **发帖频率 × 粉丝数分布**。

最终合理方案通常是混合式：普通用户写时扩散；超大账号不做全量扩散，读取时再合并。这里的核心不是记住 Twitter 架构，而是学会寻找让成本非线性增长的变量。

### Performance：response time 不是一个平均数

原书严格区分了两个常被混用的词：

- **Latency**：请求处于等待、尚未被处理的时间。
- **Response time**：客户端观察到的总时间，包括网络、排队和实际服务时间。

线上系统的响应时间是一个分布。平均值无法回答“有多少用户真的等了这么久”。因此通常看：

- P50：典型请求体验。
- P95/P99：长尾用户和容量拐点。
- P99.9：极端尾部，是否值得优化取决于业务价值与成本。

假设一个页面并行调用 20 个后端，每个后端都有 1% 的请求很慢。页面至少碰到一个慢调用的概率约为：

```text
1 - 0.99^20 ≈ 18.2%
```

这就是 **tail latency amplification（尾延迟放大）**：单个服务“只有 1% 慢”并不代表组合请求也只有 1% 慢。Scatter/Gather、搜索分片和微服务聚合都会遇到这个问题。

### SLI、SLO 与 SLA：把“够快、够可靠”写成契约

这三个缩写常被混用，但职责不同：

- **SLI（Service Level Indicator）**：实际测量值，例如成功请求比例、P99 响应时间、数据新鲜度或正确处理的事件比例。
- **SLO（Service Level Objective）**：团队希望达到的目标，例如“滚动 30 天内 99.9% 的合格请求成功”。
- **SLA（Service Level Agreement）**：与客户约定的外部合同，通常包含未达标后的赔偿或其他责任。

好的 SLI 必须接近用户体验。只监控进程存活不能代表服务可用；数据库返回 200 但数据错误，也不能算成功。对于异步系统，“API 接收成功”可能只是入口 SLI，还需要“事件在 5 分钟内完成处理”的新鲜度 SLI。

SLO 还隐含一个 **error budget（错误预算）**。99.9% 月度成功率允许约 0.1% 的请求失败，或在时间口径下约 43 分钟不可用。预算不是鼓励故障，而是让可靠性与发布速度可讨论：预算充足时可以承担受控变更；预算快速消耗时，应暂停高风险发布并优先修复系统性问题。

> [!note] 可用性口径会改变结果
> 按“分钟是否完全正常”计算，和按“合格请求成功比例”计算，会得到不同数字。低流量时段、恶意请求、客户端取消和依赖失败是否计入，都必须在 SLI 定义中写清楚。

### Queueing：高利用率为什么会突然恶化

服务器并行度有限。少量慢请求占住工作线程后，后面的快请求也必须排队，形成 **head-of-line blocking（队头阻塞）**。系统在 50% CPU 时可能很平稳，在接近饱和时，任何抖动都会转化为排队，P99 会先于平均值恶化。

Little's Law 给容量讨论一个很实用的近似：

```text
系统中平均并发数 L = 到达率 λ × 平均停留时间 W
```

如果入口是 5,000 requests/s，端到端平均停留 200ms，那么系统平均约有 `5,000 × 0.2 = 1,000` 个请求在途。若依赖变慢到 2s，即使 QPS 不变，并发也会膨胀到约 10,000；线程、连接、内存和排队空间都会一起承压。这解释了为什么“下游只是慢了”常迅速演化成全链路资源耗尽。

压测也可能撒谎。若压测客户端等待上一个请求返回才发送下一个，服务越慢，客户端施加的负载反而越低，这叫 coordinated omission。更接近生产的 open-loop 压测应按目标到达率发请求，并把排队时间计入响应时间。

> [!warning] 百分位不能直接平均
> 多台机器的 P99 不能取平均后当作全局 P99。应合并直方图或原始分布，再重新计算百分位。

### Scale up、Scale out 与 Elasticity

- **Scale up**：换更强的机器。状态系统往往能因此保持更简单的单机模型。
- **Scale out**：把负载分散到多台机器。它引入分区、复制、协调和跨节点故障。
- **Elasticity**：负载变化时自动增减资源。适合波动大、启动快的工作负载；状态系统扩容涉及数据迁移，自动控制环反而可能制造惊喜。

工程现实里不必宗教式二选一。若一台更大的数据库能把复杂度推迟两年，而且成本可接受，它可能比过早分片更好。无状态计算层可以早做横向扩展，状态层则应在容量、可用性或地域需求真正迫使它分布式时再付复杂度成本。

## Maintainability：系统的大部分成本发生在上线之后

原书把可维护性拆成三个互补目标。

### Operability：让人能预测和控制系统

好的可运维性不是“全自动、无需人类”。它让日常工作自动化，同时在异常情况下仍给操作者可见性和控制权：

- 能看到运行时内部状态，而不只是一颗绿色健康灯。
- 维护不依赖某台特殊机器或某位特殊的人。
- 默认配置合理，必要时允许人工覆盖。
- 扩容、迁移、备份恢复和版本升级有清晰的状态机与进度。
- 行为可预测：“执行 X 后会发生 Y”，而不是靠尝试理解系统。

对于数据系统，最值得做成一等指标的往往是 **lag、queue depth、watermark、ownership、version 和 recovery progress**。这些指标直接描述数据是否正在正确流动。

### Simplicity：消除 accidental complexity

简单不等于功能少，而是减少 **accidental complexity（偶然复杂度）**：那些不是业务问题本身要求，却由实现方式带来的状态爆炸、特殊分支、紧耦合和隐含假设。

好的 abstraction（抽象）能隐藏大量细节，同时保留正确的控制面。SQL 隐藏了磁盘页、索引遍历、并发控制和崩溃恢复；但抽象泄漏时，工程师仍需知道慢查询与索引的关系。成熟的工程判断不是“永远忽略底层”，而是知道何时可以依赖抽象，何时必须下潜。

### Evolvability：让改变成为常规操作

需求、法规、流量和团队都会变。可演化性意味着 schema、接口、部署单元和数据流允许不同版本共存，允许渐进迁移，并能在错误时退回。

把一次大爆炸迁移拆成这些阶段通常更安全：

```text
新增兼容字段 → 新代码双读验证 → 后台回填 → 切换读取 → 停止旧写 → 清理旧字段
```

每一步都应有观测指标和回滚点。后续 Ch4 会把这种思路扩展到编码与 schema 演化。

## 把本章变成一张工程评审表

面对任何新系统，可以先写出以下一页，而不是先画组件：

| 类别 | 必须量化的内容 |
| --- | --- |
| 负载 | 平均与峰值 QPS、读写比、对象大小、热点分布、增长率 |
| 体验 | P50/P95/P99、超时预算、允许降级的功能 |
| 正确性 | 不能破坏的不变量、可接受 RPO/RTO、重复与乱序语义 |
| 故障模型 | 单进程、单机、单 AZ、依赖变慢、错误配置分别怎么处理 |
| 运维 | 核心指标、告警、回滚、对账、容量余量和演练方式 |
| 演化 | schema、分区、依赖和版本如何渐进迁移 |

如果这些问题没有答案，讨论“Kafka 还是 RabbitMQ”“单体还是微服务”通常都太早。

## 常见理解偏差

1. **可靠性 = 高可用**：可用但返回错误数据，仍然不可靠。
2. **扩展性 = 水平扩展**：错误的分区键会让更多机器也救不了热点。
3. **平均 QPS 能代表压力**：峰值、fan-out 和对象大小可能才决定瓶颈。
4. **自动扩缩容总比人工好**：状态迁移、冷缓存和滞后指标可能让控制环震荡。
5. **可维护性以后再做**：没有观测和回滚能力，第一次发布就已经在创造维护成本。

## 本章小结

- 可靠性关注如何阻止组件 fault 升级为系统 failure。
- 扩展性必须用具体负载参数和性能分布描述，不能脱离工作负载谈架构。
- 尾延迟、排队和 fan-out 往往比平均吞吐更早暴露系统上限。
- 可维护性由可运维性、简单性和可演化性共同决定，并应从第一天进入设计。
